# Claude 생태계 문서 자동 생성 스킬

## 개요

사용자가 Claude 생태계(Claude Code, Claude API, MCP, Claude.ai 등)의 특정 주제를 지정하면, 웹 검색과 공식 문서를 조사하여 비개발자도 이해할 수 있는 한국어 문서를 생성한다.

## 실행 방식

```
/updatedocs {주제}
```

| 입력 예시 | 동작 |
|-----------|------|
| `/updatedocs MCP 서버 설정` | MCP 서버 설정에 대해 조사 → 문서 생성 |
| `/updatedocs Claude Code hooks` | hooks 기능에 대해 조사 → 문서 생성 |
| `/updatedocs 최근 업데이트` | Claude 생태계 최신 변경사항 조사 → 문서 생성 |

주제가 없으면 사용자에게 "어떤 주제를 문서화할까요?"라고 물어본다.

## 실행 절차

### Step 1: 주제 파악 및 검색 키워드 설정

사용자가 제공한 주제를 분석하여 검색 전략을 수립한다.

1. 주제에서 핵심 키워드 추출
2. 검색 쿼리 3~5개 구성 (한국어 + 영어 조합)

**검색 쿼리 예시 — 주제가 "MCP 서버 설정"인 경우:**
```
- "Claude Code MCP server setup"
- "Claude MCP configuration guide"
- "site:docs.anthropic.com MCP"
- "site:code.claude.com MCP"
- "Claude Code MCP 서버"
```

### Step 2: 정보 수집

WebSearch와 WebFetch를 활용하여 정보를 수집한다. 병렬로 여러 소스를 동시에 조사한다.

#### 필수 조사 소스

| 우선순위 | 소스 | 방법 |
|---------|------|------|
| 1 | Anthropic 공식 문서 | WebFetch: `https://docs.anthropic.com/...` |
| 2 | Claude Code 공식 문서 | WebFetch: `https://code.claude.com/docs/...` |
| 3 | 웹 검색 결과 | WebSearch로 최신 정보 탐색 |
| 4 | Claude 내장 지식 | 위 소스로 부족한 부분 보충 |

#### 조사 규칙

- **최소 2개 이상의 소스**에서 정보를 교차 확인한다
- 공식 문서와 웹 검색 결과가 충돌하면 **공식 문서를 우선**한다
- 날짜가 명시된 정보는 **최신 정보를 우선**한다
- 조사 결과가 빈약하면 사용자에게 "이 주제에 대한 공식 정보가 제한적입니다. 알고 계신 내용이 있으면 알려주세요"라고 안내한다

### Step 3: 중복 검사

문서 생성 전에 이미 같은 주제의 문서가 있는지 확인한다. 모든 카테고리 폴더를 검사:

```bash
grep -rl "{핵심키워드}" content/basics/ content/workflow/ content/tips/ content/config/ content/troubleshooting/ content/claude-code-docs/ content/youtube-update/ 2>/dev/null
```

중복 판정 기준:
- **동일 주제 문서 존재** → 사용자에게 알리고 선택지 제시:
  - (A) 기존 문서에 새 정보 추가/업데이트
  - (B) 별도 문서로 새로 작성 (다른 관점/깊이)
  - (C) 건너뛰기
- **부분 중복** → 겹치지 않는 부분만 다루는 문서 작성
- **중복 없음** → 그대로 진행

### Step 4: 문서 작성

> 문서 생성 시, `references/doc-template.md`를 읽어서 템플릿과 작성 규칙을 따른다.

#### order 값 결정

새 문서의 order 값을 정하려면 기존 문서의 최대값을 확인한다:

```bash
grep -h "^order:" content/claude-docs/*.md 2>/dev/null | sort -t: -k2 -n | tail -1
```

기존 문서가 없으면 order: 1부터 시작. 있으면 최대값 + 1.

#### 작성 원칙

1. **조사한 내용을 기반으로만 작성** — 확인되지 않은 정보는 포함하지 않는다
2. **공식 문서의 기술적 설명을 비개발자 언어로 번역** — 원문을 그대로 옮기지 않는다
3. **실전 예시는 반드시 한국 사용자 맥락** — "김대리가 보고서를 작성할 때..." 같은 구체적 시나리오
4. **코드 블록은 최소화** — 꼭 필요한 경우에만 포함하고, 코드 위/아래에 비개발자용 설명 필수

#### 여러 문서 동시 생성

주제가 넓어서 여러 문서로 나눠야 할 경우:
- 사용자에게 분할 계획을 먼저 보고하고 확인받는다
- 3개 이상이면 서브에이전트 활용 가능 (각 서브에이전트에게 `references/doc-template.md` 경로 전달)

### Step 4.5: 품질 체크

문서 생성 후, 파일을 저장하기 전에 아래 체크리스트를 통과하는지 확인한다. 통과하지 못하면 수정 후 저장.

**필수 체크:**
1. **전문 용어 괄호 설명** — 영어 기술 용어(Hook, YAML, API, CLI, SDK, MCP 등)가 처음 등장할 때 괄호 설명이 있는가?
2. **빈 공백 없음** — 표나 예시 블록 안에 빈 칸이 없는가? 데이터가 없으면 해당 섹션을 제거했는가?
3. **Mermaid 품질** — 노드 텍스트 15자 이내, 노드 8개 이하, 텍스트 색상 `#fff`, 배경 충분히 진한가?
4. **비유 존재** — "이게 뭔가요?" 섹션에 일상 비유가 최소 1개 포함되어 있는가?
5. **Mac/Windows 병기** — 단축키 설명이 있다면 두 OS 모두 표기했는가?
6. **출처 URL 유효** — source_url과 출처 섹션의 링크가 실제 접근 가능한 URL인가?
7. **예시 블록 2개 이상** — `<div class="example-case">` 블록이 최소 2개 있는가?

### Step 5: 결과 보고

최종 결과를 사용자에게 보고:

```
## 실행 결과

- 주제: {사용자 지정 주제}
- 조사 소스: {사용한 소스 목록}
- 중복 검사: {결과}

### 생성된 문서
- content/{카테고리}/{파일명}.md — "{문서 제목}" (카테고리: {한국어 카테고리명})

### 조사 중 발견한 관련 주제
- {추가 문서화하면 좋을 관련 주제 1~3개 제안}
```

보고 후 사용자에게 제안: "생성된 문서를 커밋하려면 `/push`를 실행하세요."

## 에러 처리

| 상황 | 대응 |
|------|------|
| 웹 검색 실패 | Claude 내장 지식으로 대체 작성, 사용자에게 "웹 검색 없이 작성됨" 고지 |
| 공식 문서 접근 불가 | 웹 검색 결과로 대체, 접근 불가 URL 보고 |
| 주제가 너무 광범위 | 사용자에게 세부 주제 선택 요청 (예: "Claude API"는 너무 넓음 → "Claude API 스트리밍 응답" 등으로 좁히기) |
| 정보 부족 | 확인된 내용만으로 문서 작성, 부족한 부분 명시 |
| 카테고리 디렉토리 없음 | `mkdir -p content/{카테고리}` 자동 생성 후 진행 |

## 파일 저장 위치 (카테고리 자동 분류)

문서의 **정보 소스**와 **내용**에 따라 저장 폴더가 결정된다. 이 규칙은 반드시 따른다.

### 소스 기반 분기 (우선 적용)

| 소스 URL/출처 | 저장 위치 | 비고 |
|--------------|----------|------|
| `https://code.claude.com/docs/en/...` | `content/claude-code-docs/` | 공식 문서 한국어판 전용 |
| `https://docs.anthropic.com/...` | `content/claude-code-docs/` | 공식 문서 한국어판 전용 |

위 소스에 해당하면 무조건 `claude-code-docs/`에 저장한다.

### 내용 기반 분류 (그 외 모든 소스)

공식 문서 소스가 아닌 경우 (블로그, 웹 검색, 일반 주제 등), **문서 내용을 분석하여 아래 5개 카테고리 중 가장 적합한 곳에 저장**한다:

| slug | 카테고리명 | 이런 내용이면 여기 | 예시 |
|------|----------|------------------|------|
| `basics` | 기본 기능 | Claude Code/Claude의 핵심 도구, 명령어, 새 기능 소개 | Computer Use, Dispatch, Projects, 새 모델 출시 |
| `workflow` | 워크플로우 | 프로젝트 시작~배포까지의 작업 흐름, 패턴, 프로세스 | CI/CD 연동, 코드 리뷰 워크플로우, 팀 협업 패턴 |
| `tips` | 팁 & 트릭 | 효율적인 프롬프트, 활용 패턴, 생산성 팁 | 프롬프트 작성법, 숨겨진 기능, 시간 절약 팁 |
| `config` | 설정 & 커스터마이징 | 설치, 설정, 외부 도구 연결, 환경 구성 | CLAUDE.md 작성법, MCP 서버 설정, hooks 설정 |
| `troubleshooting` | 트러블슈팅 | 자주 발생하는 문제, 에러 해결법, 디버깅 | 인증 오류, 속도 저하 해결, 충돌 해결 |

**분류 판단 기준:**
1. 문서의 핵심 주제가 "이게 뭐야? 어떤 기능이야?" → `basics`
2. "이걸 어떤 순서로 해야 해?" → `workflow`
3. "이렇게 하면 더 잘 돼" → `tips`
4. "이걸 어떻게 설정해?" → `config`
5. "이게 왜 안 돼? 어떻게 고쳐?" → `troubleshooting`

여러 카테고리에 걸칠 때는 **문서의 주된 목적**을 기준으로 하나만 선택한다.

### 사용하지 않는 카테고리 (자동 분류 대상에서 제외)

| slug | 사유 |
|------|------|
| `claude-code-docs` | `code.claude.com/docs/en/` 소스 전용 — 소스 기반 분기에서만 사용 |
| `youtube-update` | `/updateyoutube` 스킬 전용 |
| `quick-start` | 예시 가이드 전용 — 수동 작성만 |

### order 값 결정

분류된 카테고리 폴더의 최대 order 값을 확인한다:

```bash
grep -h "^order:" content/{분류된카테고리}/*.md 2>/dev/null | sort -t: -k2 -n | tail -1
```

기존 문서가 없으면 order: 1부터 시작. 있으면 최대값 + 1.

### 중복 검사 범위

5개 카테고리 + youtube-update + claude-code-docs 전체를 검사한다:

```bash
grep -rl "{핵심키워드}" content/basics/ content/workflow/ content/tips/ content/config/ content/troubleshooting/ content/claude-code-docs/ content/youtube-update/ 2>/dev/null
```

### 태그 규칙

문서의 tags에 해당 카테고리의 slug는 포함하지 않아도 된다 (폴더가 곧 카테고리). 대신 내용에 맞는 의미 태그를 사용:

| 태그 | 해당하는 내용 |
|------|-------------|
| `신기능`, `업데이트` | 새로 출시된 기능, 버전 업데이트 |
| `활용법`, `워크플로우` | 사용법, 작업 흐름, 팁 |
| `설정`, `연동` | 설치, 설정, 외부 도구 연결 |
| `비즈니스`, `자동화` | 업무 활용, 자동화 사례 |
| `개념`, `기초` | 기본 개념 설명, 입문 가이드 |

## 기존 /updateyoutube와의 관계

| 항목 | /updateyoutube | /updatedocs (공식 문서) | /updatedocs (그 외) |
|------|---------------|----------------------|-------------------|
| 소스 | YouTube 영상 | code.claude.com, docs.anthropic.com | 블로그, 웹 검색 등 |
| 저장 위치 | `content/youtube-update/` | `content/claude-code-docs/` | **5개 카테고리 중 자동 분류** |
| 분류 방식 | 고정 | 고정 | 내용 기반 자동 판단 |
| 문서 형식 | 동일 (8섹션 구조) | 동일 (8섹션 구조) | 동일 (8섹션 구조) |

세 경로 모두 같은 품질 기준과 문서 형식을 공유한다. 중복 검사 시 모든 카테고리 폴더를 확인하여 같은 주제가 이중으로 문서화되지 않도록 한다.
