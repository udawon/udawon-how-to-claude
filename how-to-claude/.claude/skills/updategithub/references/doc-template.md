# 문서 템플릿 & 작성 규칙

> 이 파일은 /updategithub 스킬이 문서를 생성할 때 참조하는 템플릿과 규칙이다.
> Step 4 (문서 생성) 시점에 읽는다. 서브에이전트에게도 이 파일 경로를 전달하여 직접 읽도록 한다.

## 대상 독자

**터미널을 열어본 적 없는 완전 생초보**부터 포함한다. 오픈소스, 레포지토리, API 같은 용어를 모르는 사람이 읽는다는 전제로 모든 문장을 작성한다. "이 정도는 알겠지"라는 가정을 하지 않는다.

## 문서 작성 규칙

### 1. 전문 용어 처리 (최우선 규칙)

기술 용어가 **처음 등장**할 때 반드시 괄호 안에 쉬운 설명을 붙인다.

**필수 괄호 설명 대상:**
- 오픈소스(누구나 무료로 쓸 수 있는 공개 소프트웨어)
- 레포지토리(코드를 저장하는 온라인 공간, 일종의 프로젝트 폴더)
- Stars(깃허브에서 좋아요를 누르는 것과 같음)
- Fork(다른 사람의 코드를 복사해서 내가 수정하는 것)
- AI(인공지능), LLM(대규모 언어 모델, ChatGPT 같은 AI의 핵심 기술)
- CLI(키보드로 명령어를 입력해서 조작하는 화면), API(프로그램끼리 대화하는 통로)
- 온디바이스(인터넷 없이 내 기기에서 직접 실행)

**같은 문서 안에서 처음 등장할 때만 괄호 설명. 두 번째부터는 생략 가능.**

나쁜 예:
```
이 레포지토리는 LLM을 온디바이스로 실행합니다.
```

좋은 예:
```
이 레포지토리(코드 저장 공간)는 LLM(대규모 언어 모델)을 온디바이스(인터넷 없이 내 기기에서 직접)로 실행합니다.
```

### 2. 비유 필수

모든 프로젝트의 "이게 뭔가요?" 섹션에 일상 비유 1개 이상 포함. 비유 없이 기능만 나열하는 것은 금지.

좋은 비유 예시:
- "마치 카카오톡 알림처럼 ___"
- "구글 지도가 길을 찾아주듯이 ___"
- "의사가 처방전을 쓰듯 ___"
- "택배 송장번호로 위치를 추적하듯 ___"
- "스마트폰 앱스토어처럼 ___"

### 3. 예시 블록 필수

레포마다 `<div class="example-case">` 블록 **최소 2개** 포함:
1. **실전 예시** — 구체적인 인물이 이 도구를 쓰는 시나리오
2. **이런 분께 추천해요** — 대상 독자 3가지 유형

마지막 "이번 주 하이라이트" 블록 포함 시 문서 전체 최소 3개 이상.

### 4. 실전 예시 작성 기준

example-case 안의 시나리오는 반드시:
- **구체적인 직업/역할**이 있는 인물 등장 (예: "마케터 김지수 씨", "스타트업 대표 박현우 씨")
- **기존의 불편함**이 묘사될 것
- **이 도구로 해결한 결과**가 명확히 서술될 것

나쁜 예:
```
이 도구는 화면 녹화에 유용합니다.
```

좋은 예:
```
스타트업에서 투자자용 데모 영상을 만들어야 했던 김민준 씨는 유료 프로그램을 사야 한다는 사실에 막막했습니다.
openscreen을 설치한 후 30분 만에 자동 줌 효과와 깔끔한 배경이 적용된 전문가 수준의 데모 영상을 완성했습니다.
```

### 5. 빈 데이터 금지

정보가 없는 항목은 표에서 제거. 빈 칸 절대 금지.

### 6. 파일명

- 트렌딩 모드: `YYYY-MM-DD-github-trending.md`
- 직접 지정 모드: `YYYY-MM-DD-{repo-name}.md`

### 7. 태그

`"github"`, `"트렌딩"`, `"오픈소스"` 태그 필수.

## Mermaid 다이어그램

분야별 분포나 트렌드 흐름을 시각화할 때 선택적으로 사용. 문서당 최대 1개.

규칙:
- 노드 텍스트 최대 15자
- 노드 최대 8개
- 텍스트 색상 `#fff`, 배경 충분히 진한 색
- 없어도 무방. 억지로 넣지 말 것.
