## 이게 뭔가요?

회사마다 오랜 시간 공들여 만든 디자인 시스템이 있습니다. 버튼은 어떻게 생겼는지, 색상은 어떤 팔레트를 쓰는지, 어떤 상황에서 어떤 컴포넌트를 써야 하는지 정의된 규칙 묶음이죠.

문제는 AI(Claude 등)에게 "우리 회사 디자인 시스템 따라서 만들어줘"라고 해도, AI가 그 규칙을 모른다는 겁니다. 결국 AI가 마음대로 만들어서 디자인 시스템이 유명무실해지는 상황이 생깁니다.

이 영상은 그 문제를 해결하는 방법을 실험한 결과입니다: **기존 디자인 시스템을 Claude Skill(스킬, AI에게 전달하는 규칙 묶음)로 변환**하면, AI가 그 규칙을 따르면서 HTML 프로토타입과 Figma(피그마, 디자인 협업 도구) 파일을 동시에 생성할 수 있습니다.

> **비유:** 신입 직원이 들어오면 회사 규정집을 읽히듯이, AI에게도 "우리 회사 디자인 규칙"을 마크다운(Markdown, 텍스트로 쓰는 문서 형식) 파일로 만들어 읽혀두는 것입니다.

## 왜 알아야 하나요?

디자이너나 개발자가 AI로 UI(사용자 인터페이스, 화면)를 만들 때 가장 큰 불만 중 하나는 "AI가 우리 디자인 가이드를 안 따른다"는 것입니다.

이 방법을 쓰면:
- AI가 무작위로 만드는 게 아니라 **우리 회사 기준에 맞춰** 화면을 생성
- Figma 파일과 실제 HTML 코드가 **동일한 결과**로 나옴
- 디자이너와 개발자가 각자 따로 작업해도 **일관성** 유지 가능

## UI 프레임워크 vs. 디자인 시스템: 뭐가 다른가요?

이 영상에서 가장 핵심적인 구분입니다.

| 구분 | 설명 | 예시 |
|------|------|------|
| **UI 프레임워크** | 컴포넌트(버튼, 입력창 등)의 **생김새** 정의 | "버튼은 높이 40px, 모서리 둥글기 4px" |
| **디자인 시스템** | 생김새 + **언제 무엇을 써야 하는가** 정의 | "라디오 버튼 대신 버튼을 쓸 상황은 언제인가?" |

AI에게 단순한 UI 프레임워크만 주면 생김새는 맞춰도 상황에 맞는 컴포넌트 선택은 못합니다. 완전한 디자인 시스템 — 원칙, 사용 조건, 예외 상황까지 — 을 Claude Skill로 만들어야 AI가 진짜 일관되게 작동합니다.

## 어떻게 하나요?

### 단계 1: 기존 시스템 분석 → 마크다운 변환

Claude에게 기존 UI 프레임워크(CSS, SaaS(스타일 시트 언어) 파일 등)를 보여주고, 마크다운 형식으로 정리해달라고 요청합니다.

이때 단순히 컴포넌트 목록만 옮기는 게 아니라, 아래 내용도 함께 포함해야 합니다:
- **디자인 원칙**: 이 디자인이 추구하는 가치
- **사용 조건**: 어떤 상황에서 어떤 컴포넌트를 써야 하는가
- **빈 상태(Empty State) 처리**: 데이터가 없을 때 화면은 어떻게 보이는가
- **색상 사용 원칙**: 언제 어떤 색을 쓰는가

<div class="example-case">
<strong>예시: 컴포넌트 정의 방식</strong>

**기존 CSS 방식 (AI가 이해하기 어려움):**
```css
.alert { padding: 12px 16px; border-radius: 4px; }
.alert-error { background: #fee2e2; color: #991b1b; }
```

**마크다운 스킬 방식 (AI가 이해하기 쉬움):**
```markdown
## Alert 컴포넌트

**언제 사용**: 사용자가 즉시 확인해야 할 중요한 정보나 오류 상황
**error 타입**: 사용자의 액션이 실패했을 때. 구체적인 해결 방법을 함께 제공할 것
**warning 타입**: 사용자가 알아야 하지만 즉시 행동이 필요하지 않은 경우
```

</div>

### 단계 2: Claude Skill 구조 만들기

변환된 마크다운 파일들을 Claude Skill로 구성합니다. 스킬은 크게 두 역할을 합니다:

1. **라우팅 규칙**: "이런 화면을 만들 때는 이 컴포넌트를 써라"는 지침
2. **출력 형식 지정**: HTML로 출력할지, Figma용 JSON(제이슨, 데이터 교환 형식)으로 출력할지 결정

이 영상에서는 **JSON을 공통 소스 오브 트루스(Source of Truth, 단일 진실 공급원)**로 사용했습니다. JSON 파일 하나를 생성하면, Figma 플러그인이 이를 읽어 Figma 캔버스에 렌더링하고, 동시에 Claude가 같은 JSON을 기반으로 HTML 코드도 만듭니다.

```
Claude Skill (규칙) → JSON 생성 → Figma 플러그인으로 Figma 파일 생성
                               → HTML 코드 생성
```

<div class="example-case">
<strong>실전 케이스: 사내 커뮤니케이션 앱 만들기</strong>

실험에서 아래 프롬프트(AI에게 보내는 요청 메시지)를 입력했습니다:

> "내부 커뮤니케이션 앱을 만들어줘. 팀원들이 프로젝트 얘기를 나누고 파일도 공유하는 앱. 채널도 있고 이벤트 RSVP 기능도 있으면 좋겠어."

Claude는 디자인 스킬을 읽은 후 먼저 몇 가지 clarifying question(명확화 질문)을 했습니다:
- 기본 뷰는 무엇으로 할까요? (→ 활동 피드)
- 어떤 종류의 회사인가요? (→ 소프트웨어 스타트업)
- 이벤트 섹션의 비중은? (→ 별도 섹션으로)

결과: Figma에는 1440px(PC), 768px(태블릿), 375px(모바일) 세 가지 반응형 레이아웃이 생성되었고, HTML 버전도 동일한 구조로 만들어졌습니다. 호버 효과(마우스를 올렸을 때 변하는 효과)와 액티브 상태(현재 선택된 메뉴 표시)도 디자인 시스템 규칙대로 적용되었습니다.

</div>

## 주의할 점

**1. UI 프레임워크만으론 부족합니다**

CSS 파일을 그대로 마크다운으로 옮기는 것만으로는 안 됩니다. "언제 이 컴포넌트를 쓰는가"까지 포함해야 AI가 상황에 맞는 선택을 할 수 있습니다.

**2. 디자인 시스템이 바뀌면 스킬도 업데이트해야 합니다**

Claude Skill은 스냅샷(특정 시점의 복사본)입니다. 실제 디자인 시스템이 바뀌면 스킬 파일도 함께 업데이트해야 일관성이 유지됩니다.

**3. 처음엔 결과물이 세련되지 않을 수 있습니다**

영상에서도 강조했듯, 결과물의 품질은 스킬에 정의된 디자인 시스템의 품질을 그대로 반영합니다. 디자인 시스템 자체가 단순하다면 결과도 단순합니다. AI가 알아서 더 예쁘게 만들어주길 기대하면 실망할 수 있습니다.

## 정리

- 기존 UI 프레임워크를 Claude Skill(마크다운 파일 묶음)로 변환하면 AI가 우리 회사 디자인 규칙을 따르게 만들 수 있다.
- JSON을 공통 기준으로 쓰면 Figma 파일과 HTML 코드를 동시에 일관되게 생성할 수 있다.
- 단순한 컴포넌트 목록이 아니라 "언제 어떤 컴포넌트를 쓰는가"까지 정의해야 진짜 디자인 시스템이다.
