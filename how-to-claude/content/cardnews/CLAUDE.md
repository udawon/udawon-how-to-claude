# 에이전트 설계서 작성 가이드

## 시작하기 전에: AI 에이전트의 본질

> **에이전트는 '똑똑한 챗봇'이 아닙니다. 당신 대신 일하는 직원입니다.**

- 24시간 일하는 마케터
- 자는 동안 콘텐츠 쓰는 작가
- 주말에도 작동하는 시스템
- *도구로 쓰면 도구가 되고, 직원으로 쓰면 직원이 됩니다.**

### 대부분이 실패하는 이유

"ChatGPT 알아요. 근데 실제로는 잘 안 써요."

알면서 실행 못 하는 건 시간이나 툴 문제가 아닙니다.

- *진짜 이유는 무엇을 위임할지 몰라서입니다.**

당신이 뭘 하는 사람인지 명확하지 않으면 AI도 못 씁니다.

### 올바른 순서

```

1. 내 역할 정의          ← 0단계

2. 반복 작업 찾기        ← 0단계

3. 일하는 방식 설계      ← 0단계

4. 그다음에 툴/설계      ← 1~4단계

```

- *툴은 나중 문제입니다.** 이 순서로 가면 AI 팀이 만들어집니다.

## 진행 방식

| 단계 | 이름 | 목적 |

|------|------|------|

| **0단계** | 업무 파악 | 무엇을 위임할지 찾기 |

| **1단계** | 작업 파악 | 자동화할 작업의 세부사항 확인 |

| **2단계** | 조직 설계 | 필요한 직원 구성 |

| **3단계** | 설계서 작성 | 통합 설계서 작성 |

| **4단계** | 리뷰 | 수정 사항 확인 |

## 0단계: 업무 파악 (무엇을 위임할지 찾기)

- *이 단계의 목표**: AI에게 위임할 작업을 찾아내기

사용자가 처음 왔을 때, 아래 질문을 순서대로 진행한다.

### 질문 1: 당신은 뭘 하는 사람인가요?

```

당신의 역할을 한 문장으로 정의해주세요.

예시:

- "직장인 대상 AI 활용 콘텐츠를 만드는 사람"
- "B2B SaaS 스타트업의 마케팅 담당자"
- "프리랜서 개발자로 클라이언트 프로젝트 진행"
- "쇼핑몰 운영하면서 상품 등록/CS 처리하는 사람"

```

### 질문 2: 일주일 동안 어떤 일을 하나요?

```

최근 일주일 동안 한 업무를 모두 적어주세요.

작은 것도 괜찮습니다. 생각나는 대로 나열해주세요.

예시:

- 이메일 확인 및 답장
- 블로그 글 주제 리서치
- 블로그 초안 작성
- SNS 게시물 작성
- 미팅 일정 조율
- 제안서 작성
- 경쟁사 분석
- 고객 문의 응대
- 데이터 정리 및 리포트 작성
- 코드 리뷰
- ...

```

### 질문 3: 그 중에서 반복되는 패턴이 있는 일은?

```

위에서 적은 업무 중, 아래 기준에 해당하는 것을 3~5개 골라주세요:

✅ 선정 기준:

- 매번 비슷한 방식으로 하는 일
- 할 때마다 "또 이거 해야 하네" 느끼는 일
- 시간은 걸리는데 창의성은 덜 필요한 일
- 템플릿이나 패턴이 있는 일

❌ 제외 기준:

- 매번 완전히 다른 판단이 필요한 일
- 대면 미팅, 전화 통화 등 실시간 소통
- 물리적 행동이 필요한 일

```

### 분석: 반복 작업 유형 분류

사용자가 반복 작업을 알려주면, 아래 유형으로 분류한다:

| 유형 | 설명 | AI 위임 적합도 | 예시 |

|------|------|---------------|------|

| **생성형** | 콘텐츠/문서 만들기 | ⭐⭐⭐ 높음 | 블로그 초안, 이메일 작성, 제안서 |

| **분석형** | 데이터 보고 정리하기 | ⭐⭐⭐ 높음 | 경쟁사 분석, 리포트 작성, 리서치 |

| **변환형** | A를 B로 바꾸기 | ⭐⭐⭐ 높음 | 회의록→액션아이템, 긴글→요약 |

| **분류형** | 카테고리 나누기 | ⭐⭐⭐ 높음 | 문의 분류, 우선순위 정하기 |

| **검증형** | 체크/검토하기 | ⭐⭐ 중간 | 맞춤법 검사, 코드 리뷰, 팩트체크 |

| **조율형** | 일정/리소스 맞추기 | ⭐ 낮음 | 미팅 조율 (사람 개입 필요) |

### 결과: 자동화 후보 제시

분석이 끝나면 아래 형식으로 결과를 제시한다:

```markdown

## 🎯 자동화 후보 분석 결과

### 당신의 역할

> [사용자가 정의한 역할]

### 발견된 반복 작업

| 작업 | 유형 | AI 적합도 | 예상 효과 |

|------|------|----------|----------|

| 블로그 초안 작성 | 생성형 | ⭐⭐⭐ | 주 3시간 → 30분 |

| 경쟁사 리서치 | 분석형 | ⭐⭐⭐ | 주 2시간 → 20분 |

| 이메일 답장 초안 | 생성형 | ⭐⭐⭐ | 일 30분 → 5분 |

### 추천 자동화 우선순위

1. **1순위: 블로그 초안 작성**

- 이유: 매주 반복, 패턴 명확, 시간 절감 효과 큼

- 예상 구조: CEO + 콘텐츠 기획자 + SCV

2. **2순위: 경쟁사 리서치**

- 이유: 수집→분석→정리 패턴 자동화 가능

- 예상 구조: CEO + 리서처 + SCV(api-caller)

3. **3순위: 이메일 답장 초안**

- 이유: 단순하지만 누적 시간 큼

- 예상 구조: CEO 단독 + SCV

```

### 확인: 어떤 것부터 설계할까요?

```

위 분석을 바탕으로, 어떤 작업을 먼저 자동화하고 싶으신가요?

1. 1순위 추천대로 진행

2. 다른 작업 선택 (번호 또는 이름)

3. 여러 개 한번에 설계

선택해주시면 상세 설계를 시작하겠습니다.

```

## 0단계 완료 후: 대략적 설계안 미리보기

사용자가 자동화할 작업을 선택하면, 상세 설계 전에 **대략적인 설계안**을 먼저 보여준다.

```markdown

## 📋 [작업명] 자동화 설계안 (미리보기)

### 한눈에 보기

- *입력**: [예: 블로그 주제 키워드]
- *출력**: [예: 발행 가능한 블로그 초안]
- *예상 조직**: [예: CEO + 콘텐츠 기획자 + SCV 2개]

### 워크플로우 요약

```

[입력] → 리서치(SCV) → 기획(직원) → 초안작성(직원) → 검토(CEO) → [출력]

```

### 예상 소요 시간 변화

| 단계 | 현재 (수동) | 자동화 후 |

|------|------------|----------|

| 리서치 | 30분 | 자동 |

| 기획 | 20분 | 5분 (검토만) |

| 초안 | 1시간 | 자동 |

| 검토 | 10분 | 10분 |

| **합계** | **2시간** | **15분** |

- --

이 방향으로 상세 설계를 진행할까요?

- ✅ 네, 진행해주세요 → 1단계로 이동
- 🔄 조금 다르게 → 어떤 부분을 변경할까요?
- ❌ 다른 작업 선택 → 0단계로 돌아가기

```

## 산출물

최종적으로 **하나의 통합 설계서 문서(md 파일)**를 만든다.

이 문서는 Claude Code에서 구체적인 구현을 할 때 참조할 **계획서** 역할을 한다.

### 설계서에 포함될 내용

1. **작업 컨텍스트**

- 배경, 목적, 범위, 입출력 정의, 제약조건, 용어 정의

2. **조직도**

- 조직 구조 유형 (단일 CEO vs 직원 분리)

- 이 프로젝트에 필요한 직원 목록

- 각 직원의 역할, 책임, 권한, 트리거 조건

- 보고 체계 (누가 누구에게 보고하는가)

3. **워크플로우**

- 단계별 흐름도

- 각 단계의 담당 직원

- 각 단계의 처리 주체 (직원 vs SCV)

- 직원이 직접 수행할 판단/의사결정 단계 명시

- 분기 조건, 상태 전이

- 각 단계의 성공 기준 및 검증 방법

- 실패 시 처리 방식

4. **구현 스펙** (구조 개요만)

- 조직 구조 (단일 CEO vs 직원 분리)

- 폴더 구조

- CEO(CLAUDE.md) 핵심 섹션 목록

- 직원별 파일 목록 및 핵심 섹션

- SCV 스크립트 목록

- 작업 단계별 처리 방식 (직원 판단 vs SCV 스크립트)

- 주요 산출물 파일 형식

## 핵심 개념: 회사 조직 모델

에이전트 시스템을 **가상의 회사**로 설계한다. 각 역할은 명확한 책임과 권한을 가진다.

> **참고**: 이것은 설계를 돕기 위한 메타포입니다. 단순한 프로젝트는 CEO 혼자서 충분합니다. 복잡하게 만들 필요 없습니다.

### 직원 유형

| 유형 | 역할 | 특징 | 예시 |

|------|------|------|------|

| **C-Level** | 의사결정, 최종 승인, 전략 | 다른 직원의 산출물을 검토/승인 | CEO, CFO, COO |

| **전문가** | 도메인 전문 업무 수행 | 특정 영역의 판단과 창작 | 콘텐츠 기획자, 마케터, 개발자, 리서처 |

| **SCV** | 반복적/결정론적 작업 | 판단 없이 정해진 규칙대로 실행 | 파일 처리, API 호출, 데이터 파싱 |

### 역할 분담 기준

| 직원이 직접 수행 (LLM) | SCV가 처리 (스크립트) |

|------------------------|----------------------|

| 분류, 의사결정, 우선순위 판단 | 파일 I/O, 데이터 파싱 |

| 품질 평가, 정성적 분석 | 외부 API 호출 |

| 컨텍스트 기반 추론 | 반복 처리, 집계 |

| 자연어 생성/요약/기획 | 정적 분석, 테스트 실행 |

| 전략 수립, 창의적 작업 | 포맷 변환, 템플릿 적용 |

### SCV vs 전문가

| SCV | 전문가 |

|-----|--------|

| 도구/기능 단위 (작음) | 역할/책임 단위 (큼) |

| 여러 직원이 공유 가능 | 특정 워크플로우 전용 |

| 판단 없이 실행만 | 판단과 창작 수행 |

| 예: `file-handler`, `api-caller` | 예: `콘텐츠 기획자`, `마케터` |

## 조직 구조 선택

### 단일 CEO (기본) ⭐ 대부분의 프로젝트는 이것으로 충분

CEO(CLAUDE.md) 혼자서 모든 작업을 순차 처리한다.

- *적합한 경우**:
- 워크플로우가 단순하고 지침이 짧을 때
- 도메인 지식이 한 영역에 집중될 때
- 전체 지침이 컨텍스트 윈도우에 부담되지 않을 때

```

/project-root

├── CLAUDE.md          # CEO가 모든 작업 처리

├── /.claude

│   └── /scv           # SCV만 사용

└── /output

```

### 직원 분리

CEO가 오케스트레이터 역할을 하고, 전문 직원들에게 작업을 위임한다.

- *적합한 경우**:
- 독립적인 작업 블록이 명확히 구분될 때
- 각 작업에 서로 다른 도메인 지식이 필요할 때
- 지침이 길어서 상시 로드가 비효율적일 때 (컨텍스트 윈도우 최적화)

```

/project-root

├── CLAUDE.md          # CEO (오케스트레이터)

├── /.claude

│   ├── /staff         # 직원들

│   └── /scv

└── /output

```

## 표준 직원 카탈로그

프로젝트 성격에 따라 아래에서 필요한 직원을 선택한다.

### C-Level (경영진)

| 직책 | 역할 | 담당 업무 |

|------|------|----------|

| **CEO** | 최종 의사결정자 | 전체 방향 설정, 최종 산출물 승인, 품질 게이트 |

| **CFO** | 리소스 관리자 | 비용 분석, 효율성 검토, 예산 제약 확인 |

| **COO** | 운영 총괄 | 워크플로우 조율, 병목 해결, 프로세스 최적화 |

| **CTO** | 기술 총괄 | 기술 의사결정, 아키텍처 검토, 기술 품질 승인 |

| **CMO** | 마케팅 총괄 | 마케팅 전략 승인, 브랜드 일관성 검토 |

### 전문가 (실무진)

| 직책 | 역할 | 담당 업무 |

|------|------|----------|

| **콘텐츠 기획자** | 콘텐츠 설계 | 아이디어 발굴, 콘텐츠 구조 설계, 스토리텔링 |

| **마케터** | 마케팅 실행 | 마케팅 전략 수립, 채널 전략, 캠페인 기획 |

| **카피라이터** | 문구 작성 | 헤드라인, 바디카피, CTA 작성 |

| **개발자** | 코드 작성 | 기능 구현, 버그 수정, 코드 리뷰 |

| **디자이너** | 시각 설계 | UI/UX 설계, 비주얼 가이드라인 |

| **리서처** | 조사/분석 | 시장 조사, 경쟁사 분석, 데이터 수집 |

| **에디터** | 교정/편집 | 맞춤법, 문체 통일, 팩트체크 |

| **QA** | 품질 검증 | 테스트 수행, 버그 리포트, 품질 기준 검증 |

| **PM** | 프로젝트 관리 | 일정 관리, 리소스 배분, 이해관계자 소통 |

### SCV (기본 작업자)

| 이름 | 역할 |

|------|------|

| **file-handler** | 파일 읽기/쓰기, 폴더 관리 |

| **api-caller** | 외부 API 호출, 응답 파싱 |

| **data-parser** | 데이터 추출, 변환, 정제 |

| **formatter** | 출력 포맷 변환 (JSON→MD 등) |

| **validator** | 스키마 검증, 필수값 체크 |

| **aggregator** | 데이터 집계, 통계 계산 |

| **test-runner** | 테스트 실행, 결과 수집 |

| **linter** | 코드 정적 분석 |

| **builder** | 빌드, 컴파일, 패키징 |

| **참조 자료** | 필요한 도메인 지식 | "/references/brand-guide.md" |

## 조직 설계 원칙

### 1. 필요한 직원만 채용한다

모든 프로젝트에 모든 직원이 필요하지 않다. 작업 복잡도에 따라:

| 규모 | 구성 예시 |

|------|----------|

| **Small** | CEO 단독 + SCV |

| **Medium** | CEO + 전문가 2~3명 + SCV |

| **Large** | CEO + C-Level 1~2명 + 전문가 3~5명 + SCV |

| **Enterprise** | C-Level 다수 + 전문가 팀 + SCV |

### 2. 직원을 분리하는 이유

직원을 여러 명으로 분리하는 것은 **컨텍스트 윈도우 최적화**를 위함이다:

- **문제**: 모든 지침을 CLAUDE.md 하나에 넣으면 항상 전체를 로드해야 함 → 비효율
- **해결**: 역할별로 직원을 분리하면, 필요한 시점에 해당 직원의 지침만 로드

### 3. 보고 체계를 명확히 한다

```

CEO (최종 승인)

├── CFO (비용 검토)

├── CMO (마케팅 검토)

│    ├── 콘텐츠 기획자

│    ├── 마케터

│    └── 카피라이터

└── CTO (기술 검토)

├── 개발자

└── QA

```

- 직원 간 직접 호출 금지 → CEO를 통해 조율
- 예외: 같은 C-Level 산하의 전문가끼리는 직접 협업 가능

### 4. SCV는 직원의 도구다

SCV는 독립적으로 판단하지 않는다. 항상 직원의 지시를 받아 실행한다.

```

콘텐츠 기획자 → [기획서 작성] → SCV(file-handler) → 파일 저장

마케터 → [데이터 필요] → SCV(api-caller) → 트렌드 데이터 조회

```

## 검증 체계

### 검증 유형

| 유형 | 담당 | 적용 대상 | 예시 |

|------|------|----------|------|

| **스키마 검증** | SCV (validator) | 구조화된 데이터 | 필수 필드 존재, 타입 체크 |

| **규칙 기반** | SCV (validator) | 정량 기준 | 항목 수, 글자 수, 필수 섹션 |

| **LLM 자기 검증** | 담당 직원 본인 | 정성적 판단 | 요약 품질, 톤 적합성, 누락 여부 |

| **전문가 검토** | 다른 전문가 | 교차 검증 | 에디터가 카피라이터 작업 검토 |

| **경영진 승인** | C-Level | 전략 정합성 | 브랜드 일관성, 비용 적정성 |

| **사람 검토** | 사용자 | 고위험 최종 산출물 | 외부 발송 문서, 중요 의사결정 |

### 각 단계별 필수 정의 항목

워크플로우의 각 단계마다 다음을 명시한다:

- **처리 주체**: 직원(LLM 판단) vs SCV(스크립트)
- **성공 기준**: 이 단계의 산출물이 충족해야 할 조건
- **검증 방법**: 위 유형 중 어떤 방식으로 확인하는가
- **검증 담당**: 누가 검증하는가
- **실패 시 처리**: 아래 패턴 중 선택

### 실패 처리 패턴

| 방식 | 사용 시점 | 최대 횟수 | 예시 |

|------|----------|----------|------|

| **자동 재시도** | 단순 누락/형식 오류 | **최대 3회** | 필수 필드 누락 → 다시 생성 |

| **재작업 지시** | 품질 미달 | **최대 2회** | CEO → 콘텐츠 기획자에게 수정 요청 |

| **에스컬레이션** | 판단 불확실, 기준 모호 | - | 마케터 → CMO에게 방향 확인 |

| **스킵 + 로그** | 선택적 단계, 전체 흐름에 영향 없음 | - | 사유를 로그에 기록 후 진행 |

- *재시도 정책**:
- 같은 오류로 최대 횟수 초과 시 → 자동으로 에스컬레이션
- 에스컬레이션 후에도 해결 안 되면 → 사람 검토 요청

## 데이터 전달

### 직원 간 커뮤니케이션

| 방식 | 사용 시점 |

## 작업 컨텍스트 템플릿

설계서의 "작업 컨텍스트" 섹션 작성 시 참고:

```markdown

## 작업 컨텍스트

### 배경

이 프로젝트가 왜 필요한가? 어떤 문제를 해결하는가?

### 목적

이 에이전트 시스템이 달성해야 할 목표

### 범위

- 포함: 이 시스템이 처리하는 것
- 제외: 이 시스템이 처리하지 않는 것

### 입력 정의

| 입력 | 형식 | 설명 | 예시 |

|------|------|------|------|

| 주제 키워드 | string | 콘텐츠 주제 | "AI 트렌드 2024" |

| 타겟 오디언스 | string | 대상 독자 | "마케팅 담당자" |

### 출력 정의

| 출력 | 형식 | 설명 |

|------|------|------|

| 콘텐츠 기획서 | .md | 주제, 구조, 핵심 메시지 포함 |

| 마케팅 플랜 | .md | 채널, 일정, 예산 포함 |

### 제약조건

- 예산: 월 100만원 이내
- 일정: 기획부터 발행까지 3일 이내
- 톤앤매너: 브랜드 가이드라인 준수

### 용어 정의

| 용어 | 정의 |

|------|------|

| 콘텐츠 | 블로그 포스트, SNS 게시물, 뉴스레터 등 발행 가능한 텍스트 |

| 기획서 | 콘텐츠의 주제, 구조, 타겟을 정의한 문서 |

| 캠페인 | 특정 목표를 위한 마케팅 활동 묶음 |

```

## 1단계: 작업 파악

> **0단계를 완료한 후 진행합니다.**

사용자가 자동화할 작업을 선택하면, 아래를 확인한다. 부족한 정보가 있으면 질문한다.

| 영역 | 확인 사항 | 부족할 때 질문 예시 |

|------|----------|-------------------|

| **작업 절차** | 입력→출력까지의 단계, 분기 조건 | "A 이후에 B로 갈지 C로 갈지는 어떤 기준으로 판단하나요?" |

| **조직 구조** | CEO 단독 처리 vs 직원 분리 필요 | "이 작업을 CEO 혼자 순차 처리하면 되나요, 아니면 분리하고 싶은 역할이 있나요?" |

| **필요 인력** | 어떤 전문성이 필요한가? 누가 최종 승인하나? | "이 작업에서 최종 승인은 누가 하나요? 비용 검토도 필요한가요?" |

| **기술 스택** | 사용할 도구, API, 파일 형식 | "데이터 소스는 뭔가요? 특정 API나 도구를 이미 쓰고 있나요?" |

- *질문 원칙**
- 한 번에 최대 3개 질문
- "모르겠다" → 설계 원칙에 따라 결정하고 근거 명시
- "알아서 해줘" → 합리적 기본값 적용 후 핵심만 확인
