## 이게 뭔가요?

Claude에게 "앱 만들어줘"라고 하면 그냥 뭔가를 만들어줍니다. 근데 그게 정말 잘 만든 건지 어떻게 알까요?

**하네스(Harness)**란 Claude가 작업할 때 따르는 "작업 구조"를 말합니다. 단순히 Claude 한 명이 혼자 일하게 두는 게 아니라, 여러 역할로 나눠서 **만드는 역할**과 **검토하는 역할**이 서로 협력하게 만드는 방식입니다.

마치 회사에서 디자이너가 시안을 만들면, 클라이언트가 피드백을 주고, 그걸 반영해서 다시 만드는 과정처럼요. 혼자 만들고 스스로 "잘 됐네"라고 끝내는 것보다 훨씬 나은 결과물이 나옵니다.

> **비유:** 자기 글을 스스로 교정하는 것보다, 편집자가 따로 교정하는 게 오류를 훨씬 잘 잡는 것과 같은 원리입니다.

이 방법은 Anthropic(앤트로픽, Claude를 만든 회사)이 실제로 프론트엔드 디자인 시스템과 풀스택 앱 개발에 적용해본 경험을 정리한 글에서 소개된 내용입니다.

## 왜 알아야 하나요?

Claude에게 복잡한 작업을 맡기면 두 가지 문제가 생깁니다.

**문제 1: 맥락 창(Context Window, AI가 한 번에 기억할 수 있는 분량) 한계**

작업이 길어지면 Claude는 앞에서 한 내용을 잊기 시작합니다. 그러면 일관성이 떨어지거나, "이제 힘드니까 여기서 대충 마무리하자"는 식으로 조기 종료를 합니다.

**문제 2: 자기 평가의 한계**

"네가 만든 게 잘 된 것 같아?" 하고 물으면, Claude는 평범한 결과물도 "잘 됐습니다!"라고 대답합니다. 특히 디자인처럼 주관적인 영역에서 이 경향이 심합니다.

이 두 문제를 해결하는 게 바로 **멀티 에이전트 하네스**입니다. 만드는 역할과 평가하는 역할을 분리하면, 평가자는 결과물에 대한 애착 없이 냉정하게 피드백을 줄 수 있습니다.

## 어떻게 하나요?

### 방법 1: 생성기 + 평가기 두 역할 분리

기본 구조는 두 에이전트(AI 역할)로 나뉩니다:

| 역할 | 하는 일 |
|------|---------|
| **생성기(Generator)** | 실제 결과물(코드, 디자인 등)을 만듦 |
| **평가기(Evaluator)** | 결과물을 직접 테스트하고 점수와 피드백을 줌 |

이 구조는 GAN(Generative Adversarial Network, 생성형 적대 신경망)에서 영감을 받은 것입니다. GAN은 AI 이미지 생성에서 쓰는 기법으로, 만드는 AI와 가짜를 감별하는 AI가 서로 경쟁하면서 품질을 높이는 방식입니다. 그 아이디어를 Claude 에이전트 구조에 적용한 것입니다.

<div class="example-case">
<strong>예시: 프론트엔드 디자인 시스템 만들기</strong>

Anthropic은 디자인 품질을 4가지 기준으로 평가하도록 설계했습니다:

1. **디자인 품질** — 전체적으로 통일된 느낌인가?
2. **독창성** — AI가 흔히 뽑아내는 뻔한 패턴을 피했는가?
3. **제작 기술** — 글자 크기, 여백, 색상 조화가 맞는가?
4. **기능성** — 실제로 쓰기 편하고 이해하기 쉬운가?

생성기가 디자인을 만들면, 평가기가 Playwright MCP(브라우저 자동화 도구 연결 기능)로 실제 화면을 직접 클릭해보면서 각 항목에 점수를 매기고 피드백을 돌려줍니다. 생성기는 그 피드백을 받아서 다시 수정합니다.

</div>

### 방법 2: 플래너 + 생성기 + 평가기 세 역할 구조

풀스택 앱(앞단 화면 + 뒷단 서버까지 갖춘 앱) 개발에는 역할이 하나 더 늘어납니다:

| 역할 | 하는 일 |
|------|---------|
| **플래너(Planner)** | 짧은 아이디어를 상세 기획서로 확장 |
| **생성기(Generator)** | 기획서를 보고 실제 코드를 스프린트(Sprint, 짧은 개발 단위) 단위로 구현 |
| **평가기(Evaluator)** | 완성된 앱을 실제로 클릭해보고 버그와 미완성 기능을 찾아냄 |

<div class="example-case">
<strong>예시: 게임 제작 도구 만들기</strong>

같은 목표로 두 방식을 비교했습니다:

- **단일 에이전트 방식** (Claude 혼자): 20분, 약 9달러 소요. 인터페이스는 있지만 게임이 실제로 작동하지 않음.
- **3역할 하네스 방식**: 6시간, 약 200달러 소요. 스프라이트(캐릭터 이미지) 애니메이션, AI 레벨 디자인, 사운드 기능까지 포함된 완성도 높은 결과물.

비용과 시간이 더 들지만, 결과물의 품질 차이가 극적으로 달랐습니다.

</div>

## 실전 예시

<div class="example-case">
<strong>실전 케이스: 디지털 음악 작업 도구(DAW) 만들기</strong>

Anthropic이 실제로 이 구조로 DAW(Digital Audio Workstation, 컴퓨터로 음악을 편집하는 도구)를 만들었습니다.

처음에는 스프린트 구조(기능을 여러 단계로 나눠서 순서대로 개발하는 방식)를 도입했지만, Claude Opus 4.6 모델의 성능이 좋아지면서 굳이 단계를 나누지 않아도 됐습니다. 구조를 단순화해도 평가기가 여전히 핵심 기능 누락을 잡아냈습니다.

평가기가 잡아낸 문제 예시:
- "클립(음악 조각)을 드래그해서 이동하는 기능이 없음"
- "신시사이저(전자 악기) 설정 화면이 빠짐"
- "효과(이펙트) 편집기가 없음"

이런 세부 기능은 생성기 혼자 작업할 때 놓치기 쉬운 것들입니다.

</div>

## 주의할 점

**1. 비용과 시간이 훨씬 더 든다**

멀티 에이전트 하네스는 혼자 작업하는 것보다 비용과 시간이 대폭 늘어납니다. 단순한 작업에는 오히려 과한 구조입니다. 결과물의 완성도가 정말 중요한 복잡한 작업에만 도입하는 게 좋습니다.

**2. 모델이 좋아질수록 하네스 구조는 단순해진다**

Claude 모델이 업그레이드될수록 복잡한 하네스 없이도 좋은 결과가 나옵니다. 하지만 "더 좋은 모델이 더 어려운 작업을 가능하게 만든다"는 특성 때문에, 도전적인 작업 수준도 함께 올라갑니다. 즉, 하네스의 필요성이 완전히 사라지지는 않습니다.

**3. 평가기에게 실제 테스트 환경을 줘야 한다**

평가기가 코드만 읽고 판단하면 생성기와 비슷한 맹점이 생깁니다. 평가기가 Playwright MCP 같은 도구로 실제 화면을 직접 조작해봐야 진짜 문제를 잡아낼 수 있습니다.

## 정리

- Claude에게 복잡한 작업을 맡길 때, **만드는 역할**과 **평가하는 역할**을 분리하면 품질이 크게 올라간다.
- 평가기는 코드를 검토하는 게 아니라 **실제 결과물을 직접 사용해보면서** 문제를 찾는 것이 핵심이다.
- 비용과 시간이 더 들지만, 완성도가 중요한 복잡한 앱 개발에는 충분히 가치 있는 투자다.
