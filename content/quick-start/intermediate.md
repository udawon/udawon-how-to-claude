## 시나리오 A: 기존 프로젝트에 합류

> "이 프로젝트 인수인계받았는데 뭐가 뭔지 모르겠어" 상황.

### 1단계: 전체 파악부터

```bash
cd 기존-프로젝트-폴더
claude
```

"<span class="keyword-highlight">이 프로젝트 전체 구조를 파악해줘.</span>
 어떤 기술을 쓰고, 주요 기능이 뭔지, 폴더 구조는 어떻게 됐는지
 신입 개발자에게 설명하듯이 알려줘."

비유: 새 직장 첫날
     "우리 팀이 뭘 하는 팀이에요?" 먼저 물어보는 것
     코드부터 보면 길을 잃음

---

### 2단계: 핵심 코드 흐름 추적

"사용자가 예약을 하면 어떤 파일들이 실행되는지
 처음부터 끝까지 추적해줘."

"인증은 어떻게 처리되고 있어?"

"DB 스키마가 어떻게 생겼어? ERD로 설명해줘."

> 한 번에 다 물어보지 말고, 핵심 흐름부터 하나씩 파악하세요.
> 전체를 한 번에 이해하려 하면 오히려 헷갈립니다.

---

### 3단계: 서브에이전트로 병렬 탐색

큰 프로젝트는 서브에이전트를 활용해 빠르게 파악합니다.

"서브에이전트들을 써서 병렬로 탐색해줘:
 - 에이전트 1: 프론트엔드 구조와 주요 컴포넌트
 - 에이전트 2: 백엔드 API 구조와 DB 스키마
 - 에이전트 3: 최근 3개월 git 로그에서 주요 변경사항

 각자 탐색 후 종합 보고서 만들어줘."

서브에이전트가 탐색하는 동안
메인 <span class="keyword-highlight">컨텍스트</span>가 오염되지 않음
요약본만 돌아옴 = 효율적

> **서브에이전트란?**
> Claude가 내부적으로 "분신"을 만들어 각자 다른 작업을 동시에 수행하는 기능입니다.
> 메인 대화에서는 결과 요약만 받으므로 컨텍스트 낭비가 없습니다.

---

### 4단계: CLAUDE.md 생성 (파악 후)

"이 프로젝트의 주요 특성, 기술 스택, 주의사항을
 <span class="keyword-highlight">CLAUDE.md로 정리해줘.</span>
 내가 나중에 이 프로젝트 작업할 때 Claude가
 놓치면 안 되는 것들 위주로."

---

### 5단계: 작은 것부터 시작

처음부터 큰 기능을 건드리지 마세요.

"기존 코드 스타일과 패턴을 파악했으니,
 이 작은 버그부터 고쳐줘: \[버그 설명\]
 <span class="keyword-highlight">기존 패턴을 최대한 따라줘.</span>"

### 전체 흐름

```
[전체 파악] "이 프로젝트 설명해줘"
    ↓
[핵심 흐름 추적] 주요 기능 동작 방식 이해
    ↓
[서브에이전트] 병렬 탐색으로 빠른 파악
    ↓
[CLAUDE.md] 파악한 내용 정리
    ↓
[작은 것부터] 기존 패턴 따라 시작
```

## 시나리오 B: 복잡한 기능 구현

> 이미 프로젝트에 익숙하고, 어려운 기능을 구현해야 하는 상황.

### 전략 1: 분할 정복

한 번에 큰 기능을 구현하면 Claude도 헷갈립니다.

```
❌ "결제 시스템 전체를 구현해줘"

✅ "결제 시스템을 구현할 건데, 단계별로 나눠서 진행하자.
   1단계: 장바구니 → 결제 요청 흐름 설계
   2단계: PG사 연동 API 구현
   3단계: 결제 완료 후 처리 로직
   4단계: 실패/취소 처리
   먼저 1단계 설계부터 시작해줘."
```

---

### 전략 2: Worktree로 기능 격리

```bash
# 복잡한 기능은 Worktree에서 격리해서 개발
claude --worktree feature-payment

# 메인 브랜치에 영향 없이 실험 가능
# 완료되면 PR로 병합
```

---

### 전략 3: ultrathink로 아키텍처 결정

```
"ultrathink - 이 결제 시스템에서 PG사 응답이 지연되면
 어떻게 처리해야 하지? 타임아웃 전략과 재시도 로직,
 그리고 중복 결제 방지 방법을 분석해줘."
```

복잡한 의사결정에서는 ultrathink를 적극 활용하세요.
Claude가 더 체계적이고 깊이 있는 분석을 제공합니다.

---

### 전략 4: Writer-Reviewer 패턴 강화

중급 수준에서는 리뷰를 더 구체적으로 요청합니다:

```
"이 코드를 리뷰해줘. 특히:
 1. 동시성 문제 (race condition)
 2. 에러 전파 경로 — 예외가 어디서 잡히는지
 3. 이 설계가 트래픽 10배 증가해도 버틸 수 있는지
 4. 기존 코드 패턴과의 일관성"
```

## 중급에서 자주 쓰는 프롬프트

| 상황 | 프롬프트 |
|------|---------|
| 레거시 코드 이해 | "이 함수가 왜 이렇게 작성됐는지 히스토리 추적해줘" |
| 리팩토링 범위 결정 | "이 코드를 리팩토링하면 영향 받는 파일 목록을 보여줘" |
| 성능 병목 탐색 | "ultrathink - 이 API가 느린 이유를 분석하고 개선안을 제시해줘" |
| 의존성 파악 | "이 모듈을 삭제하면 어디가 깨지는지 확인해줘" |
| 테스트 작성 | "이 함수의 엣지 케이스를 분석하고 테스트 코드를 작성해줘" |
