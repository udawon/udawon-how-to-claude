## 한 줄 요약

- **서브 에이전트** = 인턴에게 심부름 시키기 (결과만 받음)
- **팀 에이전트** = 프로젝트 팀 꾸려서 회의하기 (서로 대화하며 협업)

> 각 기능의 상세 설명은 별도 문서를 참고하세요:
> - [서브 에이전트 완전 가이드](sub-agents.md) — 개념, 만들기, 설정, 활용 패턴
> - [팀 에이전트 완전 가이드](agent-teams.md) — 개념, 팀 구성, 소통, 활용 예시

## 핵심 차이 한눈에 보기

|  | 서브 에이전트 | 팀 에이전트 |
|--|------------|-----------|
| **비유** | 인턴에게 심부름 | 프로젝트 팀 회의 |
| **소통 방식** | 나한테만 보고 | 팀원끼리 직접 대화 |
| **작업 공간** | 팀장 안에서 동작 | 각자 독립된 공간 |
| **관리** | 내가 모든 작업 관리 | 팀이 알아서 조율 |
| **비용** | 저렴 | 비쌈 (팀원 수만큼) |
| **상태** | 정식 기능 | 실험적 (기본 꺼져있음) |

### 그림으로 보기

```
서브 에이전트:

  [나] → [Claude 팀장]
              ├→ [인턴 A] → 결과 보고
              ├→ [인턴 B] → 결과 보고     (A, B, C는 서로 모름)
              └→ [인턴 C] → 결과 보고

팀 에이전트:

  [나] → [Claude 팀장]
              ├── [팀원 A] ←→ [팀원 B]     (서로 직접 대화)
              ├── [팀원 B] ←→ [팀원 C]     (발견 내용 공유)
              └── [팀원 A] ←→ [팀원 C]     (반박도 가능)
```

## 선택 가이드

### 3초 판단법

```
작업이 들어왔을 때 이렇게 판단:

1. 간단한 수정이야?
   → 메인 대화에서 직접 처리
   예: "글씨 크기 키워줘"

2. 조사하고 결과만 알면 돼?
   → 서브 에이전트
   예: "이 프로젝트 파일 전부 찾아줘"

3. 여러 관점에서 토론이 필요해?
   → 팀 에이전트
   예: "보안/속도/기능 관점에서 동시에 점검해줘"
```

### 상황별 선택표

| 상황 | 뭘 쓸까 | 비유 |
|------|---------|------|
| 제목 색상 바꿔줘 | 메인 대화 | 직접 하기 |
| 프로젝트 파일 전부 찾아봐 | 서브 에이전트 | 인턴 심부름 |
| 3가지를 동시에 조사해 | 서브 에이전트 x3 | 인턴 3명 투입 |
| 테스트 돌리고 실패한 것만 알려줘 | 서브 에이전트 | 인턴이 요약 보고 |
| 서로 맞춰가면서 만들어줘 | 팀 에이전트 | 팀 회의 |
| 원인 모를 문제 토론해줘 | 팀 에이전트 | 의사 합동 진찰 |
| 여러 관점에서 리뷰해줘 | 팀 에이전트 | 전문가 합동 심사 |

### 핵심 질문 하나

> **"팀원끼리 서로 대화할 필요가 있나?"**
>
> - **No** → 서브 에이전트 (각자 결과만 보고하면 됨)
> - **Yes** → 팀 에이전트 (서로 발견 내용을 공유하고 반박해야 함)

## 비용 비교

| 방식 | 비용 | 비유 |
|------|------|------|
| 메인 대화 | 기본 | 직원 1명 인건비 |
| 서브 에이전트 (Haiku) | 기본의 30~50% | 파트타임 인턴 |
| 서브 에이전트 (일반) | 기본과 비슷 | 정규직 인턴 |
| 팀 에이전트 3명 | 기본의 3~4배 | 팀원 3명 인건비 |
| 팀 에이전트 5명 | 기본의 5~7배 | 팀원 5명 인건비 |

```
비용을 아끼고 싶을 때:
  서브 에이전트 + Haiku 모델
  → "파일 찾기"처럼 단순한 작업을 저비용으로 처리

비용보다 품질이 중요할 때:
  팀 에이전트 3명
  → 비용은 3배 이상이지만 서로 검증하므로 더 정확한 결과
```

## 실전 예시 비교

### 같은 작업, 다른 접근

**과제: "프로젝트의 보안 취약점을 찾아줘"**

#### 서브 에이전트로 하면

```
나: "보안 취약점 찾아줘"
→ 서브 에이전트 1개가 혼자 전체 조사
→ "취약점 5개 발견" 보고

장점: 빠르고 저렴
단점: 한 관점에서만 봄, 놓칠 수 있음
```

#### 팀 에이전트로 하면

```
나: "보안 팀을 만들어서 취약점을 찾아줘"
→ 인증 담당: "토큰 만료 처리 없음"
→ 입력 담당: "SQL 인젝션 가능성"
→ API 담당: "인증 없는 엔드포인트 3개"
→ 입력 담당: "API 담당이 찾은 엔드포인트에도 내가 발견한 인젝션 위험이 있어"
→ 팀장: 종합 보안 보고서 작성

장점: 다각도 검토, 교차 발견 가능
단점: 비용 3배, 시간도 더 걸림
```

## 조합해서 쓰기

서브 에이전트와 팀 에이전트는 **동시에 활용**할 수 있습니다.

```
예시: 대규모 리팩터링

1단계 (서브 에이전트):
  "전체 코드를 조사해서 리팩터링 대상 목록을 만들어줘"
  → 서브 에이전트가 빠르게 조사 후 목록 전달

2단계 (팀 에이전트):
  "이 목록을 기반으로 리팩터링 팀을 만들어줘.
   모듈별로 팀원 배정하고 서로 인터페이스를 맞춰가며 작업해."
  → 팀 에이전트가 협업하며 리팩터링 수행

비유:
  1단계: 인턴이 "이 창고에서 정리할 물건 목록" 조사
  2단계: 정리 팀이 목록 보고 역할 나눠서 함께 정리
```

## 요약 정리

| 기준 | 서브 에이전트 | 팀 에이전트 |
|------|-------------|-------------|
| **핵심 가치** | 빠르고 저렴한 위임 | 다각도 협업과 토론 |
| **소통** | 일방향 (인턴→팀장) | 양방향 (팀원↔팀원) |
| **적합한 작업** | 조사, 검색, 요약, 테스트 | 리뷰, 토론, 분담 개발, 디버깅 |
| **비용** | 낮음~보통 | 높음 |
| **설정 난이도** | 쉬움 (기본 내장) | 보통 (활성화 필요) |
| **안정성** | 정식 기능 | 실험적 기능 |
