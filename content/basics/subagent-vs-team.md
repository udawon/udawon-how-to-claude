## 한 줄 요약

- **서브 에이전트** = 인턴에게 심부름 시키기 (결과만 받음)
- **팀 에이전트** = 프로젝트 팀 꾸려서 회의하기 (서로 대화하며 협업)

## 핵심 차이 한눈에 보기

|  | 서브 에이전트 | 팀 에이전트 |
|--|------------|-----------|
| **비유** | 인턴에게 심부름 | 프로젝트 팀 회의 |
| **소통 방식** | 나한테만 보고 | 팀원끼리 직접 대화 |
| **관리** | 내가 모든 작업 관리 | 팀이 알아서 조율 |
| **비용** | 저렴 | 비쌈 (팀원 수만큼) |
| **상태** | 정식 기능 | 실험적 (기본 꺼져있음) |

## 서브 에이전트 = 심부름꾼

### 어떻게 동작하나?

```
나 (메인)           ->  인턴 A: "이 자료 찾아와"
                    ->  인턴 B: "저 자료 찾아와"
인턴 A가 결과 보고  <-
인턴 B가 결과 보고  <-

인턴 A와 B는 서로 모름. 나한테만 각각 보고.
```

### 내장 서브 에이전트

| 이름 | 역할 | 비유 |
|------|------|------|
| **Explore** | 프로젝트 파일 검색/분석 (빠름) | 서류 찾아오는 인턴 |
| **Plan** | 계획 세우기 전 조사 | 시장조사 하는 인턴 |
| **General** | 복잡한 작업 처리 | 경력직 인턴 |

### 언제 쓰면 좋나?

#### 1. 자료 조사를 시킬 때
```
"이 프로젝트에서 '결제' 관련된 파일을 전부 찾아서 정리해줘"

-> 서브 에이전트가 프로젝트 전체를 뒤져서 목록을 가져옴
   나는 기다렸다가 결과만 확인

비유: "인턴아, 서류함에서 '결제' 관련 서류 전부 찾아와"
```

#### 2. 여러 조사를 동시에 시킬 때
```
"서브 에이전트 3개 띄워서 동시에 조사해줘:
 1. 경쟁사 A 사이트 구조 분석
 2. 경쟁사 B 사이트 구조 분석
 3. 경쟁사 C 사이트 구조 분석"

-> 3명이 동시에 각자 조사하고, 나한테 결과만 보고
   순서대로 하면 30분 걸릴 걸 10분에 끝남

비유: "인턴 3명한테 각각 다른 가게 둘러보고 오라고 시키기"
```

#### 3. 긴 작업의 결과만 필요할 때
```
"전체 테스트를 실행하고 문제가 있는 것만 알려줘"

-> 서브 에이전트가 수백 줄의 테스트 결과를 처리
   나한테는 "3건의 문제 발견" 요약만 전달

비유: "비서한테 100페이지 보고서 읽고 핵심 3줄로 요약해달라고 하기"
```

### 나만의 서브 에이전트 만들기

Claude Code 대화창에서 `/agents` 명령어로 만들 수 있습니다.

```
따라하기:
  1. 터미널에서 claude 입력하여 Claude Code 실행
  2. Claude Code 대화창에서 /agents 입력
  3. "Create new agent" 선택
  4. 에이전트 설명 입력
     예: "코드를 검토하고 문제점을 알려주는 검토 전문가"
  5. 저장 -> Claude가 자동으로 설정 파일을 생성
```

**예시 케이스:**
```
자주 반복하는 작업이 있다면 서브 에이전트로 만들어두면 편함:

"매번 코드 수정 후 검토해줘"라고 말하는 대신
-> code-reviewer 서브 에이전트를 한 번 만들어두면
   Claude가 코드 수정 후 자동으로 검토 에이전트에 위임

비유: 매번 "서류 검토해주세요"라고 부탁하는 대신
     "검토 담당자"를 지정해두는 것
```

## 팀 에이전트 = 프로젝트 팀

### 어떻게 동작하나?

```
팀장 (리더)
 ├── 디자이너 (팀원 A)
 ├── 개발자 (팀원 B)  <-- "디자이너, 이 버튼 크기 맞춰봐"
 └── QA (팀원 C)      <-- "개발자, 이 부분 오류 있어"

팀원끼리 직접 대화하며 협업. 팀장은 전체를 조율.
```

### 활성화 방법 (기본 꺼져있음)

```
따라하기:
  1. Claude Code 대화창에서 /config 입력
  2. 또는 직접 설정 파일을 수정:
     - 파일 위치: ~/.claude/settings.json
       (Windows: C:\Users\내이름\.claude\settings.json)
     - 메모장으로 열어서 아래 내용을 추가하고 저장

비유: 스마트폰에서 실험적 기능을 '설정 > 개발자 옵션'에서 켜는 것
```

설정 파일에 아래 내용을 추가합니다:
```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### 언제 쓰면 좋나?

#### 1. 여러 관점에서 검토가 필요할 때
```
"이 프로젝트를 점검할 팀을 만들어줘.
 - 1명은 안전한지 점검
 - 1명은 빠른지 점검
 - 1명은 잘 작동하는지 점검
 서로 발견한 문제를 공유해."

보안 담당: "로그인할 때 비밀번호 확인이 약해"
속도 담당: "그 로그인 화면 뜨는 데도 3초 걸려"
테스트 담당: "로그인 후 비밀번호 바꾸면 오류나"
보안 담당: "아! 테스트가 발견한 그 오류, 내가 말한 문제랑 같은 원인이야!"

비유: 건물 안전점검에서 전기/소방/구조 전문가가
     각자 점검하고 함께 보고서 쓰는 것
```

#### 2. 원인 모를 문제를 토론으로 찾을 때
```
"사이트가 간헐적으로 멈추는데 원인을 모르겠어.
 팀 5명을 만들어서 각각 다른 원인을 추측하고 조사하게 해.
 서로 상대방 추측이 틀렸다고 반박하게 해."

A: "서버 문제 같아"
B: "아니야, 서버 상태 정상이야. 데이터 문제 같은데"
C: "데이터 양은 적어. 외부 결제 서비스가 느린 것 같아"
A: "맞다, 결제 페이지에서만 느리니까 C 말이 맞을 수 있어"
D: "결제 서비스 응답 시간 확인해봤더니 진짜 느려!"

비유: 의사 5명이 환자 증상 놓고 토론하는 것.
     각자 다른 진단 내리고 서로 반박하다가 정답 도출
```

#### 3. 역할을 나눠서 동시에 만들 때
```
"댓글 기능을 추가할 팀을 만들어줘.
 - 1명은 댓글 화면 만들기
 - 1명은 댓글 저장/불러오기 기능
 - 1명은 욕설 필터링
 서로 맞춰가면서 만들어."

화면 담당: "댓글 입력창을 이렇게 만들었어"
저장 담당: "알겠어, 그 입력창에서 보내면 내가 저장할게"
필터 담당: "저장하기 전에 내가 먼저 욕설 체크할게"

비유: 요리팀에서 한 명은 밥, 한 명은 국, 한 명은 반찬을 동시에 만들되
     "국이 매우니까 반찬은 순하게 하자"처럼 서로 맞추는 것
```

#### 4. 설계 방향을 토론할 때
```
"SaaS 대시보드를 설계할 팀을 만들어줘.
 - UX 전문가: 사용자가 편하게 쓸 수 있는 화면 제안
 - 기술 전문가: 실제로 만들 수 있는 구조 제안
 - 비평가: 두 사람 제안의 허점 지적
 서로 토론해서 최적안 도출해."

UX: "메인 화면에 차트 4개를 한눈에 보여주자"
비평가: "한꺼번에 4개 불러오면 느릴 수 있어"
기술: "데이터를 미리 저장해두면 해결돼"
비평가: "인터넷이 끊기면? 오류 화면도 필요해"

비유: 광고 기획회의에서 카피라이터, 디자이너, 클라이언트가
     서로 피드백 주고받으며 완성하는 과정
```

## 선택 가이드

```
작업이 들어왔을 때 이렇게 판단:

1. 간단한 수정이야?
   -> 메인 대화에서 직접 처리
   예: "글씨 크기 키워줘"

2. 조사하고 결과만 알면 돼?
   -> 서브 에이전트
   예: "이 프로젝트 파일 전부 찾아줘"

3. 여러 관점에서 토론이 필요해?
   -> 팀 에이전트
   예: "보안/속도/기능 관점에서 동시에 점검해줘"
```

| 상황 | 뭘 쓸까 | 비유 |
|------|---------|------|
| 제목 색상 바꿔줘 | 메인 대화 | 직접 하기 |
| 프로젝트 파일 전부 찾아봐 | 서브 에이전트 | 인턴 심부름 |
| 3가지를 동시에 조사해 | 서브 에이전트 x3 | 인턴 3명 투입 |
| 서로 맞춰가며 만들어줘 | 팀 에이전트 | 팀 회의 |
| 원인 모를 문제 토론해줘 | 팀 에이전트 | 의사 합동 진찰 |

## 비용 비교

| 방식 | 비용 | 비유 |
|------|------|------|
| 메인 대화 | 기본 | 직원 1명 인건비 |
| 서브 에이전트 (빠른 모델) | 기본의 30~50% | 파트타임 인턴 인건비 |
| 서브 에이전트 (일반 모델) | 기본과 비슷 | 정규직 인턴 인건비 |
| 팀 에이전트 3명 | 기본의 3~4배 | 팀원 3명 인건비 |
| 팀 에이전트 5명 | 기본의 5~7배 | 팀원 5명 인건비 |

**예시 케이스:**
```
비용을 아끼고 싶을 때:
  서브 에이전트 + 빠른 모델(Haiku)
  -> "파일 찾기"처럼 단순한 작업을 저비용으로 처리

비용보다 품질이 중요할 때:
  팀 에이전트 3명
  -> 비용은 3배 이상이지만 서로 검증하므로 더 정확한 결과
```

## 팀원 화면 보는 방법

팀 에이전트를 쓰면 여러 명이 동시에 일하는데, 그 화면을 어떻게 보느냐에 따라 두 가지 모드가 있습니다.

### 기본 모드 (in-process) — 탭 전환 방식

```
화면은 하나인데, 팀원마다 탭처럼 전환해서 봅니다.

  Shift+Down  →  다음 팀원으로 전환
  (마지막 팀원 다음엔 다시 팀장으로 돌아옴)

비유: 브라우저에서 탭 여러 개 열어두고
     Alt+Tab으로 전환하는 것
```

별도 설치 없이 어디서나 작동합니다. 처음 쓴다면 이 모드로 시작하세요.

### 분할 화면 모드 (split-panes) — 모니터 여러 대 방식

```
팀원마다 별도 창이 열려서 모든 작업을 동시에 볼 수 있습니다.

비유: 주식 트레이더 책상처럼
     모니터 여러 대에 각각 다른 화면이 띄워진 것

단점: tmux 또는 iTerm2 설치가 필요합니다.
      Windows Terminal, VS Code 터미널에서는 지원 안 됨.
```

### 모드 전환 방법

```
설정 파일(settings.json)에 추가:
{
  "teammateMode": "in-process"   <- 기본 탭 방식
}

또는 실행할 때 직접 지정:
claude --teammate-mode in-process
```

## 팀원과 대화하는 단축키

| 키 | 동작 |
|---|---|
| **Shift+Down** | 다음 팀원 화면으로 전환 |
| **Enter** | 해당 팀원에게 메시지 입력 |
| **Escape** | 팀원의 현재 작업 중단 |
| **Ctrl+T** | 팀 전체 할 일 목록 보기/숨기기 |

```
예시 흐름:
  Shift+Down 눌러서 '보안 담당 팀원' 화면으로 이동
  → "이 부분 더 자세히 봐줘" 입력
  → Shift+Down 눌러서 '성능 담당 팀원' 화면으로 이동
  → 진행 상황 확인
  → Ctrl+T 눌러서 전체 할 일 목록 확인
```

## 계획 승인 워크플로우

팀원에게 중요한 작업을 맡길 때, **먼저 계획서를 받고 승인한 뒤에만 실행**하게 할 수 있습니다.

### 흐름

```
1. 팀원 생성 시 "계획 먼저 제출해" 지시

2. 팀원이 읽기 전용 모드로 조사/계획 수립
   (이 단계에서는 파일 수정 불가)

3. 팀원이 팀장에게 계획서 제출

4. 팀장이 자동으로 승인 or 반려
   → 승인: 팀원이 실제 구현 시작
   → 반려: 팀원이 계획 수정 후 재제출
```

### 핵심: 팀장이 자동으로 결재합니다

내가 직접 승인 버튼을 누르는 게 아니라, 내가 처음에 준 **기준을 팀장이 보고 자동으로 판단**합니다.

```
예시:
"인증 모듈 리팩터링해줘.
 테스트 커버리지가 포함된 계획만 승인해."

→ 팀원이 계획 제출
→ 팀장이 "테스트 커버리지 없네" → 반려
→ 팀원이 테스트 계획 추가 → 재제출
→ 팀장이 승인 → 구현 시작
```

### 비유

```
직원이 큰 예산 쓰기 전에 기안서 올리는 것과 비슷한데

차이점:
  일반 기안: 사람(나)이 직접 읽고 도장 찍음
  계획 승인: 팀장 AI가 내 기준에 맞춰 자동으로 판단
```

기준을 구체적으로 줄수록 팀장이 더 잘 판단합니다.

### 사용법

```
"인증 모듈을 리팩터링할 팀원을 만들어줘.
 변경사항 만들기 전에 계획 승인을 받도록 해줘."
```

이 한 마디로 팀원이 계획 승인 모드로 시작합니다.

## 팀 해산하기

작업이 끝나면 팀을 정리해야 합니다. 순서가 중요합니다.

```
올바른 순서:
  1. 팀원 먼저 종료: "연구원 팀원 종료해줘"
     → 팀원이 현재 작업 마무리 후 종료 (바로 꺼지지 않을 수 있음)
  2. 팀장에게 정리 요청: "팀 정리해줘"
     → 공유 리소스 삭제

잘못된 순서:
  팀원이 살아있는데 팀장이 먼저 정리 시도
  → 오류 발생. 팀장이 "아직 팀원이 작동 중"이라며 거부함
```

```
비유: 회의가 끝났을 때
     팀원들이 자리에서 일어나고 나서
     팀장이 회의실 불을 끄는 것
     팀원이 아직 앉아있는데 불 먼저 끄면 안 됨
```

## 잘 쓰는 방법

### 팀 규모는 3~5명이 적당

```
팀원 수가 많을수록:
  → 비용이 선형으로 증가 (5명이면 5배)
  → 조율하는 데 드는 시간도 늘어남
  → 일정 이상부터는 더 빨라지지 않음

대부분의 작업: 3명이면 충분
복잡한 작업: 5명까지

비유: 짜장면 배달에 배달원 10명 붙이면 오히려 문 앞이 혼잡해짐
```

### 팀원당 할 일 5~6개가 적절

```
할 일이 너무 적으면: 팀원이 금방 할 일이 없어짐
할 일이 너무 많으면: 팀원이 너무 오래 작업해서 중간 점검이 어려움

할 일 15개라면 → 팀원 3명이 적당
```

### 같은 파일을 두 팀원이 수정하면 안 됨

```
팀원 A가 login.js 수정 중
팀원 B도 login.js 수정 시작
→ 나중에 저장하는 쪽이 덮어씌움 → 한 명의 작업이 사라짐

해결책:
  각 팀원에게 담당 파일을 따로 지정
  "A는 화면 파일만, B는 서버 파일만 담당해"
```

### 중간중간 확인하기

```
팀을 믿고 오래 방치하면:
  → 엉뚱한 방향으로 작업이 진행될 수 있음
  → 뒤늦게 발견하면 되돌리기 어려움

Shift+Down으로 주기적으로 팀원 화면 확인하고
방향이 이상하면 바로 수정 지시
```

## 알려진 한계

팀 에이전트는 아직 실험적 기능입니다. 아래 한계를 미리 알아두면 당황하지 않습니다.

| 한계 | 설명 |
|------|------|
| **세션 재시작 불가** | Claude Code를 껐다 켜면 팀원이 사라짐. `/resume`으로 복구 안 됨 |
| **팀은 1개만** | 한 번에 팀 1개만 운영 가능. 새 팀 만들려면 기존 팀 해산 먼저 |
| **팀원이 팀원을 못 만듦** | 팀원이 또 다른 팀원을 만드는 건 불가. 팀장만 가능 |
| **할 일 상태 지연** | 팀원이 작업을 끝내도 완료 표시가 늦게 될 수 있음 |
| **분할 화면 제한** | tmux/iTerm2 없으면 분할 화면 불가. VS Code, Windows Terminal에서는 지원 안 됨 |

## 문제 해결

### 팀원이 화면에 안 보여요

```
→ Shift+Down을 눌러보세요.
  이미 실행 중인데 화면에 안 보이는 경우가 많습니다.

→ 요청한 작업이 너무 단순하면 Claude가 팀을 안 만들 수 있습니다.
  팀이 필요한 이유를 좀 더 명확히 설명해보세요.
```

### 권한 확인창이 너무 자주 떠요

```
팀원이 작업할 때마다 "이거 해도 될까요?" 창이 뜨면 번거롭습니다.

→ 자주 쓰는 작업(파일 읽기, 실행 등)을 미리 허용해두세요.
  팀 만들기 전에 설정해두는 게 좋습니다.
```

### 팀원이 오류에서 멈춰요

```
팀원이 문제를 만나서 더 이상 진행 안 할 때:

→ Shift+Down으로 해당 팀원 화면으로 이동
→ 직접 "이렇게 해봐" 지시를 보내거나
→ 그 팀원을 종료하고 새 팀원 투입
```

### 분할 화면이 안 열려요

```
→ tmux가 설치되어 있는지 확인:
  which tmux

→ VS Code 내장 터미널이나 Windows Terminal은 지원 안 됩니다.
  일반 터미널(macOS Terminal, iTerm2)을 써주세요.
```
