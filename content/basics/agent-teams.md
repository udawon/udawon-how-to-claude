## 한 줄 요약

**에이전트 팀 = 프로젝트 팀을 꾸려서 회의하듯 협업하는 기능**

Claude Code 하나가 혼자 하는 대신, 여러 Claude를 동시에 띄워서
팀장 아래 팀원들이 **서로 대화하고 협력하며** 일을 처리합니다.

---

## 에이전트 팀가 뭔가요?

### 비유로 이해하기

```
[나] → "이 프로젝트를 점검해줘"
  ↓
[팀장 Claude]
  ├── [보안 담당] "로그인 보안이 약해"
  ├── [성능 담당] "로그인 화면이 3초나 걸려"  ←→ 서로 대화
  └── [테스트 담당] "로그인 후 오류 발견"
                    ↕
        "아! 보안 담당이 말한 문제랑 같은 원인이야!"

→ 팀장이 모든 결과를 종합해서 나에게 보고
```

핵심 특징:
- **팀원끼리 직접 대화합니다** — 발견한 내용을 서로 공유하고 반박
- **각자 독립된 작업 공간** — 각 팀원이 자기만의 맥락(context)을 가짐
- **공유 할 일 목록** — 누가 뭘 하는지 팀 전체가 볼 수 있음
- **팀장이 전체를 조율** — 할 일 배분, 결과 종합, 품질 관리

### 서브 에이전트와의 핵심 차이

```
서브 에이전트 (인턴):
  인턴 A → 팀장에게 보고
  인턴 B → 팀장에게 보고    (A와 B는 서로 모름)
  인턴 C → 팀장에게 보고

에이전트 팀 (프로젝트 팀):
  팀원 A ←→ 팀원 B    (서로 직접 대화)
  팀원 B ←→ 팀원 C    (발견한 내용 공유)
  팀원 A ←→ 팀원 C    (서로 반박도 가능)
       ↕
     팀장이 조율
```

## 활성화 방법 (기본 꺼져있음)

에이전트 팀는 **실험적 기능**이라 직접 켜야 합니다.

### 설정 파일로 켜기

설정 파일 위치: `~/.claude/settings.json`
(Windows: `C:\Users\내이름\.claude\settings.json`)

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

**비유:** 스마트폰 설정에서 "개발자 옵션"을 켜는 것과 같습니다. 숨겨진 실험 기능을 활성화하는 것이에요.

## 팀 만들기

활성화 후, Claude Code에서 자연어로 팀을 요청하면 됩니다.

### 기본 사용법

<div class="example-case">

나: "<span class="keyword-highlight">이 프로젝트를 점검할 팀을 만들어줘</span>.
보안 담당 1명, 성능 담당 1명, 테스트 담당 1명."

→ Claude가 팀장 역할을 맡음
→ 3명의 팀원을 자동으로 생성
→ 각 팀원에게 역할 배정
→ 팀원들이 동시에 작업 시작
</div>

### 모델과 인원 지정하기

<div class="example-case">

나: "<span class="keyword-highlight">팀원 4명으로 리팩터링 팀을 만들어줘</span>.
각 팀원은 Sonnet 모델을 써줘."

비유:

팀 구성 요청서를 내는 것:

"프로젝트 점검 팀을 꾸려주세요.
- 보안 전문가 1명
- 성능 전문가 1명
- 테스트 전문가 1명
각 인원은 중급 이상으로 배치해주세요."

→ 인사팀(Claude)이 팀을 구성해서 바로 투입
</div>

## 팀 구조

에이전트 팀는 4가지 요소로 구성됩니다.

```
┌─────────────────────────────────────────┐
│                에이전트 팀               │
├─────────────────────────────────────────┤
│                                         │
│  [팀장] ─── 전체 조율, 작업 배분        │
│    │                                    │
│    ├── [팀원 A] ─── 독립 작업 공간      │
│    ├── [팀원 B] ─── 독립 작업 공간      │
│    └── [팀원 C] ─── 독립 작업 공간      │
│                                         │
│  [할 일 목록] ─── 전체 공유             │
│  [메일함]     ─── 팀원 간 메시지 전달   │
│                                         │
└─────────────────────────────────────────┘
```

| 구성요소 | 역할 | 비유 |
|---------|------|------|
| **팀장** | 팀 생성, 작업 배분, 결과 종합 | 프로젝트 매니저 |
| **팀원** | 각자 맡은 작업 수행 | 팀 구성원 |
| **할 일 목록** | 작업 상태 공유 (대기/진행/완료) | 화이트보드에 붙인 포스트잇 |
| **메일함** | 팀원 간 메시지 전달 | 사내 메신저 |

## 화면 보는 방법

팀원 여러 명이 동시에 일하는데, 그 화면을 어떻게 볼까요?

### 모드 1: 탭 전환 방식 (in-process) — 기본

하나의 화면에서 팀원을 탭처럼 전환합니다.

`[팀장]` → <kbd>Shift+Down</kbd> → `[보안 담당]` → <kbd>Shift+Down</kbd> → `[성능 담당]`
→ <kbd>Shift+Down</kbd> → `[테스트 담당]` → <kbd>Shift+Down</kbd> → `[팀장]`...

비유: 브라우저 탭을 전환하는 것
<kbd>Alt+Tab</kbd>으로 창 바꾸듯, <kbd>Shift+Down</kbd>으로 팀원을 전환

**장점:** 별도 설치 없이 어디서나 작동
**추천:** 처음 쓴다면 이 모드로 시작하세요

### 모드 2: 분할 화면 (split-panes) — 고급

팀원마다 별도 창이 열립니다.

```
┌──────────┬──────────┬──────────┐
│  팀장    │ 보안담당 │ 성능담당 │
│          │          │          │
│ (조율중) │ (검토중) │ (분석중) │
└──────────┴──────────┴──────────┘
```

비유: 주식 트레이더 모니터처럼
여러 화면에 각각 다른 팀원의 작업이 보임

**필요 조건:** tmux 또는 iTerm2 설치 필요
**제한:** VS Code 터미널, Windows Terminal에서는 지원 안 됨

> **Windows 사용자:** WSL2 안에서 tmux를 설치하면 분할 화면을 쓸 수 있습니다.
> 하지만 탭 전환 방식(in-process)으로도 충분히 사용 가능합니다.

### 모드 설정

설정 파일(settings.json)에 추가:
```json
{
  "teammateMode": "in-process"
}
```

또는 실행할 때 직접 지정:
```
claude --teammate-mode in-process
```

## 팀원 조작 단축키

| 키 | 동작 | 비유 |
|---|---|---|
| **Shift+Down** | 다음 팀원 화면으로 전환 | 탭 전환 |
| **Enter** | 해당 팀원에게 메시지 보내기 | 채팅 입력 |
| **Escape** | 팀원의 현재 작업 중단 | "잠깐 멈춰!" |
| **Ctrl+T** | 전체 할 일 목록 보기/숨기기 | 화이트보드 토글 |

**예시 흐름:**

<div class="example-case">

1. <kbd>Shift+Down</kbd> → '보안 담당' 화면으로 이동
2. "<span class="keyword-highlight">이 부분 더 자세히 봐줘</span>" 입력
3. <kbd>Shift+Down</kbd> → '성능 담당' 화면으로 이동
4. 진행 상황 확인
5. <kbd>Ctrl+T</kbd> → 전체 할 일 목록 확인
</div>

## 계획 승인 워크플로우

중요한 작업은 **계획서를 먼저 받고 승인 후 실행**하게 할 수 있습니다.

### 흐름

1. "<span class="keyword-highlight">계획 먼저 제출해</span>" 지시
2. 팀원이 조사/계획 수립 (이때 파일 수정 불가)
3. 팀원이 팀장에게 계획서 제출
4. 팀장이 기준에 따라 자동 판단
   - 승인 → 팀원이 실제 작업 시작
   - 반려 → 팀원이 계획 수정 후 재제출

### 비유

직원이 큰 예산 쓰기 전에 기안서를 올리는 것과 비슷합니다.

- 일반 기안: 내가 직접 읽고 도장 찍음
- 여기서는: 팀장 AI가 내 기준에 맞춰 자동 판단

**예시:**

<div class="example-case">

나: "<span class="keyword-highlight">인증 모듈 리팩터링해줘. 테스트 계획이 포함된 계획만 승인해.</span>"
→ 팀원: 계획서 제출 (테스트 없음)
→ 팀장: "테스트 계획 없으니 반려"
→ 팀원: 테스트 계획 추가 후 재제출
→ 팀장: "좋아, 승인" → 작업 시작
</div>

### 사용법

나: "<span class="keyword-highlight">인증 모듈을 리팩터링할 팀원을 만들어줘.
변경사항 만들기 전에 계획 승인을 받도록 해줘.</span>"

## 할 일 목록과 작업 배분

### 할 일 상태

할 일 목록 (<kbd>Ctrl+T</kbd>로 확인)

- **[대기]** 로그인 화면 검토
- **[진행중]** 결제 모듈 보안 점검 ← 보안 담당이 작업 중
- **[완료]** API 응답 속도 측정
- **[대기 - 차단됨]** 통합 테스트 ← "결제 모듈 점검"이 끝나야 시작 가능

| 상태 | 설명 | 비유 |
|------|------|------|
| 대기 | 아직 아무도 안 함 | 포스트잇이 "할 일" 칸에 있음 |
| 진행중 | 누군가 작업 중 | 포스트잇이 "진행중" 칸으로 이동 |
| 완료 | 작업 끝 | 포스트잇이 "완료" 칸으로 이동 |
| 차단됨 | 선행 작업이 안 끝남 | 앞 작업이 끝나야 시작 가능 |

### 작업 배분 방식

**방법 1:** 팀장이 직접 배분
"<span class="keyword-highlight">보안 담당은 인증 모듈, 성능 담당은 API 모듈을 봐줘</span>"

**방법 2:** 팀원이 알아서 가져감
팀원이 할 일을 끝내면 → 할 일 목록에서 다음 작업을 자동으로 가져감
(같은 작업을 두 명이 동시에 가져가지 않도록 자동 잠금)

**비유:**
- 방법 1: 팀장이 업무 분장표를 만들어서 배부하는 것
- 방법 2: 칸반 보드에서 "할 일"에 있는 포스트잇을 각자 가져가는 것

## 팀원 간 소통

에이전트 팀의 가장 큰 장점은 **팀원끼리 직접 소통**한다는 것입니다.

### 메시지 종류

**1:1 메시지 (message)**
보안 담당 → 성능 담당: "이 API 호출이 암호화 없이 가는데, 알고 있어?"

**전체 방송 (broadcast)**
팀장 → 전체: "방향이 바뀌었어. 인증 모듈 우선으로 해줘."
(주의: 비용이 팀원 수만큼 늘어남)

### 자동 알림

- 팀원이 할 일 완료 → 팀장에게 자동 알림
- 팀원이 멈춤 → 팀장에게 자동 알림
- 선행 작업 완료 → 차단된 작업이 자동으로 풀림

**비유:**

사내 메신저(슬랙)를 생각하세요:
- 1:1 메시지 = DM
- 전체 방송 = 채널에 @everyone
- 자동 알림 = 봇이 "작업 완료!" 알림 보내기

## 실전 활용 예시

### 예시 1: 다각도 코드 리뷰

<div class="example-case">

나: "<span class="keyword-highlight">PR #142를 리뷰할 팀을 만들어줘</span>.
보안 담당 1명, 성능 담당 1명, 테스트 커버리지 담당 1명.
각자 검토하고 결과를 공유해."

→ 보안 담당: "인증 토큰 만료 처리가 없어"
→ 성능 담당: "이 쿼리가 N+1 문제를 일으켜"
→ 테스트 담당: "에러 케이스 테스트가 빠져 있어"
→ 보안 담당: "아! 테스트가 말한 에러 케이스, 내가 발견한 토큰 문제와 연관돼"
→ 팀장: 종합 리뷰 보고서 작성
</div>

**왜 에이전트 팀가 좋은가?**
혼자 리뷰하면 한 번에 하나의 관점만 봄.
3명이 동시에 보면 놓치는 부분이 줄어들고, 서로 발견한 내용을 연결함.

### 예시 2: 원인 불명 버그 토론

<div class="example-case">

나: "<span class="keyword-highlight">앱이 간헐적으로 멈추는데 원인을 모르겠어</span>.
팀 5명을 만들어서 각각 다른 가설을 세우고
서로 반박하게 해줘."

→ A: "서버 메모리 부족 같아"
→ B: "서버 상태는 정상이야. 데이터베이스 쿼리 문제 같은데"
→ C: "쿼리 수는 적어. 외부 결제 API가 느린 것 같아"
→ A: "맞다! 결제 페이지에서만 멈추니까 C 말이 맞을 수 있어"
→ D: "결제 API 응답 시간을 확인해봤더니 평균 8초..."
→ E: "타임아웃 설정이 5초인데, 8초면 무조건 실패하지"
→ 팀장: "원인은 결제 API 응답 지연 + 부적절한 타임아웃 설정"

비유: 의사 5명이 환자 증상을 놓고 토론하는 것. 각자 다른 진단을 내리고 서로 반박하다 보면 정확한 원인을 찾아냄.
</div>

### 예시 3: 기능 분담 개발

<div class="example-case">

나: "<span class="keyword-highlight">댓글 기능을 추가할 팀을 만들어줘</span>.
화면 담당, 서버 담당, 필터링 담당.
서로 맞춰가면서 만들어."

→ 화면 담당: "댓글 입력창을 이렇게 만들었어"
→ 서버 담당: "알겠어, 그 입력창에서 보내면 내가 저장할게"
→ 필터 담당: "저장하기 전에 내가 먼저 욕설 체크할게"
→ 서버 담당: "좋아, 필터 통과한 것만 저장하도록 바꿀게"

비유: 요리팀에서 한 명은 밥, 한 명은 국, 한 명은 반찬을 동시에 만들되 "국이 매우니까 반찬은 순하게 하자"처럼 서로 맞추는 것.
</div>

### 예시 4: 설계 토론

<div class="example-case">

나: "<span class="keyword-highlight">SaaS 대시보드를 설계할 팀을 만들어줘</span>.
UX 전문가, 기술 전문가, 비평가.
서로 토론해서 최적안을 만들어."

→ UX: "메인에 차트 4개를 한눈에 보여주자"
→ 비평가: "한꺼번에 4개 불러오면 느릴 수 있어"
→ 기술: "데이터를 미리 캐싱하면 해결돼"
→ 비평가: "인터넷 끊기면? 오프라인 화면도 필요해"
→ UX: "좋은 지적. 스켈레톤 UI를 넣자"
→ 팀장: 최종 설계안 정리

비유: 광고 기획회의에서 카피라이터, 디자이너, 클라이언트가 서로 피드백하며 완성하는 과정.
</div>

## 품질 관리: Hooks

팀원의 작업 품질을 자동으로 관리할 수 있습니다.

### TeammateIdle — 팀원이 쉬려고 할 때

팀원이 할 일을 다 끝내고 쉬려 할 때 자동 실행.
"아직 이것도 확인해"라고 피드백을 보내서 계속 일하게 할 수 있음.

비유: 인턴이 "다 했습니다" 하면
"여기도 확인해봐"라고 추가 지시하는 것

### TaskCompleted — 할 일이 완료될 때

할 일이 "완료"로 바뀔 때 자동 실행.
품질 기준에 미달하면 완료를 거부하고 피드백 전달.

비유: QA 담당자가 "이거 아직 부족해, 다시 해"라고 돌려보내는 것

## 팀 종료하기

작업이 끝나면 팀을 정리해야 합니다. **순서가 중요합니다.**

### 올바른 순서

<div class="example-case">

**1단계:** 팀원 먼저 종료
나: "<span class="keyword-highlight">보안 담당 팀원 종료해줘</span>"
→ 팀원이 현재 작업 마무리 후 종료

**2단계:** 모든 팀원 종료 확인 후 팀 정리
나: "<span class="keyword-highlight">팀 정리해줘</span>"
→ 공유 리소스(할 일 목록, 메일함) 삭제
</div>

### 잘못된 순서

<div class="example-case">

팀원이 아직 일하는데 팀 정리 시도
→ 오류 발생. "아직 팀원이 작동 중입니다"

비유:

회의가 끝났을 때:
- 팀원들이 자리에서 일어남 → 팀장이 회의실 불 끔 (올바른 순서)
- 팀원이 아직 앉아있는데 불 먼저 끄기 → 안 됨 (잘못된 순서)
</div>

> **중요:** 팀 정리는 반드시 팀장이 해야 합니다. 팀원이 정리하면 오류가 생길 수 있습니다.

## 비용

| 방식 | 비용 | 비유 |
|------|------|------|
| 서브 에이전트 1개 | 기본의 30~100% | 인턴 1명 인건비 |
| 에이전트 팀 3명 | 기본의 3~4배 | 팀원 3명 인건비 |
| 에이전트 팀 5명 | 기본의 5~7배 | 팀원 5명 인건비 |

- **비용을 아끼고 싶을 때:** → 서브 에이전트 사용 (결과만 필요한 경우)
- **비용보다 품질/토론이 중요할 때:** → 에이전트 팀 사용 (서로 검증하므로 더 정확한 결과)

## 잘 쓰는 방법

### 1. 팀 규모는 3~5명이 적당

- **3명:** 대부분의 작업에 충분
- **5명:** 복잡한 작업까지 커버

왜?
→ 인원이 많을수록 비용이 비례해서 증가
→ 조율 시간도 늘어남
→ 일정 이상이면 더 빨라지지 않음

비유: 짜장면 배달에 배달원 10명 붙이면 오히려 문 앞이 혼잡해짐

### 2. 팀원당 할 일 5~6개가 적절

- 너무 적으면: 팀원이 금방 할 일이 없어짐
- 너무 많으면: 너무 오래 작업해서 중간 점검 어려움
- 할 일 15개 → 팀원 3명이 적당

### 3. 같은 파일을 두 팀원이 수정하지 않기

- 팀원 A와 B가 동시에 같은 파일 수정 → 나중에 저장하는 쪽이 덮어씀 → 한 명의 작업 소실
- 각 팀원에게 담당 파일을 따로 지정: "<span class="keyword-highlight">A는 화면 파일만, B는 서버 파일만 담당해</span>"

### 4. 중간중간 확인하기

팀을 믿고 오래 방치하면:
→ 엉뚱한 방향으로 진행될 수 있음
→ 뒤늦게 발견하면 <span class="keyword-highlight">되돌리기</span> 어려움

<kbd>Shift+Down</kbd>으로 주기적으로 팀원 확인하고
방향이 이상하면 바로 수정 지시

### 5. 팀장이 직접 일하지 않게 주의

간혹 팀장이 팀원 결과를 기다리지 않고 직접 작업을 시작함.

이럴 때:
"<span class="keyword-highlight">팀원들이 작업을 끝낼 때까지 기다려줘</span>"
라고 말하면 해결됨.

## 문제 해결

### 팀원이 화면에 안 보여요

→ <kbd>Shift+Down</kbd>을 눌러보세요.
이미 실행 중인데 화면에 안 보이는 경우가 많습니다.

→ 작업이 너무 단순하면 Claude가 팀을 안 만들 수 있습니다.
팀이 필요한 이유를 좀 더 명확히 설명해보세요.

### 권한 확인창이 너무 자주 떠요

팀원이 작업할 때마다 "이거 해도 될까요?" 창이 뜨면 번거롭습니다.

→ 자주 쓰는 작업을 미리 허용해두세요.
팀 만들기 전에 설정해두는 게 좋습니다.

### 팀원이 오류에서 멈춰요

→ <kbd>Shift+Down</kbd>으로 해당 팀원 화면으로 이동
→ 직접 "<span class="keyword-highlight">이렇게 해봐</span>" 지시를 보내거나
→ 그 팀원을 종료하고 새 팀원 투입

### 분할 화면이 안 열려요

```
→ tmux가 설치되어 있는지 확인
→ VS Code 내장 터미널이나 Windows Terminal은 지원 안 됩니다
→ 일반 터미널(macOS Terminal, iTerm2)이나 WSL2를 사용하세요
```

### tmux 세션이 남아있어요

```
팀 종료 후에도 tmux 세션이 남아 있다면:

터미널에서:
  tmux ls                         ← 세션 목록 확인
  tmux kill-session -t 세션이름    ← 해당 세션 종료
```

## 알려진 한계

| 한계 | 설명 |
|------|------|
| **세션 재시작 불가** | Claude Code를 껐다 켜면 팀원이 사라짐. `/resume`으로 복구 안 됨 |
| **팀은 1개만** | 한 번에 팀 1개만 운영 가능. 새 팀은 기존 팀 해산 후 생성 |
| **팀원이 팀원을 못 만듦** | 중첩 불가. 팀장만 팀원 관리 가능 |
| **할 일 상태 지연** | 작업 완료 표시가 늦게 될 수 있음 |
| **분할 화면 제한** | tmux/iTerm2 필요. VS Code, Windows Terminal 미지원 |
| **팀장 변경 불가** | 처음 팀을 만든 세션이 계속 팀장 |
| **종료가 느릴 수 있음** | 팀원이 현재 작업을 마무리한 후 종료됨 |
