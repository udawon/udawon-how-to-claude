## 페르소나 지정이 실제로 도움이 되나?

**됩니다. 하지만 '어떻게' 쓰느냐에 따라 효과 차이가 큽니다.**

```
효과가 큰 경우:
  "10년차 보안 전문가 관점에서 이 코드를 검토해줘"
  -> 일반 검토보다 보안 취약점을 더 깊이 찾아냄
  -> '관점'을 좁혀주기 때문에 답변의 깊이가 달라짐

  "비개발자도 이해할 수 있게 설명해줘"
  -> 기술 용어 없이 쉽게 설명
  -> 대상(청중)을 지정하면 톤과 수준이 맞춰짐

효과가 작은 경우:
  "너는 세계 최고의 천재 개발자야"
  -> 이런 식의 과장된 칭찬형 페르소나는 거의 효과 없음
  -> Claude는 칭찬받는다고 더 잘하지 않음

비유:
  (O) "정형외과 전문의 관점에서 진단해주세요" -> 전문 분야 집중
  (X) "당신은 세계 최고의 의사입니다" -> 의미 없는 수식어
```

## 핵심: '역할'이 아니라 '관점과 기준'

```
별로인 페르소나:
  "너는 시니어 풀스택 개발자야"
  -> 너무 넓어서 구체적 효과가 없음

좋은 페르소나:
  "너는 성능 최적화 전문가야.
   모든 코드를 성능 관점에서 판단하고,
   불필요한 연산이나 느린 패턴을 지적해줘."
  -> 관점 + 행동 기준이 명확해서 실제로 다른 결과가 나옴

비유:
  "좋은 요리사가 되어줘" vs
  "저염식 전문 요리사로서, 소금 대신 향신료로 맛을 내줘"
  후자가 훨씬 구체적인 결과를 만듦
```

## 한 번만 지정하면 되나? 중간에 또 넣어야 하나?

상황에 따라 다릅니다.

### 방법 A: CLAUDE.md에 적기 (영구 적용)

```
CLAUDE.md에 이렇게 적어두면:
  "이 프로젝트에서 Claude는 접근성(a11y) 전문가 역할.
   모든 UI 구현 시 스크린리더, 키보드 내비게이션을 고려할 것."

-> 매 세션마다 자동으로 이 관점이 적용됨
-> 다시 말할 필요 없음

적용 방법 (Claude Code 대화창에서):
  "CLAUDE.md에 이 내용 추가해줘:
   이 프로젝트에서는 접근성 전문가 관점에서 작업할 것."

비유: 사무실 벽에 "우리 팀은 고객 응대 시 존댓말 필수" 게시
     (항상 적용)
```

### 방법 B: 대화 중에 말하기 (일시 적용)

```
Claude Code 대화창에서:
  "지금부터 보안 전문가 관점에서 검토해줘"

-> 이 대화 안에서는 유지됨
-> 하지만 대화가 길어지면 잊어버릴 수 있음
-> 새 세션에서는 리셋됨

비유: 회의 시작할 때 "오늘은 비용 관점에서 토론합시다"
     (이 회의만 적용)
```

### 방법 C: 특정 작업에서만 쓰기 (1회성)

```
Claude Code 대화창에서:
  "이 함수를 성능 전문가 관점에서 개선해줘"

-> 이 요청에만 적용. 다음 요청에는 영향 없음

비유: "이 서류만 법률 관점에서 검토해주세요"
     (이 서류만 적용)
```

**추천: 프로젝트 전체에 걸친 페르소나는 CLAUDE.md에, 특정 작업의 관점 전환은 대화 중에.**

## 중간에 다른 페르소나로 바꿀 수 있나?

**가능합니다. 오히려 바꾸는 게 좋을 때가 많습니다.**

```
같은 코드를 다른 관점에서 보고 싶을 때
(Claude Code 대화창에서):

1차: "성능 전문가 관점에서 이 코드를 검토해줘"
-> "이 루프가 비효율적입니다. 캐싱을 추천합니다."

2차: "이번에는 보안 전문가 관점에서 같은 코드를 봐줘"
-> "사용자 입력값 검증이 빠져있습니다."

3차: "초보 개발자가 읽는다고 생각하고 가독성을 평가해줘"
-> "변수명이 불명확합니다. 주석도 부족합니다."

-> 같은 코드인데 3가지 다른 피드백을 받을 수 있음

비유: 같은 건물을 소방관, 건축가, 인테리어 디자이너에게
     각각 점검받는 것. 각자 다른 문제를 발견함.
```

## 주의할 점

```
페르소나를 바꾸면 이전 페르소나의 관점은 사라짐

예:
  "보안 전문가로서 작업해줘" -> 보안 중심으로 코드 작성
  (30분 후)
  "이번에는 UX 디자이너로서 화면을 만들어줘"
  -> 이 시점부터 보안보다 UX에 집중
  -> 보안 관점은 자동으로 유지되지 않음

해결법:
  프로젝트 전체에 필요한 관점은 CLAUDE.md에 적어두고
  특정 작업의 관점만 대화에서 전환

  CLAUDE.md에:
    "모든 코드에서 보안을 항상 고려할 것"
  대화에서:
    "UX 디자이너 관점에서 이 화면을 개선해줘"
  -> 보안은 CLAUDE.md로 유지 + UX 관점 추가

비유: 회사 규칙(CLAUDE.md)은 항상 지키되
     오늘 회의 주제(대화)는 바뀔 수 있는 것
```

## 실전 추천 정리

| 상황 | 추천 방법 |
|------|----------|
| 프로젝트 전체 관점 | CLAUDE.md에 적기 |
| 코드 리뷰할 때 | 특정 관점 지정 (보안/성능 등) |
| 다각도 검토 | 관점 바꿔가며 여러 번 요청 |
| 설명/문서 작성 | 대상(청중) 지정 |
| 일반 구현 | 굳이 페르소나 불필요 |

```
핵심 정리:
  "역할 이름"보다 "관점 + 행동 기준"을 주는 게 효과적
  과장된 칭찬형 페르소나는 효과 없음
  프로젝트 전체 관점은 CLAUDE.md에, 작업별 관점은 대화에서
```
