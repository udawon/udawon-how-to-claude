## 작업 범위를 명확히 제한하는 표현

| 표현 | 효과 | 비유 |
|------|------|------|
| `"이 파일만 수정해"` | 엉뚱한 파일 수정 방지 | "이 서류만 수정해" |
| `"다른 건 건드리지 마"` | 과잉 수정 방지 | "다른 서랍은 열지 마" |
| `"최소한의 변경으로"` | 불필요한 복잡함 방지 | "간단하게만 해" |
| `"기존 패턴을 따라서"` | 스타일 일관성 유지 | "기존 양식에 맞춰서" |

**예시 케이스:**

<div class="example-case">

(X) "디자인을 개선해줘"
→ 어디를? 전부? Claude가 관련 없는 곳까지 바꿀 수 있음

(O) "<span class="keyword-highlight">메인 페이지의 버튼 여백만</span> 좀 더 넓혀줘.
     <span class="keyword-highlight">다른 건 건드리지 마.</span>"
→ 어디를, 무엇만, 제한까지 명확

비유: "방을 깔끔하게 해줘" (뭘 어떻게?)
  vs "책상 위 서류만 정리해줘. 서랍은 열지 마." (명확)
</div>

## 사고 품질을 높이는 표현

| 표현 | 효과 | 비유 |
|------|------|------|
| `"먼저 분석하고, 그다음 구현해"` | 바로 만들지 않고 설계부터 | "도면 먼저 그리고 시공해" |
| `"단계별로 생각해줘"` | 중간 과정이 생겨 오류 감소 | "풀이 과정 보여줘" |
| `"3가지 방법을 비교해줘"` | 최적 방법 도출 | "견적 3곳 받아봐" |
| `"왜 이 방식을 선택했는지 설명해"` | 근거 있는 판단 | "이유를 보고서에 적어" |
| `"예상되는 문제점도 알려줘"` | 리스크 파악 | "부작용도 알려줘" |

**예시 케이스:**

<div class="example-case">

(X) "로그인 기능 추가해줘"
→ Claude가 바로 만들기 시작. 나중에 "이 방식 별로인데..." 할 수 있음

(O) "로그인 기능을 추가하려고 해.
     <span class="keyword-highlight">3가지 방법을 비교해줘.</span>
     이 프로젝트에 맞는 걸 추천하고 <span class="keyword-highlight">이유를 설명해.</span>"
→ 비교 후 근거 있는 선택

비유: 집 리모델링할 때
     바로 공사 시작하는 것 vs 디자인 시안 3개 받아보고 결정하는 것
</div>

### "단계별로 생각해줘"가 효과 있는 이유

연구로 검증된 기법 — "<span class="keyword-highlight">단계별 사고 유도</span>" (Chain-of-Thought Prompting)

(X) "이 알고리즘이 맞아?"
→ Claude가 바로 결론을 냄 (틀릴 확률 높음)

(O) "이 알고리즘이 맞는지 <span class="keyword-highlight">단계별로 검증해줘</span>"
→ 각 단계를 하나씩 확인하면서 스스로 오류를 잡음
→ 복잡한 수학/논리 문제에서 정답률이 실제로 올라감

비유: "이 계산 맞아?" vs "이 계산 풀이 과정을 보여줘"
     과정을 보여주면서 실수를 본인이 발견하게 됨

### Claude Code 전용: ultrathink

Claude Code 대화창에서 `ultrathink`를 넣으면 Claude가 최대 수준으로 깊이 사고합니다.

사용 예:
"이 버그의 원인을 <span class="keyword-highlight">ultrathink</span>해서 찾아줘"
"이 설계 방향을 <span class="keyword-highlight">ultrathink</span>로 분석해줘"

> **주의:** "think", "think hard" 같은 표현은 실제로 사고 깊이를 늘려주지 않습니다.
> 반드시 `ultrathink`를 사용하세요.

언제 쓰면 좋나:
- 원인 불명확한 복잡한 버그 추적
- 전체 아키텍처 설계 방향 결정
- 여러 파일에 걸친 리팩토링 계획

주의:
- 응답이 더 느리고 비용도 높아짐
- 간단한 작업에는 불필요

## 효과 없는 표현 (흔한 오해)

"너는 세계 최고의 전문가야"
→ 거의 효과 없음. Claude는 칭찬/아부에 반응하지 않음

"정말 중요한 작업이야, 최선을 다해줘"
→ 원래도 최선을 다하기 때문에 의미 없음

"틀리면 절대 안 돼"
→ 압박을 준다고 더 잘하지 않음

비유: 직원에게 "당신은 최고예요"라고 하면
     기분은 좋을 수 있지만 실력이 올라가진 않음

| 표현 | 효과 | 이유 |
|------|------|------|
| "단계별로 생각해줘" | 있음 | 중간 과정이 생겨 오류 감소 |
| "ultrathink" | 있음 | Claude가 최대 수준으로 깊이 사고 (Claude Code 전용) |
| "먼저 분석, 그다음 구현" | 있음 | 두 단계로 나눠 재작업 감소 |
| "너는 최고 전문가야" | 없음 | 칭찬은 성능에 영향 없음 |
| "최선을 다해줘" | 없음 | 원래 최선을 다함 |

## 출력 형태를 제어하는 표현

| 표현 | 효과 | 비유 |
|------|------|------|
| `"코드만 수정해, 설명은 짧게"` | 불필요한 장문 방지 | "보고서 말고 결과만" |
| `"단계별로 나눠서 진행하자"` | 중간 확인 가능 | "한 단계씩 보고해" |
| `"표로 정리해줘"` | 가독성 향상 | "엑셀로 정리해줘" |

**예시 케이스:**

<div class="example-case">

빠르게 수정만 원할 때:
"메인 페이지 제목만 바꿔줘. <span class="keyword-highlight">설명 필요 없어.</span>"

큰 작업을 안전하게 할 때:
"쇼핑몰을 만들 건데, <span class="keyword-highlight">단계별로 나눠서 진행하자.</span>
 먼저 1단계로 뭘 해야 하는지 알려줘."
→ 각 단계를 확인하면서 안전하게 진행

비유: 이사할 때

     "다 옮겨!" vs "먼저 큰 가구부터 옮기자. 다음은?"
</div>

## CLAUDE.md를 활용한 영구 설정

매번 같은 말을 반복하는 대신, CLAUDE.md 파일에 한 번 적어두면 모든 세션에 자동 적용됩니다.

```markdown
## 작업 규칙
- 파일을 수정하기 전에 반드시 먼저 읽을 것
- 한 번에 3개 이상 파일을 동시에 수정하지 말 것
- 새 파일보다 기존 파일 수정을 우선할 것

## 커뮤니케이션
- 변경사항 요약을 항상 알려줄 것
- 확신이 없으면 먼저 물어볼 것
```

**예시 케이스:**

<div class="example-case">

설정 없이 매번 말해야 하는 경우:
"<span class="keyword-highlight">한국어로 답변해줘</span>" (매 세션마다 반복)
"<span class="keyword-highlight">수정하기 전에 파일 먼저 읽어</span>" (매번 주의)

CLAUDE.md에 적어두면:
한 번 적어두면 영원히 자동 적용.
다시 말할 필요 없음.

비유: 매번 "아메리카노 덜 달게요"라고 말하는 것 vs
     단골 카페에 취향이 등록된 것
</div>

## 실전 프롬프트 패턴

### 패턴 A: 먼저 조사 -> 그다음 작업

<div class="example-case">

1차: "<span class="keyword-highlight">이 프로젝트가 어떤 구조인지 파악해서 정리해줘</span>"
2차: "그러면 여기에 회원가입 기능을 추가해줘"

비유: 새 직장에서 먼저 업무 파악 → 그다음 실무 시작
     현황을 모르고 바로 일하면 실수함
</div>

### 패턴 B: 제약 조건 명시

"로그인 화면을 만들어줘. 단,
 - <span class="keyword-highlight">이메일/비밀번호 방식만</span>
 - <span class="keyword-highlight">기존 디자인 스타일에 맞춰서</span>
 - <span class="keyword-highlight">에러 메시지는 한국어로</span>"

<div class="example-case">

비유: "케이크 만들어줘. 단, 딸기맛, 2단, 6인치, 생크림으로"
     제약이 구체적일수록 원하는 결과에 가까움
</div>

### 패턴 C: 역할 부여

"<span class="keyword-highlight">10년차 전문가 관점에서</span> 이 프로젝트를 검토해줘.
 안전성, 속도, 유지보수 측면에서 평가해줘"

<div class="example-case">

비유: "신입한테 검토 맡기기" vs "시니어한테 검토 맡기기"
     역할을 명시하면 더 깊이 있는 답변
</div>

### 패턴 D: 예시 기반

"<span class="keyword-highlight">A 페이지와 같은 형태로</span> B 페이지도 만들어줘"

<div class="example-case">

비유: "이 양식이랑 똑같이 나머지도 만들어줘"
     예시가 있으면 결과가 일관적
</div>

## 피해야 할 패턴

| 피해야 할 것 | 왜 문제인가 | 대안 |
|-------------|-----------|------|
| "전부 고쳐줘" | 범위 무한대 | "이 페이지만 고쳐줘" |
| "알아서 해줘" | 기준 없음 | "A 방식으로 해줘" |
| "완벽하게 만들어줘" | 완벽의 기준 불명확 | "모바일에서도 잘 보이게" |
| 한 번에 5개 요청 | 뒤쪽이 누락됨 | 하나씩 순서대로 |

**예시 케이스:**

<div class="example-case">

(X) "디자인 수정하고, 기능 추가하고, 테스트하고, 문서 쓰고, 배포해줘"
→ 5개 중 뒤쪽 2~3개는 빠지거나 대충 됨

(O) "<span class="keyword-highlight">먼저 디자인만 수정하자.</span>"
    (완료 후) "<span class="keyword-highlight">다음으로 기능을 추가해줘.</span>"
→ 하나씩 확실하게 완료

비유: 식당에서 한꺼번에 5개 메뉴를 주문하면
     뒤쪽 메뉴가 늦거나 빠지는 것과 같음
</div>

## 컨텍스트 관리 팁

- **/compact**: 대화가 길어지면 사용. 단, 중요한 건 CLAUDE.md에 먼저 적어두기
- **/context로 토큰 감사**: `/context`를 실행하면 현재 컨텍스트에 뭐가 들어있는지 시각화됨. MCP가 토큰을 과도하게 쓰고 있는지 확인하는 데 유용
- **파일 경로 직접 알려주기**: "아까 그 파일" 보다 "메인 페이지 파일" 이 정확
- **에러 메시지 그대로 복붙**: 직접 요약하지 말고 원본 그대로 붙여넣기
- **세컨드 브레인 (Second Brain)**: 작업 완료 후 핵심 컨텍스트를 로컬 `.claude/` 디렉토리에 저장 → 다음 세션에서 필요할 때만 불러오기(Lazy Loading). 모든 컨텍스트를 항상 로드하지 않고 필요할 때만 가져와서 컨텍스트를 신선하게 유지
- **Fresh Context 원칙**: "이거 해봐 → 아니 그거 말고 → 다시 이렇게" 같은 모순된 지시가 쌓이면 AI가 혼란. Plan 모드에서 방향 확정 후 `/clear` → 깨끗한 상태에서 실행하는 게 훨씬 효과적

**예시 케이스:**

<div class="example-case">

(X) "아까 그 에러 고쳐줘"
→ Claude가 어떤 에러인지 기억 못할 수 있음

(O) 에러 화면을 그대로 복사해서 붙여넣기:
"<span class="keyword-highlight">아래 에러를 고쳐줘</span>:
 (에러 메시지 전체 붙여넣기)"
→ 정확한 정보로 정확한 수정

비유: 의사에게 "어디가 아파요" 할 때

     "여기쯤 아파요" vs 검사 결과지를 보여주는 것
</div>
