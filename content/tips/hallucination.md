## 할루시네이션이란?

Claude가 모르는 정보를 "모른다"고 하지 않고 그럴듯하게 지어내는 현상입니다.
코드에 존재하지 않는 함수를 사용하거나, 없는 파일 경로를 알려주거나, 틀린 사실을 자신있게 말하는 경우가 해당됩니다.

---

## 방법 1: CLAUDE.md에 명시적 지시 추가

가장 효과적인 방법입니다. 프로젝트 루트의 `CLAUDE.md`에 아래 내용을 추가하세요.

```markdown
## 할루시네이션 방지 규칙

### 모르면 모른다고 할 것
- 확실하지 않은 정보는 추측하지 말 것
- 기억이 흐릿한 경우 반드시 파일/검색으로 확인 후 답변
- "아마도", "~인 것 같습니다" 보다 "확인이 필요합니다"를 사용

### 코드 관련
- 파일 내용을 기억에 의존하지 말고 항상 Read로 먼저 읽을 것
- 존재하지 않는 함수나 API를 임의로 만들어내지 말 것

### 사실 기반 답변
- 출처나 근거를 함께 제시할 것
- 최신 정보가 필요한 경우 검색 도구를 사용할 것
```

**예시 케이스:**

<div class="example-case">

(X) "useRouter는 next/navigation에서 import 해야 해"
→ 실제로 확인 안 하고 기억으로 답변 → 버전마다 다를 수 있어 틀릴 수 있음

(O) CLAUDE.md에 "<span class="keyword-highlight">코드 관련 답변 전 반드시 파일을 Read로 확인</span>"을 명시하면
    Claude가 직접 파일을 열어 확인한 후 답변함
</div>

---

## 방법 2: 답변 전 체크리스트 강제

프롬프트나 CLAUDE.md에 자기검증 단계를 넣으세요.

```markdown
## 답변 전 확인 절차
1. 이 정보가 확실한가?
2. 파일/검색으로 검증 가능한가?
3. 불확실하면 "모름"을 먼저 선언하고 답변할 것
```

**예시 케이스:**

<div class="example-case">

(X) "이 라이브러리는 v3부터 지원돼"
→ 확인 없이 기억으로 답변

(O) "<span class="keyword-highlight">이 라이브러리 버전 지원 여부를 확인해줘</span>"라고 요청 시
    검색 도구를 사용하거나 "확실하지 않으니 공식 문서를 확인하세요"라고 답변
</div>

## 방법 3: Extended Thinking 활용

`ultrathink` 키워드를 붙이면 Claude가 답변 전에 더 깊이 추론합니다.
성급하게 지어내는 대신 스스로 논리를 검증하는 시간을 갖게 됩니다.

(X) "이 버그 왜 생겨?"
→ 빠르게 그럴듯한 원인을 지어낼 수 있음

(O) "이 버그 왜 생기는지 <span class="keyword-highlight">ultrathink</span>해줘"
→ 코드를 단계별로 추적하며 실제 원인을 찾으려 시도함

---

## 방법 4: 도구를 먼저 사용하도록 유도

Claude가 기억에 의존하지 않고 실제 파일이나 검색을 먼저 확인하도록 요청합니다.

"<span class="keyword-highlight">답변하기 전에 관련 파일을 먼저 읽어줘</span>"
"<span class="keyword-highlight">검색해서 확인한 후 알려줘</span>"
"<span class="keyword-highlight">코드를 직접 확인하고 답변해줘</span>"

**예시 케이스:**

<div class="example-case">

(X) "이 프로젝트에서 로그인은 어떤 방식이야?"
→ Claude가 기억으로 "아마 JWT 방식일 겁니다" 라고 답변 (틀릴 수 있음)

(O) "<span class="keyword-highlight">관련 파일을 먼저 읽고</span> 로그인 방식을 알려줘"
→ Claude가 실제 파일을 열어 확인한 뒤 정확하게 답변

비유: "우리 집 보일러 모델이 뭐야?"
     기억으로 대답 vs 보일러실에 가서 직접 확인하고 대답
</div>

## CLAUDE.md로도 안 될 때

명시적 지시를 해도 할루시네이션이 발생할 수 있습니다.
이럴 때는 구조적으로 대응하세요.

### 답변 후 자기검증 요청

"<span class="keyword-highlight">방금 답변 중 확실하지 않은 부분이 있으면 표시해줘</span>"

### 역할 분리 (Writer-Reviewer 패턴)

한 Claude가 답변하고, 다른 Claude(또는 같은 대화의 후속 요청)가 검증합니다.

1단계: "이 코드의 문제점을 분석해줘"
2단계: "<span class="keyword-highlight">방금 분석한 내용이 맞는지 코드를 다시 읽고 검증해줘</span>"

### 모델 선택

Claude에는 세 가지 모델이 있습니다:
- **Opus**: 가장 똑똑함. 복잡한 분석, 정확성이 중요한 작업에 적합
- **Sonnet**: 속도와 품질의 균형. 일반적인 작업에 적합
- **Haiku**: 가장 빠름. 간단한 작업에 적합

정확성이 중요한 작업일수록 더 강력한 모델(Opus)을 선택하는 게 유리합니다.
모델 전환은 `/model` 명령어로 할 수 있습니다.

## 내 답변이 할루시네이션인지 어떻게 알까?

Claude의 답변을 그대로 믿지 말고 다음 기준으로 판단하세요.

| 신뢰도 높음 | 신뢰도 낮음 |
|-------------|-------------|
| 파일을 직접 읽은 후 답변 | 기억만으로 답변 |
| "확인이 필요합니다" 표현 | 수치/날짜를 자신있게 제시 |
| 검색 도구를 사용한 답변 | 출처 없이 사실을 단정 |
| 모른다고 인정 | 모든 질문에 답변 |

> 수치나 날짜, 버전 정보를 자신있게 말할수록 오히려 의심해보는 게 좋습니다.
> "잘 모르겠습니다"라고 말하는 Claude가 더 신뢰할 수 있습니다.
