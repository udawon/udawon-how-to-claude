## 인터뷰 패턴이란?

기능을 구현하기 전에 Claude가 먼저 당신을 인터뷰하게 만드는 패턴입니다. 본격 구현 전에 빠진 요구사항이나 놓친 관점을 발견할 수 있습니다.

**왜 필요한가?**

<div class="example-case">

일반적인 흐름:

나: "결제 기능 만들어줘"
Claude: (결제 기능 구현 시작)
나: "아, 카드 결제 말고 카카오페이도 있어야 해"
Claude: (다시 수정)
나: "그리고 실패했을 때 재시도 버튼도..."
Claude: (또 수정)

→ 중간에 방향이 계속 바뀌어 시간 낭비

<span class="keyword-highlight">인터뷰 패턴</span> 흐름:

나: "결제 기능 만들거야. <span class="keyword-highlight">인터뷰해줘</span>"
Claude: "어떤 결제 수단이 필요한가요? 실패 시 어떻게 처리할까요?..."
나: (모든 요구사항 명확히 정리)
Claude: (처음부터 제대로 구현)

비유: 건축가가 집 짓기 전에 "방은 몇 개? 화장실은? 주차는?" 묻는 것
     먼저 물어볼수록 나중에 뜯어고치는 일이 줄어듦
</div>

## 기본 사용법

"\[기능 한 줄 설명\]. AskUserQuestion 도구를 사용해서 나를 <span class="keyword-highlight">인터뷰해줘</span>.

기술 구현, UI/UX, 엣지케이스, 제약사항, 트레이드오프를 파고들어.
당연한 질문은 건너뛰고, 내가 못 생각했을 부분을 집중적으로 질문해줘.

충분히 파악됐다고 생각하면, 완전한 명세를 SPEC.md에 작성해줘."

## 실전 예시

### 예시 1: 검색 기능

<div class="example-case">

나: "검색 기능을 만들 건데, <span class="keyword-highlight">먼저 인터뷰해줘</span>.
     AskUserQuestion 도구 써서 하나씩 물어봐."

Claude: "검색 대상이 뭔가요? 상품명? 태그? 설명까지?"
나: "상품명이랑 태그"

Claude: "실시간 검색(타이핑하면서 결과 나오는)인가요,
         검색 버튼 누르면 나오는 건가요?"
나: "실시간"

Claude: "검색 결과를 어떻게 정렬할까요? 최신순? 관련도순?"
나: "관련도순인데, 관련도 계산은 어떻게 하나요?"

Claude: "한글 초성 검색이 필요한가요? (예: ㄱㄷ → 기대)"
나: "아, 맞다 그것도 필요해!"

→ 초성 검색은 미처 생각하지 못했던 요구사항
   <span class="keyword-highlight">인터뷰</span> 없었으면 나중에 추가로 만들었을 것
</div>

### 예시 2: 대시보드 기능

<div class="example-case">

나: "관리자 대시보드 만들거야. <span class="keyword-highlight">인터뷰해줘.</span>"

Claude: "누가 사용하나요? 사장님 혼자? 직원들도?"
나: "사장님이랑 매니저 2명"

Claude: "권한 차이가 있나요? 매니저는 볼 수만 있고,
         사장님은 수정도 할 수 있게?"
나: "맞아, 그렇게 해야 해"

Claude: "모바일에서도 봐야 하나요?"
나: "사장님이 출퇴근 시간에 폰으로 확인해"

Claude: "데이터 업데이트 주기는요? 실시간? 1분마다?"
나: "1분마다면 충분해"

→ 이후 SPEC.md 자동 작성
   이 명세를 바탕으로 새 세션에서 구현 시작
</div>

## 인터뷰 후 새 세션에서 구현하기

**중요**: 인터뷰로 컨텍스트가 쌓이면 새 세션에서 구현을 시작하세요.

\[인터뷰 세션\]
→ 요구사항 정리 완료
→ SPEC.md 작성됨

\[새 세션에서\]
"@SPEC.md 를 기반으로 검색 기능을 구현해줘.
 백엔드 API부터 시작해서 프론트엔드 순서로 진행해."

장점:
- 새 세션 = 깨끗한 <span class="keyword-highlight">컨텍스트</span> = 구현에만 집중
- SPEC.md가 기준점 역할 → 중간에 방향 흔들림 없음
- 나중에 "원래 요구사항이 뭐였지?" 할 때 참고 가능

## 특정 관점 인터뷰

전체 명세가 아닌 특정 부분만 집중해서 물어볼 수도 있습니다.

**보안 관점 인터뷰:**
"이 로그인 기능의 보안 요구사항을 <span class="keyword-highlight">인터뷰해줘</span>.
 인증, 인가, 세션 관리 부분에서 빠진 게 없는지 확인해."

**UX 관점 인터뷰:**
"이 회원가입 흐름의 UX를 <span class="keyword-highlight">인터뷰해줘</span>.
 사용자가 어디서 막힐 수 있는지 집중적으로 물어봐."

**엣지케이스 인터뷰:**
"이 주문 처리 로직에서 발생할 수 있는 엣지케이스를
 <span class="keyword-highlight">인터뷰</span>로 파악해줘. 특히 결제 실패, 재고 부족 상황."

## 인터뷰 패턴 vs 바로 구현 비교

| 상황 | 바로 구현 | 인터뷰 패턴 |
|------|-----------|-------------|
| 간단한 버그 수정 | 적합 | 불필요 |
| 새 기능 추가 (소규모) | 가능 | 선택적 |
| 새 기능 추가 (대규모) | 위험 | 강력 권장 |
| 요구사항이 불명확할 때 | 위험 | 필수 |
| 처음 만들어보는 기능 | 위험 | 필수 |

판단 기준:
"이 기능을 절반쯤 만들었을 때 방향이 바뀔 가능성이 있나?"
→ 그렇다면 <span class="keyword-highlight">인터뷰 패턴</span>을 써라

비유: 요리 레시피를 확인하지 않고 요리하면
     재료가 부족한 걸 중간에 발견하는 것처럼
     미리 확인하면 중단 없이 완성할 수 있음
