# 에이전트 팀 (Agent Teams)

> 에이전트 팀은 회사의 팀워크와 같습니다. 팀장이 전체를 조율하고, 팀원들은 각자 맡은 영역을 독립적으로 처리하면서 서로 직접 소통합니다.

> **주의**: 에이전트 팀은 실험적 기능으로 기본 비활성화되어 있습니다. 사용하려면 `settings.json`에 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` 환경 변수를 설정해야 합니다.

---

## 에이전트 팀이란?

여러 Claude Code 인스턴스가 함께 협력하는 모드입니다. 하나의 세션이 팀장(Team Lead) 역할을 하고, 나머지 팀원(Teammates)들이 독립적인 컨텍스트 창에서 각자의 작업을 처리합니다.

**서브 에이전트와의 핵심 차이점**: 팀원들은 서로 직접 메시지를 주고받을 수 있습니다.

---

## 예시 케이스

> **상황**: 스타트업 CTO 승호씨는 새로운 결제 모듈을 도입해야 합니다. 이 작업은 프론트엔드(UI/UX), 백엔드(API), 보안(취약점 검토) 세 영역을 동시에 다뤄야 합니다. 에이전트 팀을 사용하면 각 전문가 에이전트가 동시에 자신의 영역을 탐구하고, 서로 발견한 제약사항을 공유하면서 전체 설계를 조율할 수 있습니다.

---

## 서브 에이전트 vs 에이전트 팀

| 항목 | 서브 에이전트 | 에이전트 팀 |
|------|-------------|-----------|
| **컨텍스트** | 자체 창; 결과를 상위에 보고 | 자체 창; 완전 독립 |
| **커뮤니케이션** | 메인 에이전트에만 보고 | 팀원끼리 직접 메시지 가능 |
| **조율 방식** | 메인 에이전트가 모든 작업 관리 | 공유 작업 목록으로 자체 조율 |
| **적합한 상황** | 결과만 중요한 집중된 작업 | 토론과 협업이 필요한 복잡한 작업 |
| **토큰 비용** | 낮음: 결과만 메인 컨텍스트로 돌아옴 | 높음: 팀원 각각이 독립 Claude 인스턴스 |

---

## 활성화 방법

`settings.json`에 추가:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

---

## 에이전트 팀 시작하기

활성화 후 자연어로 팀 생성을 요청합니다:

```
개발자들이 코드베이스에서 TODO 주석을 추적하는 CLI 도구를 설계하고 있어.
에이전트 팀을 만들어 다양한 각도에서 탐구해줘:
한 팀원은 UX에, 한 팀원은 기술 아키텍처에, 한 팀원은 악마의 변호인 역할로.
```

Claude가 팀을 만들고, 팀원을 생성하고, 각자의 탐구를 조율한 뒤 결과를 종합합니다.

---

## 에이전트 팀 제어하기

### 화면 표시 모드 선택

| 모드 | 특징 | 요구사항 |
|------|------|---------|
| **In-process** | 모든 팀원이 메인 터미널 내에서 실행. Shift+Down으로 팀원 전환 | 어떤 터미널이든 가능 |
| **Split panes** | 각 팀원이 별도 패널에서 실행. 모든 출력 동시 확인 | tmux 또는 iTerm2 필요 |

기본값은 `"auto"`: tmux 세션 안이면 split panes, 아니면 in-process.

`settings.json`에서 강제 설정:

```json
{
  "teammateMode": "in-process"
}
```

### 팀원 수와 모델 지정

```
4명의 팀원을 만들어 이 모듈들을 병렬로 리팩토링해줘.
각 팀원은 Sonnet 모델을 사용해.
```

### 팀원과 직접 대화

- **In-process 모드**: `Shift+Down`으로 팀원 전환, 타이핑으로 메시지 전송
- **Split-pane 모드**: 팀원 패널 클릭 후 직접 상호작용

### 작업 계획 승인 요구

```
인증 모듈을 리팩토링할 아키텍트 팀원을 생성해줘.
변경 전에 계획 승인을 요구해.
```

팀원이 계획을 완성하면 팀장에게 승인 요청을 보냅니다. 팀장이 승인하거나 피드백과 함께 거절합니다. 승인되어야만 실제 구현을 시작합니다.

---

## 에이전트 팀 아키텍처

| 구성 요소 | 역할 |
|---------|------|
| **팀장 (Team Lead)** | 팀을 생성하고, 팀원을 소환하고, 작업을 조율하는 메인 세션 |
| **팀원 (Teammates)** | 할당된 작업을 처리하는 독립 Claude Code 인스턴스 |
| **작업 목록 (Task List)** | 팀원들이 가져가고 완료하는 공유 작업 항목 목록 |
| **메일박스 (Mailbox)** | 에이전트 간 통신 시스템 |

**저장 위치**:
- 팀 설정: `~/.claude/teams/{team-name}/config.json`
- 작업 목록: `~/.claude/tasks/{team-name}/`

---

## 에이전트 팀이 효과적인 상황

| 상황 | 이유 |
|------|------|
| **병렬 코드 리뷰** | 보안, 성능, 테스트 커버리지를 동시에 집중 검토 |
| **경쟁 가설로 디버깅** | 여러 팀원이 다른 원인을 동시에 조사하고 서로 반박 |
| **새 모듈 개발** | 프론트엔드/백엔드/테스트를 각각 독립 소유 |
| **리서치 및 리뷰** | 여러 측면을 동시 탐구 후 발견 내용 공유·검토 |

## 실용적인 사용 예시

### 병렬 코드 리뷰

```
PR #142를 리뷰할 에이전트 팀을 만들어줘. 세 명의 리뷰어를 소환해:
- 한 명은 보안 영향에 집중
- 한 명은 성능 영향 확인
- 한 명은 테스트 커버리지 검증
각자 리뷰하고 결과를 보고해.
```

### 경쟁 가설로 버그 조사

```
사용자가 메시지 한 번 후 앱이 종료된다고 보고하고 있어.
5명의 에이전트 팀원을 소환해 다른 가설을 조사해줘.
서로의 이론을 반박하는 과학적 토론을 진행하게 해.
발견한 합의 내용을 findings.md에 업데이트해.
```

## 팀 정리하기

작업이 끝나면 팀을 정리합니다:

```
팀 정리해줘
```

> **주의**: 항상 팀장에게 정리를 요청하세요. 팀원이 정리를 실행하면 리소스가 불일치 상태로 남을 수 있습니다. 정리 전에 활성 팀원들을 먼저 종료해야 합니다.

### 팀원 종료

```
researcher 팀원에게 종료를 요청해줘
```

## 모범 사례

### 팀원에게 충분한 컨텍스트 제공

팀원은 프로젝트 컨텍스트(CLAUDE.md, MCP 서버, 스킬)를 자동으로 로드하지만, 팀장의 대화 기록은 인계받지 않습니다. 소환 프롬프트에 작업별 세부사항을 포함하세요:

```
보안 리뷰어 팀원을 소환해. 프롬프트: "src/auth/의 인증 모듈을
보안 취약점으로 검토해. 토큰 처리, 세션 관리, 입력 검증에 집중.
앱은 httpOnly 쿠키에 저장된 JWT 토큰을 사용해.
심각도 등급과 함께 이슈를 보고해."
```

### 적절한 팀 크기 선택

- 대부분의 워크플로우: **3~5명** 권장
- 팀원당 **5~6개 작업**이 적당 (너무 많으면 조율 비용 증가)
- 실제로 병렬 작업이 필요한 경우에만 팀원 추가

### 작업 크기 조절

| 크기 | 문제점 |
|------|-------|
| 너무 작음 | 조율 비용이 이득보다 큼 |
| 너무 큼 | 팀원이 너무 오래 작업해 낭비 위험 |
| 적당함 | 명확한 결과물이 있는 독립 단위 (함수, 테스트 파일, 리뷰 등) |

### 파일 충돌 방지

두 팀원이 같은 파일을 편집하면 덮어쓰기가 발생합니다. 각 팀원이 다른 파일 세트를 소유하도록 작업을 분리하세요.

---

## 현재 알려진 제한사항

| 제한 | 설명 |
|------|------|
| 세션 재시작 시 In-process 팀원 복구 불가 | `/resume`으로 재시작해도 팀원이 복구되지 않음 |
| 작업 상태 지연 | 팀원이 작업 완료 표시를 잊는 경우 있음 |
| 종료가 느릴 수 있음 | 팀원이 현재 요청/도구 호출 완료 후에 종료 |
| 세션당 하나의 팀만 | 현재 팀을 정리해야 새 팀 시작 가능 |
| 중첩 팀 불가 | 팀원은 자신의 팀이나 팀원을 생성할 수 없음 |
| Split panes는 tmux/iTerm2 필요 | VS Code 터미널, Windows Terminal, Ghostty에서 미지원 |

## 문제 해결

### 팀원이 나타나지 않을 때

- In-process 모드: `Shift+Down`으로 이미 실행 중인 팀원 확인
- 작업이 팀을 정당화할 만큼 복잡한지 확인
- Split panes 요청 시 `which tmux`로 tmux 설치 확인

### 권한 요청이 너무 많을 때

팀원 소환 전에 [권한 설정](./sub-agents.md)에서 일반적인 작업을 미리 승인하세요.

### 팀장이 작업 완료 전에 종료될 때

계속 진행하라고 지시하세요. 팀장이 위임 대신 직접 작업을 시작하면 기다리라고 요청하세요:

```
팀원들이 작업을 완료할 때까지 기다려줘
```

### 남은 tmux 세션 정리

```bash
tmux ls
tmux kill-session -t <session-name>
```
