# 일반적인 워크플로우

> 코드 탐색, 버그 수정, 리팩토링, 테스트 작성, PR 생성 등 매일 쓰는 개발 작업의 단계별 가이드입니다.

---

## 새 코드베이스 파악하기

새 프로젝트에 합류했거나 낯선 코드를 봐야 할 때 Claude Code가 선배처럼 설명해줍니다.

### 전체 구조 파악

```bash
# 프로젝트 루트로 이동
cd /path/to/project

# Claude Code 시작
claude
```

```
이 코드베이스가 전체적으로 어떻게 구성되어 있는지 설명해줘
```

```
여기서 쓰는 주요 아키텍처 패턴이 뭐야?
```

```
핵심 데이터 모델이 뭐야?
```

```
인증은 어떻게 처리해?
```

**팁:**
- 먼저 넓은 질문으로 시작하고 점점 좁혀나가세요
- 프로젝트만의 용어 사전을 요청해보세요

### 특정 기능 코드 찾기

```
사용자 인증을 담당하는 파일들을 찾아줘
```

```
이 인증 파일들이 서로 어떻게 연결되어 있어?
```

```
로그인 프로세스를 프론트엔드부터 데이터베이스까지 추적해줘
```

## 버그 효율적으로 수정하기

에러 메시지가 있으면 바로 붙여넣으세요.

### 단계별 흐름

```
npm test를 실행했더니 에러가 발생했어
[에러 내용 붙여넣기]
```

```
user.ts의 @ts-ignore 문제를 고칠 방법 몇 가지 제안해줘
```

```
제안한 null 체크를 user.ts에 추가해줘
```

**팁:**
- 에러를 재현하는 명령어를 알려주면 Claude가 직접 실행해 볼 수 있습니다
- 에러가 간헐적인지 일관적인지 알려주세요
- 스택 트레이스를 전부 붙여넣으세요

## 코드 리팩토링

오래된 코드를 현대적인 방식으로 업그레이드할 때.

### 단계별 흐름

```
코드베이스에서 더 이상 사용하지 않는 (deprecated) API를 찾아줘
```

```
utils.js를 현대적인 JavaScript 방식으로 어떻게 리팩토링할지 제안해줘
```

```
utils.js를 ES2024 기능을 사용하도록 리팩토링하되 동일한 동작을 유지해줘
```

```
리팩토링된 코드 테스트를 실행해줘
```

**팁:**
- 하위 호환성이 필요하면 미리 알려주세요
- 작은 단위로 나눠서 테스트하면서 진행하세요
- Claude에게 현대적 접근법의 장점을 설명해달라고 하세요

## 계획 모드로 안전하게 분석하기

계획 모드에서는 Claude가 코드를 읽기만 하고 수정하지 않습니다. 마치 "보기 전용 모드"입니다.

### 언제 사용하나요?

| 상황 | 계획 모드 추천 이유 |
|------|-------------------|
| 여러 파일을 수정해야 할 때 | 변경 범위를 먼저 파악하기 위해 |
| 낯선 코드베이스 탐색 시 | 실수 없이 구조 파악하기 위해 |
| 복잡한 마이그레이션 계획 시 | 실행 전 전략을 세우기 위해 |

### 사용 방법

**세션 중 전환:**
`Shift+Tab`을 두 번 눌러 계획 모드 진입 (`⏸ plan mode on` 표시)

**처음부터 계획 모드로 시작:**
```bash
claude --permission-mode plan
```

**일회성 분석:**
```bash
claude --permission-mode plan -p "인증 시스템을 분석하고 개선점을 제안해줘"
```

### 예시: 복잡한 리팩토링 계획

```bash
claude --permission-mode plan
```

```
인증 시스템을 OAuth2로 마이그레이션해야 해. 상세한 마이그레이션 계획을 만들어줘
```

```
하위 호환성은 어떻게 처리해?
```

```
데이터베이스 마이그레이션은 어떻게 해야 해?
```

> `Ctrl+G`를 누르면 계획을 기본 텍스트 에디터에서 직접 편집할 수 있습니다.

## 테스트 작성하기

### 단계별 흐름

```
NotificationsService.swift에서 테스트가 없는 함수들을 찾아줘
```

```
알림 서비스에 테스트를 추가해줘
```

```
알림 서비스의 엣지 케이스에 대한 테스트 케이스를 추가해줘
```

```
새로운 테스트를 실행하고 실패하는 것들을 고쳐줘
```

Claude는 프로젝트의 기존 테스트 패턴을 분석해 동일한 스타일로 테스트를 작성합니다. 놓치기 쉬운 경계값, 에러 조건, 예외 케이스도 찾아줍니다.

## Pull Request 생성하기

### 간단하게 한 번에

```
내 변경사항으로 PR을 만들어줘
```

### 단계별로 제어하며

```
authentication 모듈에 내가 한 변경사항을 요약해줘
```

```
pr 만들어줘
```

```
보안 개선사항에 대한 더 자세한 컨텍스트를 PR 설명에 추가해줘
```

> `gh pr create`로 PR을 만들면 현재 세션이 자동으로 해당 PR에 연결됩니다. 나중에 `claude --from-pr <번호>`로 이어서 작업할 수 있습니다.

## @ 기호로 파일/폴더 참조하기

`@`를 사용하면 파일이나 폴더를 빠르게 참조할 수 있습니다.

```
@src/utils/auth.js의 로직을 설명해줘
```
→ 해당 파일의 전체 내용이 대화에 포함됩니다.

```
@src/components 구조가 어떻게 되어 있어?
```
→ 폴더 목록과 파일 정보를 제공합니다.

```
@github:repos/owner/repo/issues 데이터를 보여줘
```
→ MCP 서버를 통해 외부 리소스를 가져옵니다.

## 확장 사고 모드 (Extended Thinking)

복잡한 문제에서 Claude가 더 깊이 생각하게 할 수 있습니다.

기본적으로 활성화되어 있으며, `Ctrl+O`로 상세 출력 모드를 켜면 Claude의 내부 추론 과정을 볼 수 있습니다.

### 설정 방법

| 방법 | 설명 |
|------|------|
| 노력 수준 조정 | `/model`에서 low/medium/high 선택 |
| `ultrathink` 키워드 | 프롬프트에 포함하면 그 턴에 최대 추론 |
| `Option+T` / `Alt+T` | 세션 내 사고 모드 켜기/끄기 |
| `/config` | 전역 기본값 설정 |

> "think", "think hard" 같은 표현은 실제로 추가 추론 토큰을 할당하지 않습니다. 공식 설정을 사용하세요.

## 이전 대화 재개하기

```bash
claude --continue    # 현재 폴더의 가장 최근 대화 이어서
claude --resume      # 대화 선택기 열기
claude --from-pr 123 # 특정 PR에 연결된 세션 재개
```

세션 안에서는 `/resume`으로 다른 대화로 전환할 수 있습니다.

### 세션에 이름 붙이기

```
/rename auth-리팩토링
```

나중에 이름으로 재개:
```bash
claude --resume auth-리팩토링
```

### 세션 선택기 단축키

| 단축키 | 동작 |
|--------|------|
| `↑` / `↓` | 세션 탐색 |
| `Enter` | 선택한 세션 재개 |
| `P` | 세션 내용 미리보기 |
| `R` | 세션 이름 변경 |
| `/` | 검색 |
| `B` | 현재 Git 브랜치 세션만 필터링 |

## Git Worktree로 병렬 작업하기

여러 작업을 동시에 진행할 때 각 Claude 세션이 독립적인 코드 사본을 가질 수 있습니다. Git worktree는 **여러 작업실**처럼 같은 창고(저장소)를 공유하면서도 각자의 공간에서 작업합니다.

### 빠른 사용법

```bash
# feature-auth라는 이름의 worktree에서 Claude 시작
claude --worktree feature-auth

# 또 다른 세션을 별도 worktree에서 시작
claude --worktree bugfix-123

# 이름을 자동 생성하려면
claude --worktree
```

Worktree는 `<저장소>/.claude/worktrees/<이름>` 위치에 생성됩니다.

### 정리 방법

- 변경사항이 없으면 자동으로 삭제됩니다
- 변경사항이 있으면 유지/삭제 여부를 묻습니다

> `.claude/worktrees/`를 `.gitignore`에 추가해 메인 저장소에 표시되지 않게 하세요.

## Unix 유틸리티처럼 사용하기

### CI/CD 파이프라인에 추가

```json
// package.json
{
  "scripts": {
    "lint:claude": "claude -p '린터 역할을 해줘. main과의 변경사항을 보고 오타 관련 이슈를 리포트해줘. 파일명과 줄 번호를 첫 번째 줄에, 이슈 설명을 두 번째 줄에 써줘.'"
  }
}
```

### 파이프로 데이터 전달

```bash
cat build-error.txt | claude -p '이 빌드 오류의 근본 원인을 간결하게 설명해줘' > output.txt
```

### 출력 형식 제어

```bash
# 기본 텍스트 출력
claude -p '데이터를 요약해줘' --output-format text

# JSON 형식 (메타데이터 포함)
claude -p '버그를 분석해줘' --output-format json > analysis.json

# 실시간 스트리밍 JSON
claude -p '로그 파일을 파싱해줘' --output-format stream-json
```

## 예시 케이스

**케이스 1: 새 팀원 민지 씨의 첫날**

대기업 개발팀에 입사한 민지 씨. 수천 개 파일로 된 코드베이스가 막막합니다.

```
이 코드베이스를 이해하는 데 도움을 줘. 전체 구조, 주요 패턴, 인증 방식을 설명해줘
```

Claude가 마치 시니어 개발자처럼 프로젝트 구조를 설명합니다.

**케이스 2: 리뷰어/작성자 패턴**

복잡한 기능을 개발 중인 준혁 씨는 두 개의 세션을 씁니다.

- **세션 A (작성자)**: `결제 시스템의 rate limiter를 구현해줘`
- **세션 B (검토자)**: `@src/middleware/rateLimiter.ts를 검토해줘. 엣지 케이스, 경쟁 조건, 기존 미들웨어 패턴과의 일관성을 확인해줘`

작성자가 검토자의 피드백을 받아 개선합니다.

**케이스 3: 쇼핑몰 버그 수정**

쇼핑몰 서비스를 운영하는 유진 씨에게 사용자가 만료된 카드로 결제 시 오류가 생긴다는 버그 리포트가 들어왔습니다.

```
만료된 카드를 가진 사용자에게 결제 흐름이 깨져 있어.
관련 코드는 src/payments/에 있어. 특히 토큰 갱신 부분.
실패를 재현하는 테스트를 먼저 작성하고 그 다음 고쳐줘
```

Claude가 관련 파일을 모두 탐색하고 실패 테스트를 작성한 뒤 근본 원인을 수정합니다.
