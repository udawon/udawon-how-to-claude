# 샌드박싱: Claude Code의 격리된 작업 공간

샌드박스(Sandbox)는 마치 **투명한 유리 상자** 안에서 Claude가 일하게 하는 기능입니다. 상자 안에서는 자유롭게 작업하지만, 상자 밖(중요한 시스템 파일, 허가되지 않은 외부 서버)에는 절대 손을 뻗을 수 없습니다.

---

## 왜 샌드박싱이 필요한가요?

기존 방식은 Claude가 명령을 실행할 때마다 여러분의 허락을 구합니다. 하지만 이 방식은 단점이 있습니다.

| 문제점 | 설명 |
|---|---|
| 승인 피로 | 매번 클릭하다 보면 내용을 꼼꼼히 안 보게 됨 |
| 작업 효율 저하 | 계속 멈추고 승인하는 과정이 생산성을 낮춤 |
| 자율성 제한 | Claude가 연속 작업을 독립적으로 수행하기 어려움 |

샌드박스는 처음에 "이 범위에서는 자유롭게 일해도 돼"라는 경계를 설정하고, 그 안에서는 매번 허락 없이 작업할 수 있게 합니다.

---

## 샌드박스의 두 가지 보호막

### 1. 파일 시스템 격리 (어디를 건드릴 수 있나)

- **쓰기 가능**: 현재 작업 폴더와 그 하위 폴더
- **읽기 가능**: 컴퓨터 전체 (일부 제외)
- **쓰기 불가**: 작업 폴더 외부 (설정으로 추가 가능)

예를 들어 프로젝트가 `/home/user/my-project`에 있다면:
- `/home/user/my-project/src/` → 읽고 쓰기 가능
- `/home/user/.bashrc` → 쓰기 불가 (읽기만 가능)
- `/etc/hosts` → 쓰기 불가

이 제한은 `kubectl`, `terraform`, `npm` 같은 하위 명령도 포함해 **운영체제 수준**에서 강제됩니다.

### 2. 네트워크 격리 (어디에 연결할 수 있나)

네트워크는 프록시 서버를 통해 제어됩니다.

- 허용된 도메인만 접근 가능
- 새 도메인 요청 시 알림 표시
- `allowManagedDomainsOnly` 설정 시 허가된 도메인 외 자동 차단

---

## 운영체제별 샌드박스 기술

| 운영체제 | 사용 기술 |
|---|---|
| macOS | Seatbelt (내장) |
| Linux | bubblewrap |
| WSL2 | bubblewrap |
| WSL1 | 지원 안 함 |

---

## 샌드박스 설정 방법

### 설치 (Linux/WSL2만 해당)

```bash
# Ubuntu/Debian
sudo apt-get install bubblewrap socat

# Fedora
sudo dnf install bubblewrap socat
```

macOS는 별도 설치 없이 바로 사용 가능합니다.

### 활성화

Claude Code에서 `/sandbox` 명령을 입력하면 메뉴가 나타납니다.

### 샌드박스 모드 두 가지

| 모드 | 설명 |
|---|---|
| 자동 허용 모드 | 샌드박스 안에서는 자동 실행, 범위 밖은 일반 승인 절차 |
| 일반 권한 모드 | 샌드박스 안에서도 매번 승인 필요 |

---

## 특정 경로에 쓰기 권한 추가하기

`~/.kube`, `/tmp/build` 처럼 프로젝트 외부에 써야 하는 경우가 있습니다. `settings.json`에 추가할 수 있습니다.

```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube", "//tmp/build"]
    }
  }
}
```

### 경로 표기법

| 접두사 | 의미 | 예시 |
|---|---|---|
| `//` | 루트부터 절대 경로 | `//tmp/build` → `/tmp/build` |
| `~/` | 홈 디렉토리 기준 | `~/.kube` → `/home/user/.kube` |
| `/` | 설정 파일 위치 기준 | `/build` → `$설정파일위치/build` |
| `./` 또는 없음 | 상대 경로 | `./output` |

## 보안 효과

### 프롬프트 주입 공격 차단

악의적인 코드나 파일이 Claude를 속이더라도 샌드박스가 다음을 막습니다.

**파일 보호:**
- `~/.bashrc` 같은 설정 파일 수정 불가
- `/bin/` 등 시스템 파일 수정 불가

**네트워크 보호:**
- 공격자 서버로 데이터 유출 불가
- 악성 스크립트 다운로드 불가
- 승인되지 않은 API 호출 불가

### 위협 범위 축소

다음 위협도 샌드박스로 제한됩니다.

- **악성 npm 패키지**: 외부 서버 연결 차단
- **보안 취약점 있는 빌드 스크립트**: 시스템 파일 접근 차단
- **프롬프트 주입**: 파일/네트워크 실제 피해 방지

---

## 주의할 보안 한계

| 한계 | 설명 |
|---|---|
| 트래픽 내용 비검사 | 허용된 도메인의 실제 내용은 검사하지 않음 |
| `github.com` 허용 위험 | 넓은 도메인 허용 시 데이터 유출 가능성 |
| Unix 소켓 위험 | `docker.sock` 허용 시 호스트 시스템 접근 가능 |
| 넓은 쓰기 권한 위험 | `$PATH` 포함 폴더 쓰기 허용 시 권한 상승 가능 |

---

## 샌드박스와 권한 설정의 관계

두 기능은 **겹치지 않고 보완**합니다.

| 구분 | 적용 대상 | 적용 시점 |
|---|---|---|
| 권한 설정 | 모든 도구 (Bash, Read, Edit, WebFetch 등) | 도구 실행 전 |
| 샌드박스 | Bash 명령과 그 하위 프로세스 | 실행 중 OS 수준에서 |

---

## 오픈소스 샌드박스 런타임

샌드박스 핵심 기능은 npm 패키지로 공개되어 있습니다. MCP 서버나 다른 프로그램을 격리 실행할 때도 활용할 수 있습니다.

```bash
npx @anthropic-ai/sandbox-runtime <실행할_명령>
```

---

## 알려진 제한 사항

| 항목 | 설명 |
|---|---|
| 성능 | 파일 작업이 약간 느려질 수 있음 |
| 호환성 | 일부 도구는 특별 설정이 필요하거나 샌드박스 밖에서만 작동 |
| 플랫폼 | macOS, Linux, WSL2 지원. WSL1과 Windows 네이티브는 미지원 |
| `watchman` | 샌드박스와 호환 안 됨. Jest는 `--no-watchman` 옵션 사용 |
| `docker` | 샌드박스와 호환 안 됨. `excludedCommands`에 추가 권장 |

## 예시 케이스

### 상황 1: 알 수 없는 npm 패키지 설치
새 패키지를 설치했는데 그 안에 악성 코드가 포함된 경우, 샌드박스는 패키지가 외부 공격자 서버에 연결하거나 홈 디렉토리의 SSH 키를 읽으려는 시도를 차단합니다.

### 상황 2: AI가 자율적으로 코드를 작성할 때
긴 작업을 Claude에게 맡기고 자리를 비울 때, 샌드박스를 활성화하면 Claude가 프로젝트 폴더 밖의 시스템 파일을 건드리지 않아 마음 놓고 자리를 비울 수 있습니다.

### 상황 3: 쿠버네티스 설정 파일 업데이트 필요
`kubectl`이 `~/.kube`에 접근해야 하는 경우, `allowWrite: ["~/.kube"]`를 추가하면 그 폴더만 쓰기 허용됩니다.
