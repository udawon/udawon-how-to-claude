# 확장 기능 완전 비교 가이드

> Claude Code의 다양한 확장 기능들을 언제, 어떻게 써야 하는지 한눈에 비교합니다.

## 한눈에 보는 확장 기능 지도

Claude Code는 기본 기능 위에 여러 확장 기능을 쌓아 사용할 수 있습니다. 마치 스마트폰 앱처럼, 기본 기능만으로도 충분하지만 필요에 따라 기능을 추가할 수 있습니다.

```
[Claude Code 기본]
        ↓
  항상 필요한 것 → CLAUDE.md (프로젝트 지침서)
        ↓
  가끔 필요한 것 → 스킬 (커스텀 명령어)
        ↓
  외부 연결 필요 → MCP (외부 서비스 연결)
        ↓
  자동화 필요   → 훅 (이벤트 기반 자동 실행)
        ↓
  복잡한 작업   → 서브에이전트 (격리된 병렬 작업)
```

---

## 기능별 한눈에 비교

| 기능 | 하는 일 | 언제 쓰나요? | 예시 |
|------|---------|-------------|------|
| **CLAUDE.md** | 매 대화에 불려오는 지속적 맥락 | "항상 이렇게 해" 규칙 | "npm 대신 pnpm 사용. 커밋 전 테스트 실행." |
| **스킬** | Claude가 사용할 수 있는 지식·워크플로우 | 재사용 가능한 내용, 반복 작업 | `/deploy`로 배포 체크리스트 실행, API 스타일 가이드 |
| **서브에이전트** | 격리된 실행 컨텍스트 (요약만 반환) | 컨텍스트 격리, 병렬 작업 | 많은 파일을 읽지만 핵심 결과만 반환하는 리서치 |
| **에이전트 팀** | 독립적인 Claude Code 세션 조율 | 병렬 연구, 기능 개발, 경쟁 가설 | 보안·성능·테스트를 동시에 검토하는 리뷰어 팀 |
| **MCP** | 외부 서비스 연결 | 외부 데이터 또는 작업 | 데이터베이스 쿼리, Slack 전송, 브라우저 제어 |
| **훅** | 이벤트에 반응하는 결정론적 스크립트 | LLM 없이 예측 가능한 자동화 | 파일 편집 후 ESLint 자동 실행 |

**플러그인**: 스킬, 훅, 서브에이전트, MCP 서버를 하나의 설치 패키지로 묶는 포장 레이어. 여러 저장소에 동일한 설정을 재사용하거나 다른 사람과 공유할 때 사용합니다.

## 비슷해 보이는 기능 구별하기

### 스킬 vs 서브에이전트

| 측면 | 스킬 | 서브에이전트 |
|------|------|------------|
| **무엇인가요?** | 재사용 가능한 지침·지식·워크플로우 | 별도 컨텍스트를 가진 격리된 작업자 |
| **핵심 장점** | 여러 컨텍스트에서 내용 공유 | 컨텍스트 격리. 작업 내용은 별도, 요약만 반환 |
| **가장 좋을 때** | 참고 자료, 호출 가능한 워크플로우 | 많은 파일을 읽는 작업, 병렬 작업, 전문 작업자 |

**스킬**: 참고 자료형(API 스타일 가이드처럼 대화 내내 참고) 또는 실행형(`/deploy`처럼 특정 행동 지시) 두 종류 모두 가능합니다.

**서브에이전트**: 수십 개 파일을 읽거나 광범위한 검색을 하더라도 메인 대화에는 요약만 전달됩니다. 컨텍스트를 아껴야 할 때 유용합니다.

**조합 사용**: 서브에이전트가 특정 스킬을 미리 불러올 수 있고(`skills:` 필드), 스킬이 격리된 컨텍스트에서 실행될 수도 있습니다(`context: fork`).

---

### CLAUDE.md vs 스킬

| 측면 | CLAUDE.md | 스킬 |
|------|-----------|------|
| **불려오는 시점** | 매 세션 자동으로 | 필요 시 (요청 또는 자동 감지) |
| **파일 임포트** | 가능 (`@경로` 문법) | 가능 |
| **워크플로우 실행** | 불가능 | 가능 (`/<이름>`) |
| **가장 좋을 때** | "항상 이렇게 해" 규칙 | 참고 자료, 호출 가능한 워크플로우 |

**CLAUDE.md에 넣을 것**: Claude가 항상 알아야 하는 것 — 코딩 규칙, 빌드 명령어, 프로젝트 구조, "절대 하지 말 것" 규칙.

**스킬에 넣을 것**: 가끔 필요한 참고 자료(API 문서, 스타일 가이드)나 `/<이름>`으로 실행하는 워크플로우(deploy, review, release).

**실용 규칙**: CLAUDE.md를 200줄 이하로 유지하세요. 커지면 참고 자료를 스킬로 옮기거나 `.claude/rules/`로 분리하세요.

---

### CLAUDE.md vs Rules vs 스킬 3방향 비교

| 측면 | CLAUDE.md | `.claude/rules/` | 스킬 |
|------|-----------|------------------|------|
| **불려오는 시점** | 매 세션 | 매 세션, 또는 해당 파일 접근 시 | 요청 시 또는 관련성 감지 시 |
| **범위** | 프로젝트 전체 | 파일 경로별 범위 가능 | 특정 작업 |
| **가장 좋을 때** | 핵심 규칙과 빌드 명령어 | 언어/디렉토리별 가이드라인 | 참고 자료, 반복 워크플로우 |

---

### 서브에이전트 vs 에이전트 팀

| 측면 | 서브에이전트 | 에이전트 팀 |
|------|------------|-----------|
| **컨텍스트** | 자체 컨텍스트, 결과를 호출자에게 반환 | 자체 컨텍스트, 완전히 독립적 |
| **소통 방식** | 메인 에이전트에게만 결과 보고 | 팀원끼리 직접 메시지 주고받음 |
| **조율 방식** | 메인 에이전트가 모든 작업 관리 | 공유 작업 목록으로 자체 조율 |
| **가장 좋을 때** | 결과만 중요한 집중 작업 | 논의와 협력이 필요한 복잡한 작업 |
| **토큰 비용** | 낮음: 결과가 요약되어 반환 | 높음: 각 팀원이 별도 Claude 인스턴스 |

**서브에이전트 사용**: 빠르고 집중적인 작업이 필요할 때 — 질문 조사, 주장 검증, 파일 검토. 결과만 반환하고 대화는 깔끔하게 유지됩니다.

**에이전트 팀 사용**: 팀원들이 결과를 공유하고, 서로 도전하고, 독립적으로 조율해야 할 때. 경쟁 가설을 가진 연구, 병렬 코드 리뷰, 각 팀원이 별도 부분을 담당하는 신기능 개발에 적합합니다.

> **참고**: 에이전트 팀은 실험적 기능으로 기본 비활성화 상태입니다.

### MCP vs 스킬

| 측면 | MCP | 스킬 |
|------|-----|------|
| **무엇인가요?** | 외부 서비스 연결 프로토콜 | 지식·워크플로우·참고 자료 |
| **제공하는 것** | 도구와 데이터 접근 | 지식, 워크플로우, 참고 자료 |
| **예시** | Slack 연동, 데이터베이스 쿼리 | 코드 리뷰 체크리스트, 배포 워크플로우 |

이 두 가지는 다른 문제를 해결하며 함께 사용할 때 강력합니다.

**MCP**: Claude가 외부 시스템과 상호작용하는 능력을 줍니다. MCP 없이는 데이터베이스 쿼리나 Slack 메시지 전송이 불가능합니다.

**스킬**: Claude에게 그 도구를 효과적으로 사용하는 방법을 가르칩니다. 데이터베이스 스키마, 일반적인 쿼리 패턴, 어떤 테이블을 언제 써야 하는지 등을 담을 수 있습니다.

**예시**: MCP 서버로 데이터베이스에 연결하고, 스킬로 Claude에게 데이터 모델, 일반 쿼리 패턴, 작업별 테이블 사용법을 가르칩니다.

## 컨텍스트 비용 이해하기

각 기능은 Claude의 컨텍스트 창을 다르게 소비합니다. 너무 많으면 컨텍스트 창이 가득 차거나, Claude의 집중도가 낮아질 수 있습니다.

| 기능 | 불려오는 시점 | 불려오는 내용 | 컨텍스트 비용 |
|------|-------------|------------|------------|
| **CLAUDE.md** | 세션 시작 | 전체 내용 | 매 요청마다 발생 |
| **스킬** | 세션 시작 + 사용 시 | 시작 시 설명, 사용 시 전체 내용 | 낮음 (설명만 매 요청) |
| **MCP 서버** | 세션 시작 | 모든 도구 정의와 스키마 | 매 요청마다 발생 |
| **서브에이전트** | 생성 시 | 새 컨텍스트 + 지정된 스킬 | 메인 세션과 격리 |
| **훅** | 트리거 시 | 없음 (외부에서 실행) | 제로 (출력 반환 시 제외) |

**절약 팁**:
- CLAUDE.md는 200줄 이하로 유지
- `disable-model-invocation: true`로 수동 호출 스킬은 컨텍스트에서 완전히 숨기기
- MCP 도구 검색으로 불필요한 도구 정의 로드 방지
- 많은 파일을 읽는 작업은 서브에이전트에게 위임

## 기능 조합 패턴

기능들은 서로 보완하도록 설계되어 있습니다.

| 조합 패턴 | 작동 방식 | 예시 |
|-----------|---------|------|
| **스킬 + MCP** | MCP로 연결, 스킬로 활용법 가르치기 | MCP가 DB 연결, 스킬이 스키마와 쿼리 패턴 제공 |
| **스킬 + 서브에이전트** | 스킬이 서브에이전트를 병렬로 실행 | `/audit` 스킬이 보안·성능·스타일 서브에이전트 생성 |
| **CLAUDE.md + 스킬** | CLAUDE.md에 항상 켜는 규칙, 스킬에 가끔 쓰는 참고 자료 | CLAUDE.md: "우리 API 규칙 따르기", 스킬: 전체 API 스타일 가이드 |
| **훅 + MCP** | 훅이 이벤트 발생 시 MCP를 통해 외부 서비스에 알림 | 중요 파일 수정 시 Slack 알림 전송 |

## 기능 선택 결정 가이드

```
무엇이 필요한가요?

→ Claude가 항상 알아야 할 규칙/설정?
   ✅ CLAUDE.md

→ 특정 상황이나 명령으로 실행하는 작업?
   ✅ 스킬 (Skills)

→ 외부 서비스(DB, Slack, GitHub)와 상호작용?
   ✅ MCP

→ 특정 이벤트 발생 시 자동 실행?
   ✅ 훅 (Hooks)

→ 많은 파일 처리, 컨텍스트 격리, 병렬 작업?
   ✅ 서브에이전트 (Subagents)

→ 여러 Claude 세션이 협력해야 하는 복잡한 작업?
   ✅ 에이전트 팀 (Agent Teams) [실험적]

→ 이 모든 설정을 패키지로 만들어 공유?
   ✅ 플러그인 (Plugins)
```

## 예시 케이스

### 케이스 1: 개인 개발자의 기본 설정

**상황**: 혼자 개발하는 사이드 프로젝트에 Claude Code를 처음 설정합니다.

**해결책**:
1. **CLAUDE.md**: 프로젝트 스택, 빌드 명령어, 코딩 규칙 저장
2. 필요 시 **스킬** 추가: 커밋 작성 스킬, 코드 리뷰 스킬
3. 데이터 작업이 많아지면 **MCP**로 DB 연결

```markdown
# 프로젝트 설정 (CLAUDE.md)
- 빌드: `npm run build`
- 테스트: `npm test`
- 스택: Next.js, Prisma, Tailwind
- 절대 any 타입 사용 금지
```

### 케이스 2: 팀 프로젝트 설정

**상황**: 5인 개발팀이 Claude Code를 통일되게 사용하고 싶습니다.

**해결책**:
1. **`.claude/CLAUDE.md`**: 팀 규칙을 버전 관리에 포함
2. **`.mcp.json`**: 팀 공유 MCP 서버 설정 (GitHub, Sentry 등)
3. **스킬**: 팀 배포 워크플로우, PR 리뷰 기준
4. **훅**: 파일 편집 후 자동 린팅

모든 팀원이 `git clone` 하면 즉시 동일한 환경을 사용할 수 있습니다.

### 케이스 3: 대규모 자동화 파이프라인

**상황**: 새 기능을 개발할 때 Claude가 코드 작성부터 PR 생성, 테스트 확인까지 자동으로 해주길 원합니다.

**해결책**:
1. **MCP**: GitHub, Jira 연결
2. **스킬** (`/feature-complete`): 체크리스트 자동 실행
3. **훅**:
   - `PreToolUse`: 위험한 명령 차단
   - `PostToolUse`: 파일 편집 후 자동 린팅
   - `Stop` (에이전트 기반): 모든 테스트 통과 확인
4. **서브에이전트**: 보안 리뷰, 성능 분석을 병렬로 실행

```
/feature-complete JIRA-456
```

Claude가 JIRA 이슈를 읽고, 코드를 작성하고, 테스트를 추가하고, 서브에이전트로 보안과 성능을 동시에 검토하고, 모든 것이 통과하면 PR을 생성합니다.
