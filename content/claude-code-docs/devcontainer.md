# 개발 컨테이너(Devcontainer) — 안전하고 일관된 개발 환경

개발 컨테이너(devcontainer)는 마치 "개발용 모래 놀이터"와 같습니다. 아이들이 모래 놀이터에서 마음껏 뛰어놀아도 다른 곳이 더러워지지 않는 것처럼, 컨테이너 안에서는 Claude Code가 마음껏 코드를 실행해도 내 실제 컴퓨터에 영향을 주지 않습니다.

Anthropic이 제공하는 레퍼런스 devcontainer 설정은 이미 보안이 강화된 환경으로 미리 구성되어 있어, 팀 전체가 동일한 개발 환경을 빠르게 시작할 수 있습니다.

---

## 컨테이너가 필요한 이유

Claude Code를 일반 모드로 사용하면 특정 작업마다 권한을 묻는 창이 뜹니다. 하지만 컨테이너 안에서는 실제 시스템과 격리되어 있으므로, `--dangerously-skip-permissions` 옵션으로 권한 확인 없이 완전 자동화된 방식으로 Claude Code를 실행할 수 있습니다.

> **주의**: 컨테이너도 완전한 보안을 보장하지는 않습니다. 악의적인 코드가 컨테이너 내의 Claude Code 자격증명을 탈취할 수 있습니다. **신뢰할 수 있는 저장소에서만 사용하세요.**

---

## 주요 특징

| 특징 | 설명 |
|------|------|
| **Node.js 20 기반** | 최신 안정 버전의 Node.js로 미리 구성된 환경 |
| **방화벽 내장** | 필요한 서비스에만 네트워크 접근 허용 |
| **개발 도구 포함** | git, ZSH, fzf 등 생산성 도구 기본 설치 |
| **VS Code 통합** | 확장 프로그램과 설정이 미리 최적화됨 |
| **세션 유지** | 컨테이너를 재시작해도 명령 기록과 설정이 보존됨 |
| **플랫폼 호환** | macOS, Windows, Linux 모두 지원 |

---

## 4단계로 시작하기

1. VS Code와 **Remote - Containers 확장 프로그램**을 설치합니다.
2. [Claude Code 레퍼런스 구현 저장소](https://github.com/anthropics/claude-code/tree/main/.devcontainer)를 클론합니다.
3. VS Code에서 해당 저장소를 엽니다.
4. 팝업이 뜨면 **"Reopen in Container"** 를 클릭합니다.
   (팝업이 없으면 `Cmd+Shift+P` → "Remote-Containers: Reopen in Container")

---

## 설정 파일 구조

devcontainer는 세 가지 주요 파일로 구성됩니다:

| 파일 | 역할 |
|------|------|
| `devcontainer.json` | 컨테이너 설정, VS Code 확장, 볼륨 마운트 등 전반적인 설정 |
| `Dockerfile` | 컨테이너 이미지 정의 — 어떤 도구가 설치될지 결정 |
| `init-firewall.sh` | 컨테이너 시작 시 방화벽 규칙 적용 |

---

## 보안 기능 상세

컨테이너의 방화벽은 다층 보안을 구현합니다:

### 허용되는 외부 연결
- npm 레지스트리
- GitHub
- Claude API
- DNS 조회 (도메인 이름 해석)
- SSH 연결

### 차단되는 연결
- 위 목록 외 모든 외부 네트워크 연결

이를 통해 악의적인 코드가 내부 데이터를 외부로 전송하거나, 예상치 못한 외부 서버와 통신하는 것을 방지합니다.

### 격리 원칙
- 컨테이너 안에서 일어나는 일은 내 실제 컴퓨터에 영향을 주지 않습니다.
- 방화벽은 컨테이너 시작 시 자동으로 검증됩니다.

---

## 커스터마이징 옵션

기본 설정을 조직의 필요에 맞게 수정할 수 있습니다:

- **VS Code 확장 추가/제거**: 팀의 개발 워크플로에 맞게 조정
- **리소스 할당 조정**: 다양한 하드웨어 환경에 맞게 메모리/CPU 설정
- **네트워크 접근 권한 수정**: 추가로 허용할 도메인 지정
- **셸 설정 커스터마이즈**: 팀 공통 개발 도구와 별칭(alias) 설정

## 활용 사례

### 클라이언트 프로젝트 격리

여러 클라이언트의 프로젝트를 각각 별도 컨테이너에서 개발합니다. 클라이언트 A의 자격증명과 코드가 클라이언트 B의 환경에 절대 섞이지 않습니다.

### 신규 팀원 온보딩

신규 팀원이 모든 도구와 설정이 미리 구성된 개발 환경을 몇 분 안에 시작할 수 있습니다. "내 컴퓨터에서는 되는데…"라는 문제가 없어집니다.

### CI/CD 환경 일치

devcontainer 설정을 CI/CD 파이프라인과 동일하게 유지하면 개발 환경과 운영 환경의 차이로 인한 버그를 방지할 수 있습니다.

## 예시 케이스

**상황**: 보안 컨설팅 회사의 개발자가 동시에 3개 클라이언트의 코드베이스 작업을 합니다. 각 클라이언트는 자신의 코드와 비밀 정보가 다른 클라이언트에게 노출되지 않기를 원합니다. 또한 보안팀은 Claude Code가 자동으로 다양한 코드 수정을 수행하길 원하지만, 매번 권한 확인을 거치는 것은 비효율적입니다.

**해결책**:
1. 각 클라이언트 프로젝트마다 별도의 devcontainer를 설정합니다.
2. 컨테이너의 방화벽이 외부 데이터 전송을 차단하므로 데이터 격리가 보장됩니다.
3. 컨테이너 안에서 `claude --dangerously-skip-permissions`로 권한 확인 없이 Claude Code를 실행합니다.
4. 작업 완료 후 컨테이너를 종료해도 명령 기록이 유지되어 다음 작업 때 이어서 사용합니다.

결과: 클라이언트 간 데이터 격리가 보장되고, Claude Code가 완전 자동화 모드로 효율적으로 작동합니다.
