## 이게 뭔가요?

서브에이전트(sub-agent)는 **특정 종류의 일만 전담하도록 따로 만들어 둔 AI 조수**입니다. 회사로 비유하면, 사장(메인 대화)이 모든 일을 혼자 처리하는 대신 "코드 검사는 검사 담당에게, 버그 잡는 일은 디버깅 담당에게" 하고 업무를 나눠 맡기는 것과 같습니다.

핵심은 서브에이전트가 **자기만의 책상(별도의 대화 창)** 에서 일한다는 점입니다. 예를 들어 테스트를 돌리면 수천 줄짜리 로그(작업 기록)가 쏟아지는데, 이 지저분한 결과물은 서브에이전트의 책상에만 쌓이고, 사장(메인 대화)에게는 "실패한 테스트 3개, 원인은 X, 수정 완료" 같은 **깔끔한 요약만** 돌아옵니다.

각 서브에이전트는 다음을 자기만의 것으로 가집니다.

- 독립된 대화 창(컨텍스트 윈도우 — AI가 한 번에 기억하는 작업 공간)
- 맞춤 지시문(시스템 프롬프트 — 이 조수가 어떻게 행동할지 정해 둔 설명서)
- 쓸 수 있는 도구의 범위 (예: 읽기만 허용, 파일 수정 금지)
- 별도의 권한 설정

Claude는 사용자의 요청이 어떤 서브에이전트의 "설명"과 맞아떨어지면, 알아서 그 조수에게 일을 넘깁니다.

```mermaid

    A["메인 대화"] -->|"작업 위임"| B["서브에이전트<br/>별도의 책상"]
    B -->|"요약만 반환"| A
    B --> C["대량 로그·검색<br/>여기서 처리"]
    style A fill:#1d4ed8,color:#fff
    style B fill:#15803d,color:#fff
    style C fill:#475569,color:#fff
```

## 왜 알아야 하나요?

서브에이전트를 쓰면 좋은 점을 한눈에 정리하면 이렇습니다.

| 장점 | 설명 |
|------|------|
| **대화창이 깨끗해짐** | 검색 결과·로그 같은 잡다한 내용이 메인 대화에 쌓이지 않아, Claude가 핵심 맥락을 더 오래 기억합니다 |
| **실수 방지 (제약)** | 특정 조수에게는 "읽기만 가능, 파일 수정 금지"처럼 권한을 묶어 둘 수 있어 사고를 막습니다 |
| **재사용** | 한 번 만든 조수를 모든 프로젝트에서 다시 쓸 수 있습니다 |
| **전문화** | 한 가지 일에 집중된 지시문을 줘서 그 분야를 더 잘하게 만듭니다 |
| **비용 절감** | 간단한 일은 빠르고 저렴한 모델(예: Haiku)에게 맡겨 돈을 아낍니다 |

반대로 모르면, 모든 일을 하나의 대화창에서 처리하다가 맥락이 꽉 차서 Claude가 앞부분을 잊어버리거나, 위험한 작업까지 권한 제한 없이 실행될 수 있습니다.

## 어떻게 하나요?

### 방법 1: `/agents` 명령어로 만들기 (권장)

가장 쉬운 방법은 Claude Code 안에서 `/agents` 명령어를 입력하는 것입니다. 화면 안내를 따라가며 클릭만 하면 됩니다.

1. Claude Code에서 `/agents` 입력
2. **Library**(보관함) 탭으로 이동 → **Create new agent**(새 에이전트 만들기) → **Personal**(내 계정용) 선택. 이렇게 하면 `~/.claude/agents/` 폴더에 저장되어 모든 프로젝트에서 쓸 수 있습니다
3. **Generate with Claude**(Claude가 자동 생성) 선택 후, 원하는 조수를 말로 설명. Claude가 이름·설명·지시문을 알아서 만들어 줍니다
4. **도구 선택**: 읽기만 하는 검토용 조수라면 "Read-only tools"(읽기 전용 도구)만 남기고 나머지는 끕니다
5. **모델 선택**: 코드 패턴 분석에는 균형 잡힌 Sonnet이 무난합니다
6. **색상 선택**: 화면에서 어떤 조수가 일하는지 구분하기 위한 표시 색입니다
7. **기억 설정**: "User scope"(사용자 범위)를 고르면 이 조수가 여러 대화에 걸쳐 배운 내용을 기억합니다. 필요 없으면 "None"(없음)
8. `s` 또는 `Enter`로 저장하면 바로 사용 가능

### 방법 2: 마크다운 파일로 직접 만들기

조수는 결국 **마크다운(.md) 파일 한 장**입니다. 파일 위쪽에 설정을 적고(YAML — 설정을 줄 단위로 적는 형식), 아래쪽에 지시문을 적습니다.

```markdown
---
name: code-reviewer
description: 코드 품질과 모범 사례를 검토합니다
tools: Read, Glob, Grep
model: sonnet
---

당신은 코드 리뷰어입니다. 호출되면 코드를 분석하고
품질·보안·모범 사례에 대해 구체적이고 실행 가능한 피드백을 주세요.
```

- 위쪽 `---` 사이가 **설정 칸**(이름, 설명, 쓸 도구, 사용할 모델 등)
- 아래쪽 글이 **지시문**(이 조수가 어떻게 행동할지 알려주는 시스템 프롬프트)

> 주의: 파일을 직접 만들거나 고쳤다면, Claude Code를 껐다 켜야(세션 재시작) 새 조수가 인식됩니다. 반면 `/agents` 화면으로 만든 조수는 즉시 적용됩니다.

### 기본으로 내장된 서브에이전트

직접 만들지 않아도, Claude Code에는 처음부터 들어 있어 상황에 맞게 자동으로 쓰이는 조수들이 있습니다.

| 조수 | 모델 | 도구 | 하는 일 |
|------|------|------|---------|
| **Explore** | Haiku (빠름) | 읽기 전용 | 코드베이스를 뒤져 파일을 찾고 분석 |
| **Plan** | 메인과 동일 | 읽기 전용 | 계획 모드에서 필요한 정보 수집 |
| **General-purpose** | 메인과 동일 | 모든 도구 | 탐색과 수정이 둘 다 필요한 복잡한 작업 |
| **statusline-setup** | Sonnet | 설정용 | `/statusline` 실행 시 상태 표시줄 설정 |
| **claude-code-guide** | Haiku | 안내용 | Claude Code 기능에 대해 물어볼 때 답변 |

### 설정 항목 (파일 위쪽 칸에 적는 것들)

이름(`name`)과 설명(`description`)만 필수이고, 나머지는 선택입니다.

| 항목 | 필수 | 설명 |
|------|------|------|
| `name` | 필수 | 고유 이름 (소문자와 하이픈만 사용) |
| `description` | 필수 | Claude가 "언제 이 조수에게 일을 맡길지" 판단하는 기준 |
| `tools` | 선택 | 쓸 수 있는 도구 목록. 생략하면 메인 대화의 모든 도구를 물려받음 |
| `disallowedTools` | 선택 | 금지할 도구 목록 |
| `model` | 선택 | `sonnet`·`opus`·`haiku`·`fable`·전체 모델 ID·`inherit`(메인과 동일). 기본값은 `inherit` |
| `permissionMode` | 선택 | 권한을 어떻게 처리할지 (아래 표 참고) |
| `maxTurns` | 선택 | 멈추기 전 최대 작업 횟수 |
| `skills` | 선택 | 시작할 때 미리 넣어 둘 스킬(추가 능력) 목록 |
| `mcpServers` | 선택 | 이 조수만 쓸 외부 도구 연결(MCP) |
| `hooks` | 선택 | 특정 시점에 자동 실행할 동작 |
| `memory` | 선택 | 대화 사이에 기억을 남길 범위 (`user`·`project`·`local`) |
| `background` | 선택 | `true`면 항상 뒤에서(백그라운드) 실행 |
| `effort` | 선택 | 작업 강도 (`low`·`medium`·`high`·`xhigh`·`max`) — 모델에 따라 선택지 다름 |
| `isolation` | 선택 | `worktree`면 저장소 사본을 따로 떼어 격리된 공간에서 작업 |
| `color` | 선택 | 화면 표시 색 (`red`·`blue`·`green` 등) |
| `initialPrompt` | 선택 | 이 조수가 메인 역할로 켜질 때 자동으로 던지는 첫 메시지 |

### 저장 위치와 우선순위

같은 이름의 조수가 여러 곳에 있으면, **우선순위가 높은 쪽이 이깁니다.**

| 위치 | 적용 범위 | 우선순위 |
|------|---------|---------|
| 관리자 배포 설정 (managed settings) | 회사 조직 전체 | 1 (가장 높음) |
| `--agents` 명령행 옵션 | 지금 켠 세션에만 | 2 |
| `.claude/agents/` | 현재 프로젝트 | 3 |
| `~/.claude/agents/` | 내 모든 프로젝트 | 4 |
| 플러그인의 `agents/` 폴더 | 플러그인 켜진 곳 | 5 (가장 낮음) |

팀과 함께 쓸 조수라면 프로젝트 폴더(`.claude/agents/`)에 넣고 버전 관리(Git)에 올려, 팀원 모두가 같이 쓰고 개선하게 하는 것이 좋습니다.

### 도구와 권한 제어하기

조수가 무엇을 할 수 있는지를 두 가지 방식으로 묶을 수 있습니다.

- **허용 목록(allowlist)**: `tools`에 적은 것만 쓸 수 있음

  ```yaml
  ---
  name: safe-researcher
  description: 제한된 기능으로 조사만 하는 조수
  tools: Read, Grep, Glob, Bash
  ---
  ```

- **금지 목록(denylist)**: `disallowedTools`에 적은 것만 막고 나머지는 다 물려받음

  ```yaml
  ---
  name: no-writes
  description: 파일 쓰기·수정만 빼고 전부 사용
  disallowedTools: Write, Edit
  ---
  ```

권한 모드(`permissionMode`)는 작업할 때마다 "물어볼지 / 자동 승인할지"를 정합니다.

| 모드 | 동작 |
|------|------|
| `default` | 표준 — 필요할 때 사용자에게 물어봄 |
| `acceptEdits` | 작업 폴더 안의 파일 편집을 자동 승인 |
| `auto` | 자동 모드 — 뒤에서 분류기가 명령을 점검해 처리 |
| `dontAsk` | 묻는 작업은 자동 거부 (미리 허용한 도구만 실행) |
| `bypassPermissions` | 권한 확인을 전부 건너뜀 (위험 — 아래 주의 참고) |
| `plan` | 계획 모드 — 읽기만 하고 수정 안 함 |

### 지속 기억 (Persistent Memory)

`memory` 항목을 켜면, 조수가 여러 대화에 걸쳐 배운 내용(코드 패턴, 반복되는 문제 등)을 파일로 쌓아 둡니다. 다음 작업 때 "이전 기억을 확인하고 시작해줘"라고 하면 그걸 참고합니다.

| 범위 | 저장 위치 | 언제 쓰나 |
|------|---------|---------|
| `user` | `~/.claude/agent-memory/<이름>/` | 모든 프로젝트에서 배운 걸 공유 |
| `project` | `.claude/agent-memory/<이름>/` | 이 프로젝트 전용 (Git에 올려 팀과 공유) |
| `local` | `.claude/agent-memory-local/<이름>/` | 이 프로젝트 전용 (Git에 안 올림) |

### 앞에서 vs 뒤에서 실행 (Foreground / Background)

- **앞에서(포그라운드)**: 조수가 일을 끝낼 때까지 메인 대화가 기다립니다. 권한이 필요하면 그때그때 사용자에게 물어봅니다.
- **뒤에서(백그라운드)**: 사용자가 다른 일을 하는 동안 조수가 동시에 일합니다. 단, 이미 허용된 권한만 쓰고, **새로 물어봐야 하는 작업은 자동 거부**됩니다.

실행 중인 작업을 뒤로 넘기려면 **Ctrl+B**(맥·윈도우 공통)를 누르거나, "이건 백그라운드로 돌려줘"라고 말하면 됩니다.

### 조수가 또 다른 조수 부르기 (중첩 서브에이전트)

Claude Code **v2.1.172 버전부터**, 서브에이전트(조수)가 **자기 밑에 또 다른 조수를 둘 수 있습니다**. 회사로 치면, 사장이 부른 팀장(서브에이전트)이 일이 커지자 자기 팀원(하위 서브에이전트)을 따로 뽑아 나눠 맡기는 것과 같습니다. 이렇게 하면 중간에 오가는 잡다한 작업 내용은 팀장 책상에서만 처리되고, 사장(메인 대화)에게는 맨 위 조수의 **최종 요약만** 올라옵니다.

대표적인 쓰임새는 이렇습니다. 코드 리뷰 조수가 문제를 여러 개 찾았을 때, 발견한 건마다 "이게 진짜 문제인지 검증하는 조수"를 하나씩 띄워 병렬로 확인시키는 식입니다. 검증 과정의 지저분한 중간 출력은 리뷰 조수 선에서 끝나고, 메인 대화에는 정리된 결과만 돌아옵니다.

쓰려면 두 가지 조건이 있습니다.

- **조건 1 — `Agent` 도구 보유**: 그 조수의 `tools` 목록에 `Agent`(다른 조수를 부르는 도구)가 들어 있어야 합니다. `tools`를 아예 안 적으면 메인 대화의 모든 도구를 물려받으므로 `Agent`도 자동으로 포함됩니다. 반대로 `tools`에 쓸 도구를 직접 나열(허용 목록)했다면, 거기에 `Agent`를 빠뜨리면 하위 조수를 못 부릅니다.
- **조건 2 — 최대 5단계 깊이**: 조수가 조수를 부르는 깊이는 5단계가 한계입니다. 5단계째 조수는 `Agent` 도구를 아예 받지 못해 더 밑으로 내려갈 수 없습니다. 이 한계는 고정값이라 늘릴 수 없습니다 (조수가 끝없이 새끼를 쳐서 통제 불능이 되는 걸 막는 안전장치입니다).

끄는 법은 간단합니다. 특정 조수가 하위 조수를 만들지 못하게 하려면, `tools`에서 `Agent`를 빼거나 `disallowedTools: Agent`로 막으면 됩니다. 그러면 그 조수는 자기 일만 하고 다른 조수는 못 부르는 '말단' 조수가 됩니다.

<div class="example-case">
<strong>예시: 리뷰 조수가 검증 조수를 띄우기</strong>

태양씨의 코드 리뷰 조수가 수정된 파일에서 의심스러운 부분 5곳을 찾았습니다. 이 조수의 `tools`에 `Agent`가 들어 있어, 조수는 의심 건 5개에 대해 각각 "정말 버그인지 따져 보는 검증 조수"를 하나씩 띄웁니다. 다섯 검증 조수가 동시에 돌며 코드를 파헤치지만, 그 과정의 긴 분석 기록은 전부 리뷰 조수 책상에만 쌓입니다. 메인 대화에는 "5곳 중 실제 버그 2곳, 나머지 3곳은 오탐" 같은 최종 요약만 올라옵니다.

</div>

## 실전 예시

<div class="example-case">
<strong>실전 케이스 1: 코드 리뷰 전담 조수 (읽기 전용)</strong>

개발자 태양씨는 코드를 수정할 때마다 품질 점검을 받고 싶지만, 리뷰 조수가 실수로 코드를 건드리는 건 싫었습니다. 그래서 파일 수정 도구(Edit·Write)를 빼고 읽기 도구만 준 조수를 만들었습니다.

```markdown
---
name: code-reviewer
description: 전문 코드 리뷰어. 코드 작성·수정 직후 즉시 사용하세요.
tools: Read, Grep, Glob, Bash
model: inherit
---

당신은 코드 품질과 보안의 높은 기준을 유지하는 시니어 코드 리뷰어입니다.

호출되면:
1. git diff로 최근 변경사항 확인
2. 수정된 파일에 집중
3. 즉시 리뷰 시작

체크리스트: 가독성, 적절한 이름, 중복 없음, 오류 처리, 노출된
비밀번호·키 없음, 입력값 검증, 테스트 충분, 성능.

피드백은 중요도순으로: 반드시 수정 / 수정 권장 / 개선 제안.
```

설명에 "직후 즉시 사용"처럼 적극적으로 부르는 표현을 넣어 두면, Claude가 코드 수정 후 알아서 이 조수를 부릅니다.

</div>

<div class="example-case">
<strong>실전 케이스 2: 대량 출력 가두기</strong>

태양씨가 "테스트를 돌리고 실패한 것만 고쳐줘"라고 했더니, 테스트 로그가 수천 줄이라 메인 대화창이 꽉 차고 Claude가 앞 맥락을 잊는 일이 생겼습니다. 이럴 때 서브에이전트에게 맡기면 됩니다.

"서브에이전트로 테스트를 실행하고, 실패한 테스트와 오류 메시지만 보고해줘"

수천 줄의 로그는 조수의 책상에만 남고, 메인 대화에는 "실패 3건, 원인 X, 수정 완료" 같은 요약만 돌아와 대화가 깔끔하게 유지됩니다.

</div>

## 주의할 점

- **`bypassPermissions`는 신중하게**: 권한 확인을 전부 건너뛰므로, 조수가 승인 없이 파일을 쓰거나 위험한 작업을 할 수 있습니다. 꼭 필요한 경우가 아니면 쓰지 마세요. (단 `rm -rf /` 같은 최상위·홈 폴더 삭제는 이 모드에서도 여전히 물어봅니다.)
- **백그라운드 조수는 못 물어봅니다**: 뒤에서 도는 조수는 권한이 필요한 작업을 만나면 묻지 못하고 자동 거부합니다. 권한 부족으로 실패하면, 같은 일을 앞에서(포그라운드) 다시 돌려 직접 승인하면 됩니다.
- **결과가 많으면 맥락을 잡아먹습니다**: 여러 조수가 각자 긴 결과를 돌려주면, 그 요약들이 메인 대화에 쌓여 다시 공간을 차지합니다. "요약만 보고해줘"라고 범위를 좁혀 주세요.
- **파일을 직접 고쳤으면 재시작**: 마크다운 파일을 손으로 만들거나 수정했다면 세션을 껐다 켜야 반영됩니다. `/agents` 화면으로 만든 건 바로 적용됩니다.
- **이름은 겹치지 않게**: 같은 범위 안에서 두 파일이 같은 `name`을 쓰면, 하나만 남고 나머지는 경고 없이 무시됩니다.

## 정리

- 서브에이전트는 특정 일만 전담하는 AI 조수로, **자기만의 대화 창**에서 일하고 메인 대화에는 요약만 돌려줍니다.
- `/agents` 명령어로 클릭 몇 번에 만들거나, 마크다운 파일 한 장으로 직접 정의할 수 있습니다.
- 도구·권한·기억·모델을 조수마다 따로 묶어, 안전하게 그리고 저렴하게 일을 나눠 맡길 수 있습니다.
