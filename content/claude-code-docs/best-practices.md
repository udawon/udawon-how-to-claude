# 모범 사례

> Anthropic 내부 팀과 다양한 개발자들이 검증한 효과적인 사용 패턴을 소개합니다.

---

## 가장 중요한 원칙: 컨텍스트 윈도우 관리

Claude Code를 잘 쓰기 위한 핵심은 하나입니다.

**컨텍스트 윈도우(칠판)가 꽉 찰수록 성능이 저하됩니다.**

컨텍스트 윈도우는 대화 전체를 담습니다 — 모든 메시지, Claude가 읽은 파일, 실행한 명령어 출력. 하나의 디버깅 세션이나 코드베이스 탐색으로 수만 토큰이 쉽게 소모됩니다.

컨텍스트가 꽉 차면:
- 초반에 준 지시사항을 "잊기" 시작합니다
- 실수가 늘어납니다
- 성능이 눈에 띄게 떨어집니다

> `/context` 명령으로 현재 사용량을 확인하세요.

---

## 1. 검증 수단을 제공하라

> Claude가 스스로 확인할 수 있는 기준을 주세요. 이것이 가장 효과적인 단일 개선책입니다.

검증 수단이 없으면 Claude는 그럴듯해 보이지만 실제로 동작하지 않는 코드를 만들 수 있습니다. 여러분이 유일한 피드백 루프가 되면 모든 실수를 직접 잡아야 합니다.

| 전략 | 모호한 요청 | 검증 포함 요청 |
|------|-----------|--------------|
| **검증 기준 제공** | "이메일 주소를 검증하는 함수를 만들어줘" | "validateEmail 함수를 만들어줘. 테스트 케이스: user@example.com은 true, invalid는 false, user@.com은 false. 구현 후 테스트를 실행해줘" |
| **UI 변경 시각 확인** | "대시보드를 더 보기 좋게 만들어줘" | "[스크린샷 붙여넣기] 이 디자인을 구현해줘. 구현 후 스크린샷을 찍어 원본과 비교하고 차이점을 목록으로 만들어 고쳐줘" |
| **근본 원인 해결** | "빌드가 실패하고 있어" | "빌드가 이 에러로 실패해: [에러 붙여넣기]. 수정하고 빌드가 성공하는지 확인해줘. 에러를 억제하지 말고 근본 원인을 해결해줘" |

## 2. 탐색 → 계획 → 구현 순서로 진행하라

> 바로 코딩에 뛰어들면 잘못된 문제를 해결할 수 있습니다.

**4단계 워크플로우:**

### 1단계: 탐색 (계획 모드)

계획 모드로 진입 (Shift+Tab 두 번)하고 코드를 읽기만 합니다.

```
src/auth를 읽고 세션과 로그인 처리 방식을 파악해줘.
그리고 시크릿 정보를 위한 환경 변수 관리 방법도 살펴봐.
```

### 2단계: 계획 (계획 모드)

```
Google OAuth를 추가하려면 어떤 파일을 바꿔야 해?
세션 흐름은 어떻게 돼? 계획을 세워줘.
```

> `Ctrl+G`로 계획을 에디터에서 직접 편집할 수 있습니다.

### 3단계: 구현 (일반 모드)

```
세운 계획대로 OAuth 흐름을 구현해줘.
콜백 핸들러 테스트를 작성하고 실패하면 고쳐줘.
```

### 4단계: 커밋

```
설명이 잘 된 메시지로 커밋하고 PR을 열어줘
```

> **언제 계획을 건너뛰나요?** 단순한 작업(오타 수정, 로그 추가, 변수명 변경)은 바로 실행하세요. 한 문장으로 변경사항을 설명할 수 있다면 계획은 불필요한 오버헤드입니다.

## 3. 구체적인 프롬프트를 써라

> 정확한 요청일수록 수정 횟수가 줄어듭니다.

| 전략 | 모호한 요청 | 구체적인 요청 |
|------|-----------|-------------|
| **범위 지정** | "foo.py에 테스트를 추가해줘" | "foo.py에서 사용자가 로그아웃된 엣지 케이스를 다루는 테스트를 작성해줘. mock은 쓰지 마." |
| **출처 지시** | "ExecutionFactory가 왜 이런 이상한 API를 가지고 있어?" | "ExecutionFactory의 git 히스토리를 살펴보고 API가 어떻게 만들어졌는지 요약해줘" |
| **패턴 참조** | "캘린더 위젯을 추가해줘" | "홈 페이지의 기존 위젯들(HotDogWidget.php 참고)을 보고 패턴을 파악해서, 사용자가 월을 선택하고 앞뒤로 탐색할 수 있는 캘린더 위젯을 코드베이스에 이미 있는 라이브러리만 써서 구현해줘" |
| **증상 설명** | "로그인 버그 고쳐줘" | "세션 만료 후 로그인이 실패한다는 신고가 있어. src/auth/의 인증 흐름을 확인해줘. 특히 토큰 갱신 부분. 문제를 재현하는 실패 테스트를 작성한 뒤 고쳐줘" |

### 풍부한 정보 제공하기

- **`@`로 파일 참조**: 파일 위치를 설명하는 대신 직접 참조
- **이미지 직접 붙여넣기**: 디자인 목업, 에러 스크린샷 등
- **URL 제공**: 문서, API 레퍼런스 링크
- **데이터 파이핑**: `cat error.log | claude`로 파일 내용 직접 전달

## 4. 환경을 잘 설정하라

### 효과적인 CLAUDE.md 작성

CLAUDE.md는 **프로젝트 규칙서**입니다. Claude가 매 세션마다 자동으로 읽습니다.

`/init`으로 프로젝트를 분석해 자동으로 초안을 생성할 수 있습니다.

```markdown
# 코드 스타일
- ES 모듈 (import/export) 사용, CommonJS (require) 사용 금지
- 가능하면 구조분해 임포트 사용 (예: import { foo } from 'bar')

# 워크플로우
- 일련의 코드 변경 후 타입 체크 필수
- 성능을 위해 전체 테스트 스위트 대신 개별 테스트만 실행
```

**포함해야 할 것:**

| 포함 O | 포함 X |
|--------|--------|
| Claude가 추측할 수 없는 Bash 명령어 | Claude가 코드를 읽으면 알 수 있는 것 |
| 기본값과 다른 코드 스타일 규칙 | Claude가 이미 아는 표준 언어 관례 |
| 테스트 지침 및 선호 테스트 러너 | 자세한 API 문서 (대신 링크 제공) |
| 저장소 에티켓 (브랜치 명명, PR 관례) | 자주 바뀌는 정보 |
| 프로젝트 특유의 아키텍처 결정 | "깨끗한 코드를 써라" 같은 당연한 지침 |

> CLAUDE.md가 너무 길면 중요한 규칙이 묻혀 Claude가 무시합니다. 정기적으로 가지치기하세요.

**임포트 구문 사용:**
```markdown
# CLAUDE.md
프로젝트 개요는 @README.md 참고, 가용 npm 명령어는 @package.json 참고.

# 추가 지침
- Git 워크플로우: @docs/git-instructions.md
```

### 권한 설정

기본적으로 Claude는 파일 수정, 명령어 실행 전에 매번 승인을 요청합니다. 자주 쓰는 안전한 명령어는 허용 목록에 추가하세요.

```json
// .claude/settings.json
{
  "permissions": {
    "allow": ["npm run lint", "git status", "npm test"]
  }
}
```

또는 `/sandbox`로 OS 수준 격리를 활성화하면 Claude가 더 자유롭게 작업할 수 있습니다.

### CLI 도구 활용

`gh` (GitHub CLI), `aws`, `gcloud` 등 CLI 도구를 설치해두면 Claude가 효율적으로 외부 서비스와 상호작용할 수 있습니다.

### MCP 서버 연결

```bash
claude mcp add
```

Notion, Figma, 데이터베이스 등 외부 서비스와 연결합니다.

### 훅(Hook) 설정

훅은 Claude의 특정 동작 시 자동으로 스크립트를 실행합니다. CLAUDE.md 지침과 달리 **반드시 실행됩니다**.

```
파일 편집 후마다 eslint를 실행하는 훅을 작성해줘
```

### 스킬 만들기

반복적인 워크플로우를 스킬로 패키지화하세요.

```markdown
# .claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: GitHub 이슈 수정
disable-model-invocation: true
---
GitHub 이슈를 분석하고 수정: $ARGUMENTS

1. `gh issue view`로 이슈 내용 확인
2. 관련 파일 검색
3. 필요한 변경사항 구현
4. 테스트 작성 및 실행
5. 커밋하고 PR 생성
```

`/fix-issue 1234`로 호출합니다.

## 5. 효과적으로 소통하라

### 코드베이스 질문

마치 시니어 개발자에게 묻듯이 질문하세요.

```
로깅은 어떻게 동작해?
```

```
새 API 엔드포인트를 어떻게 만들어?
```

```
foo.rs 134번째 줄의 async move { ... }가 뭘 하는 거야?
```

```
CustomerOnboardingFlowImpl은 어떤 엣지 케이스를 처리해?
```

### Claude에게 인터뷰를 받아라

큰 기능 개발 전에 Claude가 여러분을 인터뷰하게 하세요.

```
[간단한 기능 설명]을 만들려고 해. AskUserQuestion 도구를 써서 자세하게 인터뷰해줘.

기술 구현, UI/UX, 엣지 케이스, 우려 사항, 트레이드오프에 대해 물어봐.
자명한 건 묻지 말고 내가 생각 못 했을 어려운 부분을 파고들어.

모든 내용을 다루면 완전한 명세서를 SPEC.md에 작성해줘.
```

명세서가 완성되면 **새 세션**에서 구현을 시작하세요. 새 세션은 구현에만 집중된 깨끗한 컨텍스트를 가집니다.

## 6. 세션을 잘 관리하라

### 빠르게 방향 수정하기

- `Esc`: Claude를 즉시 중단 (컨텍스트 유지)
- `Esc + Esc` 또는 `/rewind`: 이전 상태로 되돌리기
- `"이거 취소해줘"`: Claude가 변경사항을 되돌림
- `/clear`: 관련 없는 작업 사이에 컨텍스트 초기화

> 같은 문제로 두 번 이상 수정했다면 `/clear`하고 더 구체적인 프롬프트로 다시 시작하세요.

### 컨텍스트 적극 관리하기

```
/clear       # 관련 없는 작업 사이에 완전히 초기화
/compact     # 중요한 것만 요약해 공간 확보
/compact OAuth 변경사항에 집중해  # 특정 주제로 요약
```

CLAUDE.md에 압축 지침 추가:
```markdown
# 압축 지침
압축 시 항상 수정된 파일 전체 목록과 테스트 명령어를 보존해줘
```

빠른 정보 확인은 `/btw`를 사용하세요 — 컨텍스트를 소비하지 않습니다.

### 서브에이전트로 조사 위임하기

```
서브에이전트를 써서 인증 시스템의 토큰 갱신 방식과
재사용 가능한 OAuth 유틸리티가 있는지 조사해줘
```

서브에이전트는 별도 컨텍스트 창에서 실행되어 조사 과정이 메인 대화에 쌓이지 않습니다.

## 7. 자동화하고 병렬화하라

### 비대화형 모드 (-p 플래그)

```bash
# 일회성 질문
claude -p "이 프로젝트가 뭘 하는지 설명해줘"

# 스크립트용 구조화된 출력
claude -p "API 엔드포인트 목록" --output-format json

# 실시간 스트리밍
claude -p "이 로그 파일을 분석해줘" --output-format stream-json
```

### 여러 Claude 세션 병렬 실행

- **데스크탑 앱**: 여러 로컬 세션 시각적으로 관리
- **웹**: Anthropic의 격리된 VM에서 실행
- **에이전트 팀**: 공유 작업과 메시징으로 여러 세션 자동 조율

**작성자/검토자 패턴:**

| 세션 A (작성자) | 세션 B (검토자) |
|----------------|----------------|
| `API 엔드포인트에 rate limiter를 구현해줘` | |
| | `@src/middleware/rateLimiter.ts를 검토해줘. 엣지 케이스, 경쟁 조건, 기존 미들웨어와의 일관성을 확인해줘` |
| `검토 피드백: [세션 B 출력]. 이 문제들을 해결해줘` | |

### 파일 대량 처리

```bash
# 작업 목록 생성
claude -p "마이그레이션이 필요한 모든 Python 파일을 나열해줘"

# 각 파일에 대해 반복 실행
for file in $(cat files.txt); do
  claude -p "$file를 React에서 Vue로 마이그레이션해줘. OK 또는 FAIL을 반환해줘." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

## 흔한 실패 패턴

| 패턴 | 문제 | 해결책 |
|------|------|--------|
| **주방 싱크 세션** | 관련 없는 작업들을 한 세션에 섞음 → 컨텍스트가 노이즈로 가득 | 관련 없는 작업 사이에 `/clear` |
| **끝없는 수정** | 같은 문제를 계속 수정 → 컨텍스트가 실패한 시도들로 오염 | 두 번 수정 후에도 안 되면 `/clear`하고 더 나은 프롬프트로 재시작 |
| **과도한 CLAUDE.md** | 너무 길어서 중요한 규칙이 묻힘 | 정기적으로 가지치기 |
| **믿고 안 확인** | 그럴듯해 보이지만 엣지 케이스를 처리하지 않는 구현 | 항상 검증 수단 제공 |
| **무한 탐색** | 범위를 정하지 않은 "조사" 요청 → 파일 수백 개 읽어서 컨텍스트 소진 | 조사 범위를 좁게 지정하거나 서브에이전트 사용 |

## 예시 케이스

**케이스 1: 복잡한 기능 개발 — 결제 시스템 추가**

쇼핑몰을 운영하는 수현 씨는 새 결제 방법을 추가해야 합니다.

1. 계획 모드에서 탐색:
```
src/payments/ 폴더를 읽고 현재 결제 흐름을 파악해줘
```

2. 인터뷰 요청:
```
AskUserQuestion 도구를 써서 새 결제 방법 추가에 대해 인터뷰해줘. 엣지 케이스와 보안 문제를 집중적으로 파고들어
```

3. 새 세션에서 명세서 기반으로 구현

**케이스 2: 레거시 코드 마이그레이션 — 2000개 파일**

레거시 시스템을 담당하는 기업 개발팀이 2000개의 Python 파일을 마이그레이션해야 합니다.

```bash
# 먼저 5개 파일로 테스트
for file in $(head -5 files.txt); do
  claude -p "$file를 새 API 패턴으로 마이그레이션해줘" \
    --allowedTools "Edit,Bash(pytest *)"
done
```

5개를 검토해 프롬프트를 개선한 뒤 전체에 실행합니다.

**케이스 3: 신규 입사자 온보딩**

대기업에 입사한 첫날, 영훈 씨:

```
이 코드베이스를 이해하는 데 도움을 줘.
로깅은 어떻게 동작해? 새 API 엔드포인트는 어떻게 만들어?
CustomerOnboardingFlowImpl이 어떤 엣지 케이스를 처리하는지 설명해줘
```

하루만에 코드베이스를 파악하고 첫 PR을 올릴 수 있었습니다.

## 직관 개발하기

이 가이드의 패턴들은 출발점입니다. 상황에 따라 다르게 적용하세요.

- 때로는 컨텍스트를 쌓는 것이 좋습니다 — 복잡한 문제에서 히스토리가 중요할 때
- 때로는 계획을 건너뛰는 것이 좋습니다 — 탐색적인 작업일 때
- 때로는 모호한 프롬프트가 좋습니다 — Claude의 해석이 궁금할 때

좋은 결과가 나올 때 무엇을 했는지 기억하세요. Claude가 헤맬 때는 왜 그런지 분석하세요. 시간이 지나면 가이드로 설명할 수 없는 직관이 생깁니다.
