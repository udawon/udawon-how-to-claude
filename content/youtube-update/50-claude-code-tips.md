## 이게 뭔가요?

6개월간 매일 Claude Code를 사용한 엔지니어가 **실전에서 깨달은 50가지 팁**을 정리한 종합 가이드입니다. 기초 설정부터 멀티 인스턴스 병렬 개발까지, "처음부터 알았으면 좋았을 것들"을 모았습니다.

> 비유: 게임 공략집 같은 거예요. 혼자 플레이하면 100시간 걸려 발견할 팁을, 공략집 보면 10분 만에 알 수 있잖아요. 이 영상이 Claude Code 공략집입니다.

## 왜 알아야 하나요?

- **시행착오 단축**: 남이 6개월 걸려 배운 걸 한번에 습득
- **토큰(AI가 처리하는 텍스트 단위) 비용 절약**: 컨텍스트 관리법을 알면 불필요한 토큰 낭비 방지
- **생산성 극대화**: 병렬 개발, Plan 모드 우선 등 검증된 워크플로우 적용
- **자가 개선 루프**: 검증(Validation) 루프를 설계하면 AI가 스스로 버그를 고침

## 어떻게 하나요?

### 기초 설정 (Tip 1~2)

**루트 디렉토리에서 실행하기**

Claude Code는 실행한 디렉토리를 기준으로 프로젝트를 파악합니다. 반드시 프로젝트의 **최상위 폴더**에서 `claude`를 실행하세요.

```bash
# O — 프로젝트 루트에서 실행
cd my-project
claude

# X — 하위 폴더에서 실행하면 전체 구조를 못 봄
cd my-project/src/components
claude
```

> "처음 시작할 때 토큰이 많이 쓰이는 이유" = Claude가 프로젝트를 분석하고 CLAUDE.md를 읽는 과정 때문

**`/init`으로 CLAUDE.md 자동 생성**

```
/init
```

Claude가 코드베이스를 분석하고 `.claude/CLAUDE.md`를 자동 생성합니다. 새 프로젝트에서 가장 먼저 할 일.

<div class="example-case">
<strong>💬 예시: /init 결과</strong>

```
/init 실행 → Claude가 코드 분석 중...
→ "이 프로젝트는 Next.js 15 앱이고, TypeScript를 사용하고..."
→ .claude/CLAUDE.md 파일 자동 생성
→ "Yes" 입력 → 커밋 완료
```

</div>

### CLAUDE.md 심화 (Tip 3~6)

**우선순위는 위에서 아래로**

CLAUDE.md는 **위에 적힌 규칙이 더 높은 우선순위**를 가집니다. 가장 중요한 규칙을 맨 위에 적으세요.

**적정 크기: ~300줄**

너무 길면 토큰 소비가 늘고, AI의 집중도가 떨어집니다.

**Never/Always 규칙 + 코드 예시**

```markdown
# CLAUDE.md 예시

## 절대 하지 마 (Never)
- transition-all 사용 금지
- console.log를 프로덕션 코드에 남기지 말 것

## 항상 해 (Always)
- 파일 수정 전 반드시 먼저 읽을 것
- 에러 핸들링에 구체적인 메시지 포함

## 빌드/검증 루프
\`\`\`bash
npm run build && npm run test
\`\`\`
```

> 핵심 인사이트: **검증 루프(Validation Loop)가 AI 코딩 품질의 가장 중요한 요소**. CLAUDE.md에 빌드/테스트 커맨드를 포함하면, Claude가 코드 작성 → 빌드 → 에러 수정 → 다시 빌드를 **자동으로 반복**합니다.

<div class="example-case">
<strong>💬 예시: 검증 루프의 위력</strong>

```
[검증 루프 없이]
Claude가 코드 작성 → 완료 보고 → 실제로 빌드하면 에러 → 다시 요청

[검증 루프 있으면]
Claude가 코드 작성 → 자동으로 빌드 실행 → 에러 발견 → 스스로 수정 → 다시 빌드 → 성공!
→ 한 번의 지시로 완성까지 자동 진행
```

</div>

**Compound Engineering: CLAUDE.md를 커밋하기**

CLAUDE.md를 Git(코드 버전 관리 도구)에 커밋하면 **팀 전체의 AI 코딩 경험이 향상**됩니다. 단, 파일 경로 같은 개인 설정은 제거하고, 정말 도움이 되는 규칙만 넣으세요.

**규칙 업데이트는 Claude에게 시키기**

```
"CLAUDE.md에 '테스트 파일은 항상 __tests__ 폴더에 만들 것' 규칙을 추가해줘"
```

수동으로 편집하지 말고, Claude에게 지시하면 형식에 맞게 자동으로 추가합니다.

### 키보드 단축키 (Tip 7~10)

아래 단축키는 Claude Code 자체 단축키이므로 **Mac과 Windows 모두 동일**하게 작동합니다.

| 단축키 | 기능 | 언제 쓰나 |
|--------|------|-----------|
| **Shift+Tab** | Plan ↔ Accept 모드 전환 | 새 기능 시작 시 Plan 모드로 전환 |
| **ESC** (1번) | 진행 중단(인터럽트) | AI가 방향을 잘못 잡았을 때 |
| **ESC** (2번, 입력 있을 때) | 입력창 전체 삭제 | 긴 텍스트를 잘못 붙여넣었을 때 |
| **ESC** (2번, 입력 없을 때) | 이전 컨텍스트로 되감기 | 이전 상태로 복원하고 싶을 때 |

> 핵심 팁: **인터럽트(ESC)를 주저하지 마세요.** AI가 잘못된 방향으로 가고 있으면 바로 중단하고 방향을 수정하는 게 토큰도 아끼고 결과도 좋습니다. Claude Code는 인터럽트 후에도 맥락을 잘 유지하므로, 위 화살표(↑)를 눌러 이전 입력을 복원하거나 새로운 방향을 바로 지시할 수 있습니다.

### 필수 슬래시 명령어 (Tip 11~16)

| 명령어 | 기능 | 핵심 팁 |
|--------|------|---------|
| `/clear` | 컨텍스트 초기화 | 새 작업 시작 시 이전 컨텍스트 오염 방지 |
| `/context` | 현재 컨텍스트 시각화 | MCP가 토큰을 얼마나 먹는지 감사(audit) |
| `/models` | AI 모델 변경 | 여유 있으면 Opus 추천 |
| `/resume` | 이전 세션 복구 | 실수로 인스턴스를 닫았을 때 |
| `/mcp` | 설치된 MCP(외부 도구 연결 기능) 확인 | MCP는 토큰 폭탄 — 꼭 필요한 것만 |
| `/help` | 전체 도움말 | 모든 명령어와 단축키 확인 |

<div class="example-case">
<strong>💬 예시: /context로 토큰 사용량 감사</strong>

```
/context 실행
→ 현재 컨텍스트에 어떤 파일과 MCP가 로드되어 있는지 시각화
→ "이 MCP가 토큰의 40%를 차지하네?" → 해당 MCP 비활성화
→ 토큰 비용 절감!
```

비유: 월간 지출 내역을 확인하는 것과 같아요. "이 구독료가 이렇게 비쌌어?" 하고 발견해서 정리하는 거죠.

### 워크플로우 전략 (Tip 17~22)

**Plan 모드 우선 원칙**

새 기능을 시작할 때 **절대로 바로 코드 작성 모드로 들어가지 마세요**. `Shift+Tab`으로 Plan 모드를 켜고, 방향을 충분히 검토한 후에 실행합니다. (Mac/Windows 모두 동일)

```
1. Shift+Tab → Plan 모드 진입 (Mac/Windows 동일)
2. "이 기능을 어떻게 구현할지 계획 세워줘"
3. 계획 검토 → 수정 요청 → 합의
4. Shift+Tab → Accept 모드로 전환
5. 실행 시작
```

> "코드 생성은 쉬운 부분입니다. 어려운 건 좋은 컨텍스트를 만드는 겁니다."

**Fresh Context > Bloated Context**

```
[나쁜 패턴]
"이거 해봐" → "아니 그거 말고" → "다시 이렇게" → "저번 거 되돌려"
→ 컨텍스트가 모순된 지시로 가득 → AI가 혼란

[좋은 패턴]
Plan 모드에서 방향 확정 → /clear → 깨끗한 상태에서 실행
→ 명확한 컨텍스트 → 정확한 결과
```

비유: 깨끗한 도화지에 그림 그리기 vs 이미 낙서가 가득한 종이에 그리기

**세컨드 브레인 (Second Brain)**

작업이 끝나면 **핵심 컨텍스트를 로컬에 저장**해두고, 다음 세션에서 필요할 때만 불러오는 방식입니다.

<div class="example-case">
<strong>💬 예시: 세컨드 브레인 워크플로우</strong>

```
[작업 완료 후 — 저장]
"지금까지 한 작업을 로컬 .claude/ 디렉토리에 정리해서 저장해줘"
→ 핵심 결정사항, TODO, 아키텍처 노트가 저장됨

[다음 세션 — 불러오기]
"로컬 프로젝트 디렉토리에서 이전 컨텍스트를 불러와줘"
→ 처음부터 설명할 필요 없이 이전 작업 이어서 진행

[핵심: Lazy Loading]
모든 컨텍스트를 항상 로드하지 않고, 필요할 때만 불러옴
→ 컨텍스트를 신선하고 가볍게 유지
```

비유: 매일 출근할 때 업무 일지를 처음부터 다시 쓰는 게 아니라, 어제 퇴근 전에 메모해둔 걸 보고 바로 이어서 시작하는 거예요.

</div>

### 파워유저 기능 (Tip 23~30)

**워크플로우 → 스킬로 저장**

반복하는 작업이 있으면, 한 번 수행한 후 바로 스킬로 저장합니다.

```
1. 원하는 작업을 Claude에게 시킴 (예: "해커뉴스에서 iOS 뉴스를 찾아서 정리해줘")
2. 작업 완료 후: "지금 한 작업을 'fetch-hackernews'라는 스킬로 저장해줘"
3. 다음부터: /fetch-hackernews 로 한 줄 실행
```

> 최근 변경: **슬래시 커맨드와 스킬이 통합**되었습니다. 스킬을 만들면 자동으로 슬래시 커맨드로도 사용 가능.

**Sub Agent 올바른 사용법**

| 좋은 사용법 | 나쁜 사용법 |
|-------------|-------------|
| 독립적인 사이드 작업 | 메인 컨텍스트가 필요한 작업 |
| 병렬 실행이 필요한 작업 | CEO/PM/디자인 에이전트 역할 분리 |
| 컨텍스트 보호가 필요한 작업 | 조사 → 결과만 가져오기 |

> 핵심 원칙: **"작업을 컨텍스트로 가져오기"** — Sub Agent가 돌아올 때는 결과만 가져오므로, 과정이 중요한 작업은 메인 세션에서 유지하세요.

<div class="example-case">
<strong>💬 예시: Sub Agent 사용 판단</strong>

```
[O — Sub Agent 적합]
"코드 스타일 검사를 백그라운드에서 돌려줘"
→ 독립적, atomic한 작업

[X — 메인 세션에서 해야 함]
"테스트 에이전트가 코드를 검증하고 버그를 찾아줘"
→ 테스트하려면 코드 작성 컨텍스트가 필요. Sub Agent는 그 맥락을 못 봄
```

</div>

**MCP 사용 주의사항**

- MCP는 **컨텍스트 윈도우를 크게 소비**합니다
- 해당 프로젝트에 꼭 필요한 MCP만 설치
- `/context`로 주기적으로 토큰 사용량 확인
- MCP 설치도 Claude에게 요청 가능: "Figma MCP를 찾아서 설치해줘"

### 고급 워크플로우 (Tip 31~50)

**멀티 인스턴스 병렬 개발**

Claude Code의 진짜 강점은 **여러 인스턴스를 동시에 실행**할 수 있다는 것입니다.

```
[터미널 탭 1] claude → "로그인 기능 구현해줘"     → 작업 중...
[터미널 탭 2] claude → "API 문서 정리해줘"        → 작업 중...
[터미널 탭 3] claude → "테스트 코드 작성해줘"     → 작업 중...

→ 탭 전환하면서 3개 작업을 동시 진행
→ "스타크래프트 유닛 컨트롤하는 느낌"
```

**Mac과 Windows에서 멀티 인스턴스 사용 방법:**

| 작업 | Mac (iTerm) | Windows (Windows Terminal) |
|------|-------------|---------------------------|
| **새 탭 생성** | `Cmd+T` | `Ctrl+Shift+T` |
| **패널 분할** | `Cmd+D` | `Alt+Shift+D` |
| **탭 전환** | `Cmd+[` / `Cmd+]` | `Ctrl+Tab` / `Ctrl+Shift+Tab` |
| **패널 전환** | `Cmd+Option+←/→` | `Alt+←/→/↑/↓` |
| **탭 이름 변경** | 탭 우클릭 → Rename | 탭 우클릭 → Rename Tab |

> VS Code 터미널을 사용하는 경우: `` Ctrl+Shift+` `` (Mac/Windows 공통)으로 새 터미널을 열고, `Ctrl+PageUp` / `Ctrl+PageDown`으로 터미널 간 전환할 수 있습니다.

<div class="example-case">
<strong>💬 예시: 멀티 인스턴스 워크플로우</strong>

**Mac (iTerm 기준):**
```
1. iTerm에서 Cmd+D로 패널 분할
2. 각 패널에서 claude 실행
3. Cmd+[ / Cmd+] 로 패널 전환
4. 탭 이름을 "login-feature", "api-docs" 등으로 변경
```

**Windows (Windows Terminal 기준):**
```
1. Windows Terminal에서 Ctrl+Shift+T로 새 탭 생성 (또는 Alt+Shift+D로 패널 분할)
2. 각 탭/패널에서 claude 실행
3. Ctrl+Tab / Ctrl+Shift+Tab으로 탭 전환 (또는 Alt+←/→ 로 패널 전환)
4. 탭 우클릭 → "Rename Tab"으로 이름 변경
```

**공통 팁:**
- 한 탭에서 지시 내리기 → 다른 탭으로 전환 → 지시 내리기
- 완료 알림이 오면 해당 탭으로 가서 결과 확인
- 병목: "내 머리의 컨텍스트 스위칭 능력"

비유: 요리사가 여러 버너에 냄비를 올려놓고, 이쪽 저쪽 왔다 갔다 하면서 동시에 여러 요리를 만드는 거예요.

</div>

**Git Worktree로 충돌 방지**

같은 프로젝트에서 여러 인스턴스가 코드를 수정하면 충돌이 생깁니다. **Git Worktree**를 사용하면 각 인스턴스가 독립적인 작업 브랜치에서 작업합니다.

```bash
# 새 워크트리 생성
git worktree add ../my-project-feature-a feature-a
git worktree add ../my-project-feature-b feature-b

# 각 워크트리에서 별도 Claude 인스턴스 실행
cd ../my-project-feature-a && claude
cd ../my-project-feature-b && claude
```

**알림 설정**

```
"작업이 끝나면 알림 소리를 울려줘"
```

여러 인스턴스를 돌릴 때 완료 알림을 설정하면, 다른 작업을 하다가도 돌아갈 타이밍을 알 수 있습니다.

**/chrome으로 브라우저 제어**

```
/chrome → 브라우저 열기 → "YouTube를 열어서 cooking을 검색해줘"
→ Claude가 스크린샷 찍고, 클릭하고, 입력하고, 탐색
```

API가 없는 서비스도 브라우저를 통해 접근할 수 있어서, 스킬이나 워크플로우와 조합하면 강력합니다.

<strong>📌 실전 케이스: 하루 종일 Claude Code로 일하는 방법</strong>

**상황**: 여러 기능을 동시에 개발해야 하는 날

**워크플로우**:
1. **아침**: 터미널 열고 탭 3개 생성 (Mac: iTerm `Cmd+T` / Windows: Windows Terminal `Ctrl+Shift+T`). 각각 다른 기능 할당
2. **탭 1**: Plan 모드(`Shift+Tab`)로 기능 A 설계 시작 → 합의 후 실행 모드로 전환
3. **탭 2**: 기능 B 관련 파일 분석 지시 (Plan 모드)
4. **탭 3**: 어제 하던 작업 `/resume`으로 복구
5. **오전 중**: 탭 전환(Mac: `Cmd+[/]` / Windows: `Ctrl+Tab`)하며 지시 → 확인 → 지시 반복
6. **점심 전**: `/context`로 각 탭의 토큰 사용량 확인. 비대해진 탭은 `/clear` 후 재시작
7. **오후**: 결과물 리뷰. 세컨드 브레인에 오늘 작업 내용 저장
8. **퇴근 전**: Git 커밋 + 내일 TODO 정리

**결과**: 혼자서 3명분의 개발 작업을 동시 진행

</div>

## 주의할 점

- **`--dangerously-skip-permissions`는 신중하게**: 모든 확인을 건너뛰므로, 안전한 환경에서만 사용. OS 수준 작업은 절대 금지 (리눅스 반쯤 날린 실제 사례 있음)
- **MCP는 토큰 폭탄**: 꼭 필요한 것만 설치하고, `/context`로 주기적 감사
- **Sub Agent를 남용하지 말 것**: 컨텍스트가 필요한 작업을 Sub Agent로 보내면 품질 하락
- **CLAUDE.md 커밋 시 주의**: 개인 경로나 민감 정보 제거 후 커밋. 팀원 컨텍스트에 불필요한 부담 주지 않기
- **컨텍스트는 신선하게**: 오래된 컨텍스트보다 새로운 컨텍스트가 항상 낫습니다

## 정리

- **컨텍스트가 왕**: 신선하고 관련성 있는 컨텍스트만 유지하면 AI 성능이 극대화
- **Plan 먼저, 실행 나중에**: Shift+Tab으로 Plan 모드 → 충분한 검토 → 실행
- **검증 루프가 핵심**: CLAUDE.md에 빌드/테스트 명령을 넣으면 AI가 자가 개선
- **병렬 개발로 생산성 극대화**: 멀티 인스턴스 + Git Worktree = 혼자서 팀 수준의 아웃풋
