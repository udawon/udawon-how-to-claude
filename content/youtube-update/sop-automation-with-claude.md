## 이게 뭔가요?

프로젝트를 진행할 때 **기획서, SOP(표준 운영 절차), 스펙 문서**를 작성하잖아요. 문제는 이 문서를 개발팀에 전달하면 **해석이 달라지거나 빠지는 부분**이 생긴다는 거예요.

Claude Code로 스펙 문서를 관리하면, **문서 자체가 곧 구현 지시서**가 됩니다.

> 비유: 건축 설계도를 건설 현장에 가져가면, 설계도를 보고 사람이 해석해서 시공하잖아요. 근데 만약 설계도를 넣으면 **로봇이 알아서 시공**해준다면? 해석 오류가 사라지는 거예요.

## 왜 알아야 하나요?

- **기획서 → 코드 변환** 과정에서 발생하는 커뮤니케이션 비용 제거
- **스펙 변경 시 코드 자동 업데이트**: 문서만 수정하면 Claude가 코드에 반영
- **일관성 유지**: 여러 사람이 작업해도 하나의 스펙 문서가 기준점
- **비개발자도 구현 관리 가능**: 기획자가 스펙 문서를 관리하면 Claude가 구현

## 어떻게 하나요?

### 방법 1: 스펙 문서를 CLAUDE.md에 연결

프로젝트의 스펙 문서를 CLAUDE.md에서 `@import`로 참조합니다:

```markdown
# CLAUDE.md
기능 스펙: @docs/feature-spec.md
API 설계: @docs/api-design.md
UI 가이드: @docs/ui-guidelines.md
```

Claude Code가 세션을 시작하면 이 문서들을 자동으로 읽고, 구현 시 참고합니다.

<div class="example-case">
<strong>💬 예시: 기능 스펙으로 구현 지시</strong>

```
# docs/feature-spec.md

## 로그인 기능
- 이메일 + 비밀번호 로그인
- 소셜 로그인 (Google, GitHub)
- 비밀번호 찾기 (이메일 발송)
- 로그인 실패 시 3회까지 재시도, 이후 5분 잠금

## 회원가입
- 이메일 중복 체크
- 비밀번호 규칙: 8자 이상, 대소문자+숫자+특수문자
- 이메일 인증 필수
```

→ Claude Code에게 "로그인 기능을 feature-spec.md대로 구현해줘" 한마디면 됩니다.

</div>

### 방법 2: 스펙 문서를 Skills로 관리

반복적으로 사용하는 스펙 문서 패턴을 Skills로 만들어두면 더 효율적입니다:

```
.claude/skills/
└── implement-spec/
    └── SKILL.md
```

```markdown
# 스펙 기반 구현

## 트리거
- `/implement` 입력 시 실행

## 절차
1. docs/ 폴더에서 관련 스펙 문서를 읽는다
2. 현재 코드와 스펙을 비교한다
3. 스펙에 맞게 누락된 부분을 구현한다
4. 구현 후 스펙 체크리스트로 검증한다
```

<div class="example-case">
<strong>💬 예시: 스펙 변경 → 자동 반영</strong>

```
1. 기획자가 docs/feature-spec.md에서 "로그인 실패 시 3회"를 "5회"로 수정
2. Claude Code에게: "feature-spec.md가 변경됐어. 코드에 반영해줘"
3. Claude가 변경 사항을 감지하고 관련 코드를 자동 수정
4. 테스트까지 실행해서 확인
```

</div>

## 실전 예시

<div class="example-case">
<strong>📌 실전 케이스: 기획자 혼자서 프로토타입 관리</strong>

**상황**: 스타트업 기획자가 개발자 없이 MVP(최소기능제품)를 만들고 싶음

**워크플로우**:
1. **스펙 작성**: 기능 목록, UI 흐름, 비즈니스 규칙을 마크다운으로 정리
2. **CLAUDE.md 연결**: `@docs/spec.md`로 참조
3. **구현 지시**: "이 스펙대로 로그인 기능 만들어줘"
4. **스펙 수정 시**: 문서만 고치고 "변경사항 반영해줘" → 코드 자동 업데이트

**결과**: 기획서 = 코드의 원천. 문서와 코드가 항상 동기화됨

</div>

<div class="example-case">
<strong>📌 실전 케이스: SOP 기반 업무 자동화</strong>

**상황**: 매월 반복하는 리포트 생성 SOP가 있음

**SOP 문서**:
```
1. 이번 달 매출 데이터 수집 (Google Sheets)
2. 카테고리별 집계
3. 전월 대비 증감률 계산
4. 경영진 보고용 요약 3줄 작성
5. 슬랙 #report 채널에 공유
```

→ 이 SOP를 Skills로 만들면, `/monthly-report` 한 줄로 전체 프로세스 실행

</div>

## 주의할 점

- **스펙 문서의 품질이 핵심**: "대충 써도 AI가 알아서 해주겠지"는 위험. 명확하고 구체적인 스펙이 좋은 결과를 만듦
- **스펙과 코드 동기화 주기적 확인**: 스펙을 안 고치고 코드만 바꾸면 불일치 발생
- **민감한 비즈니스 로직 주의**: 스펙 문서에 영업 비밀이 포함되면 `.gitignore`에 추가하거나 별도 관리
- **CLAUDE.md 길이 관리**: 스펙이 너무 길면 `@import`로 분리해서 CLAUDE.md는 30줄 이내로 유지

## 정리

- 스펙 문서를 CLAUDE.md에 연결하면 "문서 = 구현 지시서"가 됨
- 문서만 수정하면 Claude Code가 코드에 자동 반영 가능
- 비개발자도 스펙 문서 관리만으로 프로토타입을 만들고 유지할 수 있음
