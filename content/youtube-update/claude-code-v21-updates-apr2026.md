## 이게 뭔가요?

Claude Code(키보드로 명령어를 입력하는 화면에서 Claude와 함께 작업하는 공식 개발 도구) v2.1.105부터 v2.1.114까지 10개 버전이 **2026년 4월 14일~18일 사이**에 연달아 출시되었습니다. 핵심 변화는 5가지입니다.

1. **1시간 프롬프트 캐시**(프롬프트 - AI에게 보내는 요청 메시지) — 반복 요청 비용 절감
2. **Opus 4.7 xhigh effort** — Opus 4.7 전용 새 사고 강도 레벨
3. **Control+O 동작 재설계** + `/focus` 분리
4. **네이티브 바이너리 전환**(OS에 맞는 실행 파일로 바뀜) — 기동 속도·예측 가능성 향상
5. **보안 강화** — 위험한 삭제 경로 차단, bash deny 규칙 확장, 샌드박스(보호된 실행 환경) 비활성화 차단 등

업데이트는 일본 현지 시간 기준이며, 한국 시간과 동일합니다.

## 왜 알아야 하나요?

비개발자라도 Claude Code를 Claude Design 핸드오프·자동화 스크립트·스킬 개발에 쓰는 경우가 늘고 있습니다. 이 구간 업데이트는 다음 이유로 놓치기 아깝습니다.

- **비용**: 1시간 캐시 한 줄 설정으로 같은 대화 재사용 비용이 크게 준다
- **품질**: Opus 4.7 xhigh는 "중요하고 깊게 생각해야 하는 작업"에 기본값 이상을 쓰게 해 준다
- **보안**: 모르고 지나가면 위험한 삭제 명령이 그대로 실행되는 경우가 있었는데, 이번 구간에서 그 구멍이 여러 개 막혔다
- **호환성**: Control+O 동작이 바뀌었기 때문에 이전 방식대로 누르면 당황할 수 있다

### 2. 1시간 프롬프트 캐시 (v2.1.108)

프롬프트 캐시(Prompt Cache)란 이전 요청의 문맥을 저장해 두었다가 같은 내용을 다시 보내면 AI가 재계산하지 않고 재활용하는 기능입니다. 같은 수업 자료로 여러 번 질문할 때 선생님이 매번 처음부터 다시 준비하지 않고 "아까 그 자료"를 그대로 쓰는 느낌입니다.

**기존**: 5분 TTL(Time To Live - 저장 유지 시간) → 잠시 자리 비우면 만료
**v2.1.108부터**: 환경 변수(시스템이 프로그램에 전달하는 설정값) `ENABLE_PROMPT_CACHING_1H=1` 하나로 **1시간 TTL** 활성화

설정 방법:

```bash
# Mac/Linux 터미널
export ENABLE_PROMPT_CACHING_1H=1

# Windows PowerShell
$env:ENABLE_PROMPT_CACHING_1H = "1"
```

- Anthropic 직접 호출·Bedrock·Vertex 모든 프로바이더(서비스 제공자)에서 동작
- Bedrock 전용 구형 환경 변수는 비권장이지만 계속 존중됨(이전 스크립트가 깨지지 않는다는 뜻)

주의: v2.1.108 직후 "Disable Low-tier 설정 사용자가 1시간이 아닌 5분 캐시로 돌아가던" 버그가 있었으며 같은 버전에서 수정되었습니다.

### 3. Recap 기능 (v2.1.108)

세션에서 오래 떨어졌다가 돌아왔을 때 지금까지의 맥락을 요약해 주는 기능입니다.

- 자동 활성화: `/config`에서 toggle
- 수동 호출: `/recap`
- 텔레메트리(사용 로그 수집)를 끈 환경에서도 강제 활성화하려면 `CLAUDE_CODE_ENABLE_AWAY_SUMMARY=1`

### 4. Skill 안에서 내장 슬래시 커맨드 호출 (v2.1.108)

Skill(Claude Code의 재사용 가능한 작업 단위) 2.0부터 `/init`·`/review`·`/security-review` 같은 내장 명령을 Skill 내부에서 호출할 수 있게 되었습니다. Skill이 "필요하면 먼저 리뷰 돌려라"처럼 자율 판단하는 구성이 가능해진 셈입니다.

이 업데이트는 기존 사용자가 **동작이 바뀌어서 당황할 수 있는** 변화라 특별히 언급합니다.

### 6. Opus 4.7 xhigh effort (v2.1.111)

Claude Opus 4.7은 **2026년 4월 16일 공개된 새 모델**(코딩·에이전트·비전·멀티스텝 태스크 강화, 중요 작업의 사고 깊이와 일관성 향상). Claude Code에는 다음 날인 v2.1.111에서 통합되었습니다.

**effort level(사고 강도)** 이란 모델이 한 번 답하기까지 얼마나 깊게 생각하는지의 단계입니다. 의사 처방에 비유하면 "증상에 따라 약 용량을 조절"하는 느낌입니다.

- `xhigh` = Opus 4.7 **전용**. `high`와 `max`의 중간 위치(속도↔지능 균형).
- 다른 모델에서 `xhigh`를 고르면 `high`로 자동 대체(fallback).

사용 방법:

```bash
# 슬래시 커맨드로 직접 지정
/effort xhigh

# 플래그로 지정
claude --effort xhigh

# 모델 피커에서 선택
# UI 왼쪽 아래 모델명 클릭 → effort 슬라이더
```

인자 없이 `/effort`를 실행하면 **대화형 슬라이더**가 열려 화살표+엔터로 고를 수 있습니다.

**Auto mode** 는 `ENABLE_AUTO_MODE` 플래그가 이제 필요 없어졌고, Max 구독자는 Opus 4.7에서 Auto mode를 바로 쓸 수 있습니다.

### 7. `/ultrareview` (v2.1.111)

"클라우드에서 병렬 멀티 에이전트로 종합 코드 리뷰를 돌리는" 신규 명령입니다.

- 인자 없이 실행 → 현재 브랜치를 리뷰
- 브랜치·PR 번호 전달 → 해당 대상을 리뷰
- 여러 에이전트가 동시에 보는 구조이므로 일반 리뷰보다 커버리지가 넓다는 점이 차별점. 단일 에이전트 리뷰로 충분한 소형 변경에는 과할 수 있습니다.

v2.1.113에서 `/ultrareview`는 병렬 기동이 더 빨라졌고 런치 시점에 git stat(변경 파일 통계) 표시가 추가됐습니다.

### 8. `/less-permission-prompts` 스킬 (v2.1.111)

Claude Code가 명령어를 실행할 때마다 "이거 해도 돼?"라고 물어보는 권한 프롬프트가 너무 많다고 느끼는 사람을 위한 스킬입니다. 호출 명령은 `/less-permission-prompts`이며, 화면 표시 이름은 "Fewer Permission Prompts"입니다.

동작:

1. 과거 대화 내용(transcripts)을 스캔
2. 자주 등장한 **읽기 전용** Bash 호출·MCP(외부 도구 연결 기능) 호출을 추출
3. 프로젝트 설정 파일(`.claude/settings.json`)에 우선순위 순 허용 리스트 제안

글롭 패턴(glob - 파일 경로 와일드카드) 기반 자동 승인도 확장됐습니다. `ls *.ts`처럼 **읽기 전용** 명령과 `cd project/ && echo ...`처럼 안전한 조합은 권한 프롬프트 없이 실행됩니다.

### 9. 네이티브 바이너리 전환 (v2.1.113)

이번 구간에서 가장 큰 내부 변경입니다. 지금까지 Claude Code는 **번들된 JavaScript를 Node(자바스크립트 실행 환경)로 돌리는** 구조였습니다. v2.1.113부터 **CLI가 OS에 맞는 네이티브 실행 파일**을 기동합니다.

- 배포는 Optional Dependency(선택적 의존성)로 이뤄지며, 설치 시점에 OS에 맞는 바이너리가 선택됨
- 기동 비용 감소, 실행 예측 가능성 향상

비개발자가 체감하는 포인트는 **첫 실행이 약간 빨라진다**는 것. 구조 변경이 크기 때문에 문제가 생기면 구버전 롤백이 어렵지 않도록 공식 채널 공지를 보고 업데이트 타이밍을 잡는 편이 안전합니다.

### 10. 보안 강화 묶음 (v2.1.113)

영상이 "여러 방향의 구멍을 막았다"고 표현한 변화들입니다.

| 항목 | 이전 | 이후 |
|---|---|---|
| `rm *` 허용 규칙 | `/private/var` 등 macOS 민감 경로까지 자동 허용 가능 | 위험 삭제 타깃으로 분류·차단 |
| `bash deny` 규칙 | 직접 명령어만 매칭 | `env`·`script` 같은 **엑시큐트 래퍼**(다른 명령을 감싸 실행하는 도구) 경유 명령도 매칭 |
| `find *` 허용 규칙 | `find ... -exec rm` 자동 승인 | `find -exec delete` 자동 승인 안 함 |
| 멀티라인 명령의 첫 줄 | 첫 줄이 주석이면 전체가 트랜스크립트에 숨어 UI 우회 가능 | 전체 표시 + UI 우회 벡터 차단 |
| `--dangerously-disable-sandbox` 프롬프트 없이 실행 | 가능 | 차단(반드시 사용자 확인) |

### 11. 권한 요청 훅(Hook)의 입력이 재검증 (v2.1.110)

Hook(특정 시점에 자동 실행되는 기능)이 권한 요청을 조작해 우회하려는 벡터가 있었습니다. v2.1.110부터 Permission Request Hook에서 반환되는 input이 `permission.deny` 규칙과 다시 대조됩니다. 권한 우회 한 갈래가 막힌 셈입니다.

## 주의할 점

**업데이트 시점에 주의**
- 4일 동안 10개 버전이 나왔고 핫픽스(긴급 수정)가 두 번 있었습니다(v2.1.112, v2.1.114). 제일 안전한 선택은 **v2.1.114 이상**으로 바로 올라가는 것입니다. 중간 버전(v2.1.111·v2.1.112)에 머물면 Auto mode 표시 이슈·권한 다이얼로그 크래시를 만날 수 있습니다.

**캐시가 무조건 싸지는 것은 아니다**
- 1시간 캐시는 "같은 맥락을 재사용할 때"만 절감 효과가 있습니다. 매번 다른 문서를 던지는 짧은 일회성 작업에는 오히려 캐시 쓰기 오버헤드만 붙습니다. 작업 성격에 맞는지 1~2주 관찰 후 고정 여부 결정을 권장합니다.

**보안 강화가 기존 스크립트를 깨뜨릴 수 있다**
- `find -exec delete`를 자동 승인에 의존해 왔다면 v2.1.113부터는 수동 확인이 뜹니다. 자동화 파이프라인에 `find *` 패턴을 무심코 넣어 둔 경우 실행이 멈출 수 있으니, 허용 규칙을 다시 점검해야 합니다.

**네이티브 바이너리는 되돌리기 어렵다**
- 구조 변경이 커서 설치 환경에 따라 예기치 못한 이슈가 나올 수 있습니다. 회사 CI(자동 빌드 환경)나 공유 머신에 배포 중이라면 소규모 머신에서 먼저 검증한 뒤 롤아웃하는 편이 안전합니다.

**xhigh는 Opus 4.7 전용**
- Sonnet·Haiku에서 `/effort xhigh`를 쳐도 자동으로 `high`로 떨어집니다. "왜 내 Sonnet이 xhigh로 안 도는가"를 의심하기 전에 모델 확인부터 합니다.

## 정리

- 4월 14~18일 Claude Code는 v2.1.105~v2.1.114까지 10판 출시됐고, 핵심은 1시간 프롬프트 캐시·Opus 4.7 xhigh·네이티브 바이너리 전환·보안 강화다.
- 비개발자가 가장 먼저 적용할 가치가 큰 것은 `ENABLE_PROMPT_CACHING_1H=1` 한 줄. 비용이 바로 줄어든다.
- Control+O 동작 변경과 네이티브 바이너리 전환은 사전 인지 없이 업데이트하면 당황할 수 있으니, v2.1.114 이상으로 한 번에 올라가고 `/focus`·`/tui fullscreen` 같은 새 명령을 기억해 둔다.
