## 이게 뭔가요?

gstack은 Y Combinator(세계 최대 스타트업 엑셀러레이터) CEO인 Garry Tan이 만든 오픈소스 프레임워크(뼈대 도구)입니다. Claude Code에 **28가지 스킬(슬래시 커맨드)**을 설치하면, AI가 단순히 코드만 짜는 게 아니라 CEO 의사결정, 디자인 리뷰, QA(품질 검증) 테스트, 보안 점검, 배포까지 전체 업무 사이클을 자동으로 처리합니다.

쉽게 말하면, Claude Code에 **"회사 운영 매뉴얼"**을 통째로 넣어주는 것입니다. 비서에게 업무 매뉴얼을 주면 알아서 일하듯이, gstack의 스킬들이 Claude Code에게 "이런 상황에선 이렇게 해"라는 지침을 제공합니다.

출시 48시간 만에 GitHub 스타 10,000개를 돌파했고, 현재 **43,000+ 스타, 5,500+ Fork**를 기록하며 큰 주목을 받고 있습니다.

## 왜 알아야 하나요?

- **혼자서도 팀 규모의 업무를 처리할 수 있습니다** — CEO 리뷰, 디자인 검토, QA 테스트, 보안 감사를 AI가 대신하니까요
- **Garry Tan의 실전 경험이 녹아 있습니다** — 60일간 60만 줄 이상의 코드를 작성하고, 하루 1~2만 줄을 커밋한 실전 사용자의 노하우입니다 (AI 생성 코드 포함, 35%는 테스트 코드)
- **스킬 시스템의 활용 범위를 넓혀줍니다** — 코딩 도우미를 넘어 비즈니스 운영 도구로 Claude Code를 확장하는 구체적인 방법을 보여줍니다

## 어떻게 하나요?

### 방법 1: gstack 설치하기

터미널(명령어 입력 화면)에서 아래 3단계를 실행합니다.

1. gstack 저장소를 내 컴퓨터에 다운로드합니다:
```bash
git clone https://github.com/garrytan/gstack.git ~/.claude/skills/gstack
```

2. 다운로드한 폴더로 이동합니다:
```bash
cd ~/.claude/skills/gstack
```

3. 설치 스크립트를 실행합니다:
```bash
./setup
```

### 방법 2: 팀에서 공유하기

프로젝트의 `.claude/skills/` 폴더에 gstack을 넣으면, 같은 저장소를 쓰는 팀원 모두 동일한 스킬을 사용할 수 있습니다.

## gstack의 28가지 스킬

gstack은 7가지 영역에 걸쳐 28개의 스킬을 제공합니다.

### 기획 & 전략 (4개)

| 커맨드 | 역할 | 설명 |
|--------|------|------|
| `/office-hours` | 제품 사고 코치 | YC 스타일로 요청을 재구성하고 전제를 도전합니다 |
| `/plan-ceo-review` | CEO | 창업자 모드로 "10점짜리 제품"을 찾아냅니다 |
| `/plan-eng-review` | 엔지니어링 매니저 | 아키텍처 다이어그램과 엣지 케이스를 분석합니다 |
| `/plan-design-review` | 디자인 리더 | 디자인 차원별 0~10점 평가를 제공합니다 |

### 디자인 (2개)

| 커맨드 | 역할 | 설명 |
|--------|------|------|
| `/design-consultation` | 디자인 컨설턴트 | 처음부터 완전한 디자인 시스템을 구축합니다 |
| `/design-review` | 디자인 QA | 80항목 체크리스트로 라이브 사이트를 시각적으로 감사하고 CSS를 자동 수정합니다 |

### 코드 품질 & 디버깅 (4개)

| 커맨드 | 역할 | 설명 |
|--------|------|------|
| `/review` | 스태프 엔지니어 | CI를 통과하지만 프로덕션에서 터질 버그를 찾습니다 |
| `/investigate` | 디버깅 전문가 | "근본 원인 없이 수정 없다" 원칙으로 체계적 디버깅합니다 |
| `/qa` | QA 리드 | 테스트 → 버그 수정 → 회귀 테스트 자동 생성까지 처리합니다 |
| `/qa-only` | QA 리포터 | `/qa`와 동일하지만 코드 수정 없이 리포트만 작성합니다 |

### 배포 & 문서 (7개)

| 커맨드 | 역할 | 설명 |
|--------|------|------|
| `/ship` | 릴리즈 엔지니어 | 동기화 → 테스트 → 커버리지 확인 → PR 생성을 한 번에 합니다 |
| `/land-and-deploy` | 배포 담당 | PR 머지부터 프로덕션 배포까지 처리합니다 |
| `/canary` | 모니터링 담당 | 배포 후 에러, 성능 이상을 자동 감시합니다 |
| `/benchmark` | 성능 측정 | 페이지 로드, Core Web Vitals, 번들 크기를 측정합니다 |
| `/cso` | 보안 감사관 | OWASP Top 10과 STRIDE 위협 모델링으로 보안 감사합니다 |
| `/document-release` | 문서 관리자 | 배포된 코드에 맞게 프로젝트 문서를 자동 업데이트합니다 |
| `/retro` | 스프린트 회고 리더 | 커밋 분석 기반 주간 회고를 진행합니다 |

### 브라우저 & 테스트 (2개)

| 커맨드 | 역할 | 설명 |
|--------|------|------|
| `/browse` | 브라우저 자동화 | 실제 Chromium 브라우저에서 클릭, 스크린샷 등을 수행합니다 (~100ms/커맨드) |
| `/setup-browser-cookies` | 쿠키 관리 | Chrome, Arc, Brave, Edge 등 실제 브라우저의 쿠키를 가져옵니다 |

### 멀티 AI & 안전 도구 (6개)

| 커맨드 | 역할 | 설명 |
|--------|------|------|
| `/codex` | 세컨드 오피니언 | 다른 AI 모델로 독립적인 코드 리뷰를 수행합니다 |
| `/careful` | 안전 감시 | `rm -rf`, `DROP TABLE` 같은 위험한 명령 실행 전 경고합니다 |
| `/freeze` | 편집 잠금 | 특정 디렉토리만 편집 가능하도록 잠급니다 |
| `/guard` | 최대 안전 모드 | `/careful` + `/freeze`를 동시에 적용합니다 |
| `/unfreeze` | 잠금 해제 | `/freeze`로 설정한 편집 제한을 해제합니다 |
| `/gstack-upgrade` | 자동 업그레이드 | gstack 자체를 최신 버전으로 업데이트합니다 |

### 자동화 (3개)

| 커맨드 | 역할 | 설명 |
|--------|------|------|
| `/autoplan` | 자동 기획 파이프라인 | CEO + 엔지니어링 + 디자인 리뷰를 한 번에 순차 실행합니다 |
| `/setup-deploy` | 배포 설정 | 배포 플랫폼, URL, 헬스체크를 자동 감지하고 설정합니다 |
| `/setup-browser-cookies` | 인증 설정 | 실제 브라우저의 로그인 상태를 테스트 환경에 복사합니다 |

## gstack의 핵심 설계 철학

Garry Tan은 두 가지 핵심 원칙을 강조합니다:

### 원칙 1: AI에게 명확한 역할을 부여하라

> "AI의 힘으로 10배에서 100배 생산성을 올리되, 각 부분을 해결할 때 항상 구조적으로 접근한다"

CEO에게는 CEO 리뷰를, QA 담당자에게는 테스트를 시키듯이, AI에게도 **역할별로 스킬을 나눠서** 맡기는 방식입니다.

### 원칙 2: 자동화를 신뢰하되 검증하라

> "100% 커버리지를 목표로 하고 항상 QA와 회귀 테스트를 먼저 작성하는 것이 gstack의 DNA"

코드를 작성하면 반드시 테스트가 따라붙고, 테스트를 통과해야 다음 단계로 넘어갑니다.

## 기술적 특징 (개발자 참고)

### 왜 Bun을 선택했나?

gstack은 Node.js 대신 Bun(번, 빠른 자바스크립트 런타임)을 사용합니다.

| 비교 항목 | Node.js | Bun |
|-----------|---------|-----|
| CLI(명령어 도구) 시작 시간 | 40~120ms | 8~15ms |
| 설치 후 파일 | node_modules 폴더 필요 | 단일 바이너리 |
| SQLite 지원 | 별도 설치 필요 | 내장 |
| TypeScript | 별도 변환 필요 | 네이티브 지원 |
| HTTP 서버 | 외부 프레임워크 필요 | 내장 (Bun.serve) |

### 영속적 브라우저 데몬

gstack의 가장 혁신적인 부분은 **브라우저를 백그라운드에서 계속 켜두는 방식**입니다. 보통 AI 도구가 브라우저를 쓰려면 매번 새로 열어야 하지만, gstack은 헤드리스 Chromium(화면 없이 동작하는 브라우저)을 데몬(백그라운드 프로세스)으로 띄워놓고 HTTP로 통신합니다.

| 비교 항목 | 매번 새로 여는 방식 | 영속적 데몬 방식 |
|-----------|-------------------|----------------|
| 첫 실행 | 2~3초 | 2~3초 (비슷) |
| 이후 명령 | 매번 2~3초 | ~100ms |
| 쿠키/로그인 상태 | 매번 새로 로그인 | 유지됨 |
| 토큰(AI 사용량) | 스크린샷 이미지라 많음 | 접근성 트리(텍스트)라 적음 |

gstack은 웹 페이지를 분석할 때 **스크린샷 대신 접근성 트리**(Accessibility Tree)를 사용합니다. 접근성 트리란, 웹 페이지의 구조를 텍스트로 변환한 것입니다. 마치 시각장애인용 화면 읽기 프로그램이 페이지를 읽는 것처럼, 화면을 "텍스트로 보는" 방식입니다.

### 보안 아키텍처

gstack은 보안에 특히 신경을 씁니다:

1. **macOS Keychain 연동** — 비밀번호를 운영체제의 보안 저장소에 저장합니다 (현재 macOS만 지원)
2. **PBKDF2 + AES-128-CBC 암호화** — 키체인 비밀번호에서 안전한 암호화 키를 파생시킵니다
3. **메모리 보호** — 암호화 키는 메모리에서만 존재하고, 디스크에 기록되지 않습니다
4. **자동 파기** — 프로그램 종료 시 캐시와 키가 자동으로 삭제됩니다
5. **로컬호스트 전용** — 브라우저 데몬은 외부 네트워크에서 접근할 수 없습니다
6. **Bearer 토큰 인증** — 세션마다 랜덤 UUID 토큰으로 인증합니다

### MCP를 쓰지 않는 이유

gstack은 MCP(외부 도구 연결 기능)를 **아키텍처적으로 사용하지 않습니다.** 단순히 "권장하지 않는다" 수준이 아니라, 설계 원칙으로 배제했습니다.

> "MCP는 요청마다 JSON 스키마 오버헤드가 추가되고 지속적 연결이 필요하다. 순수 HTTP + 텍스트 출력이 토큰 비용이 적고 디버깅도 쉽다." — gstack ARCHITECTURE.md

대신 Playwright 기반 HTTP 방식으로 브라우저를 제어하여, MCP 대비 약 30배 빠른 응답(~100ms)을 달성합니다.

## 주의할 점

- **현재 macOS만 완전 지원됩니다** — 보안 기능(Keychain 연동)이 macOS 전용으로 설계되어 있습니다. Windows/Linux에서는 Keychain 관련 기능이 작동하지 않습니다
- **Bun 설치가 필요합니다** — Mac/Linux에서 터미널에 한 줄 입력으로 설치 가능합니다
- **Playwright(웹 자동화 도구)가 필요합니다** — `/browse`, `/qa` 등 브라우저 관련 스킬을 사용하려면 Playwright가 설치되어 있어야 합니다
- **MCP 도구와 함께 쓰지 마세요** — gstack 공식 문서에서 `mcp__claude-in-chrome__*` 도구 사용을 명시적으로 금지합니다. gstack의 브라우저 데몬이 더 빠르고 효율적입니다
- **gstack은 빠르게 업데이트됩니다** — `/gstack-upgrade` 커맨드로 최신 버전을 유지하세요

## 정리

- gstack은 Claude Code에 28개 스킬을 설치하여 기획 → 디자인 → 개발 → QA → 배포 → 회고까지 전체 워크플로우를 자동화하는 프레임워크입니다
- Bun + Playwright + 영속적 브라우저 데몬 + 접근성 트리 방식으로 빠르고 안전하게 동작합니다
- Y Combinator CEO의 실전 경험에서 나온 도구이므로, 특히 1인 개발자나 소규모 팀이 "혼자서 팀처럼 일하기"에 활용할 수 있습니다
