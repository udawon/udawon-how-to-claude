## 이게 뭔가요?

회사에 새로 입사한 직원이 수백 개의 내부 문서를 처음부터 다 읽어야 한다고 상상해보세요. 매번 프로젝트를 시작할 때마다 그 문서들을 처음부터 읽어야 한다면 엄청난 시간 낭비입니다. **LLM Wiki Compiler**는 그 수백 개의 문서를 미리 한 번 정독해서 "핵심 요약집"을 만들어두는 도구입니다.

Claude(클로드, Anthropic이 만든 AI)와 함께 일할 때, Claude는 매 대화마다 관련 파일들을 처음부터 읽습니다. 파일이 많을수록 그만큼 토큰(AI가 글자를 읽는 단위, 많이 쓸수록 비용 증가)을 많이 소비합니다. 이 도구는 파일들을 미리 정리된 위키(wiki, 체계적으로 정리된 지식 문서)로 컴파일(compile, 여러 자료를 하나로 합치는 작업)해두어, 매번 원본 파일을 다시 읽지 않아도 되게 만들어줍니다.

실제 수치로는 **세션(session, AI와의 한 번의 대화)당 토큰 사용량을 약 79,000개에서 8,500개로, 89% 줄였습니다.**

## 핵심 기능 상세 분석

### 1. 두 가지 컴파일 모드

**Knowledge Mode (지식 모드):** 이미 작성된 마크다운(markdown, 간단한 기호로 서식을 표현하는 문서 형식) 파일들을 정리합니다. 예를 들어 회의록, 기획서, 개발 가이드 등 흩어진 문서들을 주제별로 묶어 정리된 위키로 만들어줍니다.

**Codebase Mode (코드베이스 모드):** 프로그램 소스코드(source code, 프로그램을 만드는 원본 텍스트)를 분석해서 아키텍처(architecture, 프로그램의 전체 설계 구조), API(Application Programming Interface, 프로그램끼리 대화하는 통로), 배포 방법 등을 자동으로 문서화합니다.

### 2. 지능형 구조 제안

`/wiki-init` 명령어를 실행하면, 폴더 안의 파일들을 샘플링(sampling, 전체 중 일부를 먼저 살펴보는 것)해서 프로젝트에 맞는 위키 구조를 자동으로 제안합니다. 스타트업(startup, 초기 단계의 기술 회사)인지, 연구팀인지, 개발팀인지에 따라 다른 구조를 추천합니다.

### 3. 신뢰도 표시 (Coverage Indicator)

각 위키 섹션에 **High / Medium / Low** 신뢰도를 표시합니다. 원본 파일에서 충분한 정보를 찾았으면 High, 정보가 부족하면 Low로 표시해 어느 부분을 더 확인해야 하는지 알려줍니다.

### 4. 지식 그래프 시각화

`/wiki-visualize` 명령어로 주제들 사이의 연결 관계를 그래프(graph, 점과 선으로 관계를 표현한 그림)로 볼 수 있습니다. Obsidian(옵시디언, 마크다운 기반의 인기 노트 앱)과도 연동됩니다.

### 5. 상태 관리

`/wiki-lint` 명령어로 위키의 오래된 내용이나 연결이 끊긴 항목을 자동으로 감지합니다. 원본 파일을 수정한 후 위키가 최신 상태인지 확인할 때 사용합니다.

## 사용 가능한 명령어

| 명령어 | 하는 일 |
|--------|---------|
| `/wiki-init` | 폴더를 분석해서 위키 구조 자동 제안 |
| `/wiki-compile` | 전체 문서를 위키로 컴파일 (처음 한 번, 이후 변경분만 업데이트) |
| `/wiki-ingest` | 새 파일 하나를 대화형으로 위키에 추가 |
| `/wiki-search` | 위키 전체에서 키워드 검색 |
| `/wiki-visualize` | 주제 간 관계를 그래프로 시각화 |
| `/wiki-lint` | 오래된 내용과 끊긴 링크 점검 |

Claude Code를 도입한 지 한 달이 지난 스타트업 CTO 박성민 씨는 매달 API(프로그램끼리 대화하는 통로) 비용 청구서를 받을 때마다 당혹감을 느꼈습니다. 1,200개가 넘는 코드 파일과 문서들이 매 대화마다 Claude에게 전달되고 있었고, 세션당 약 79,000 토큰이 소모됐습니다. 월 비용이 예상의 3배를 넘어섰습니다.

## 주의할 점과 한계

**콘텐츠는 자동으로 업데이트되지 않습니다.** 원본 마크다운 파일을 수정하면 위키도 다시 컴파일해야 합니다. `/wiki-lint`로 변경된 항목을 확인하고 `/wiki-compile`을 다시 실행하는 습관이 필요합니다.

**신뢰도 표시(Coverage Indicator)는 절대적이지 않습니다.** High라고 표시되어 있어도 원본 파일에서 중요한 내용을 놓쳤을 수 있습니다. 중요한 결정에는 원본 파일을 직접 확인하는 것을 권장합니다.

**처음 설정에 손이 조금 갑니다.** `/wiki-init`이 제안하는 구조가 항상 최적이 아닐 수 있어서, `.wiki-compiler.json` 파일을 직접 수정해야 할 수도 있습니다.

**Claude Code 전용입니다.** Cursor나 다른 AI 에디터에서는 동작하지 않습니다.

## 관련 유사 프로젝트

- **memory-compiler**: 세션 간 컨텍스트(context, 대화의 맥락과 배경 정보)를 자동으로 저장하고 다음 세션에 주입하는 Claude Code 훅(hook, 특정 시점에 자동 실행되는 스크립트) 도구. LLM Wiki Compiler가 "파일 지식"을 정리한다면, memory-compiler는 "대화 기억"을 관리합니다.
- **Andrej Karpathy의 LLM Knowledge Base 패턴**: LLM Wiki Compiler가 직접 구현한 개념의 원형. Karpathy가 제안한 "LLM을 위한 지식베이스 구축" 철학을 Claude Code 플러그인으로 실체화한 것이 이 프로젝트입니다.
