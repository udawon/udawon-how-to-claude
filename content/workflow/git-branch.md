## 브랜치(Branch)란?

**평행 세계**입니다.

```
메인(master)이라는 "안전한 세계"는 건드리지 않고
별도의 세계에서 마음껏 실험하는 것

비유: 중요한 발표 자료를 수정해야 할 때
     원본 파일은 그대로 두고
     "발표자료_실험용" 복사본을 만들어서 수정
     잘 되면 원본을 교체하고
     망하면 복사본만 삭제하면 됨
```

### 왜 브랜치를 쓰나?

```
상황: 포트폴리오 사이트를 운영 중인데,
     디자인을 완전히 바꿔보고 싶음

(X) 브랜치 없이:
  현재 사이트(master)에서 직접 수정 시작
  -> 3시간 동안 작업했는데 오히려 더 못생겨짐
  -> "원래 어떻게 생겼었더라?" 돌아가기 어려움
  -> 비유: 원본 서류에 직접 빨간펜으로 수정하는 것
           "아까 지운 문장이 뭐였지?" 복구 불가

(O) 브랜치 사용:
  "실험/새디자인" 브랜치를 만들어서 작업
  -> 3시간 동안 작업했는데 별로면?
  -> 브랜치만 삭제하면 끝. 원래 사이트는 그대로!
  -> 비유: 복사본에 수정하고, 괜찮으면 원본에 반영하는 것
           별로면 복사본만 휴지통에 버리면 됨
```

## 브랜치 기본 사용법

### Claude Code에서 쓰기 (전체 흐름 예시)

```
상황: 포트폴리오에 "프로젝트 갤러리" 페이지를 추가하고 싶음
     잘 될지 모르니 브랜치에서 먼저 해보기로 함

[1단계] 새 브랜치 만들기
나: "기능/프로젝트-갤러리 브랜치 만들어줘"
Claude: 새 브랜치 '기능/프로젝트-갤러리'를 생성하고 이동했습니다.
        현재 브랜치: 기능/프로젝트-갤러리

-> 이제부터 하는 모든 작업은 이 브랜치에만 기록됨
   master는 전혀 영향 없음!

[2단계] 작업하고 커밋
나: "프로젝트 갤러리 페이지 만들어줘. 카드 형태로 보여주고 싶어"
Claude: (갤러리 페이지 구현)
나: "좋아, 커밋해줘"
Claude: "프로젝트 갤러리 페이지 구현" 메시지로 커밋했습니다.

[3단계] 결과 판단

경로A - 마음에 든다!
나: "master로 돌아가서 이 브랜치 merge 해줘"
Claude: master로 이동 후 '기능/프로젝트-갤러리' 브랜치를 merge했습니다.
        갤러리 페이지가 master에 반영되었습니다.
-> 성공! 갤러리가 메인 사이트에 추가됨

경로B - 마음에 안 든다...
나: "master로 돌아가고 이 브랜치 삭제해줘"
Claude: master로 이동 후 '기능/프로젝트-갤러리' 브랜치를 삭제했습니다.
-> master는 갤러리 작업 시작 전 그대로! 아무 일도 없었던 것처럼
```

### 그림으로 보기

```
               갤러리 작업 시작        merge (합치기)
                    ↓                    ↓
master     ──●──●──●──────────────────●──  (항상 안전)
                     \                /
기능/갤러리    ──●──●──●──  (여기서 마음껏 실험)

만약 마음에 안 들면?

master     ──●──●──●────────────────────  (그대로 유지)
                     \
기능/갤러리    ──●──●──●  ← 이것만 삭제하면 끝!
```

### 명령어 요약

| 하고 싶은 것 | Claude에게 말하기 | git 명령어 |
|-------------|------------------|-----------|
| 브랜치 목록 보기 | "브랜치 목록 보여줘" | `git branch` |
| 새 브랜치 만들고 이동 | "○○ 브랜치 만들어줘" | `git checkout -b 이름` |
| master로 돌아가기 | "master로 돌아가줘" | `git checkout master` |
| 브랜치 합치기 | "master에 merge 해줘" | `git merge 이름` |
| 브랜치 삭제 | "이 브랜치 삭제해줘" | `git branch -D 이름` |

```
참고: Claude Code 대화창에서 한국어로 말하면 됩니다.
     git 명령어를 직접 입력할 필요 없습니다.
     명령어는 "이게 실제로 무슨 동작이구나" 참고용입니다.
```

## 브랜치 실전 활용

### 케이스 1: 새 기능을 안전하게 시도하기

```
상황: 내 블로그에 다크모드를 추가하고 싶은데,
     CSS를 많이 바꿔야 해서 기존 디자인이 망가질까 걱정

나: "기능/다크모드 브랜치 만들어줘"
Claude: 새 브랜치 '기능/다크모드'를 생성했습니다.

나: "다크모드 구현해줘. 토글 버튼도 넣어줘"
Claude: (다크모드 구현 - CSS 변수 추가, 토글 버튼 생성)

나: "커밋해줘"
Claude: "다크모드 기능 구현" 메시지로 커밋했습니다.

[브라우저에서 확인해봄]

결과A - "오 깔끔하다!":
  나: "master로 돌아가서 merge 해줘"
  -> 다크모드가 블로그에 정식 반영!
  -> 푸시하면 실제 사이트에도 적용

결과B - "음... 색상이 이상한데":
  나: "색상 좀 수정해줘. 배경은 #1a1a2e로 바꿔"
  Claude: (수정)
  나: "커밋해줘"
  -> 수정 후 다시 확인, 괜찮으면 merge

결과C - "다크모드 자체가 별로야. 없던 걸로 하자":
  나: "master로 돌아가고 이 브랜치 삭제해줘"
  -> 블로그는 다크모드 작업 전 그대로!
  -> 비유: 미용실에서 가발 써보고 "아, 안 어울려요" 하는 것
           내 진짜 머리카락은 그대로!
```

### 케이스 2: 두 가지 디자인을 비교해보기

```
상황: 메인 페이지의 네비게이션을
     "상단 탭" vs "왼쪽 사이드바" 중 고민 중

[방법 A 만들어보기]
나: "실험/탭-네비 브랜치 만들어줘"
Claude: 새 브랜치 '실험/탭-네비'를 생성했습니다.

나: "상단 탭 방식으로 네비게이션 만들어줘"
Claude: (탭 네비게이션 구현)
나: "커밋해줘"

-> 브라우저에서 확인, 스크린샷 찍어두기

[방법 B 만들어보기]
나: "master로 돌아가줘"
Claude: master 브랜치로 이동했습니다.

나: "실험/사이드바-네비 브랜치 만들어줘"
Claude: 새 브랜치 '실험/사이드바-네비'를 생성했습니다.

나: "왼쪽 사이드바 방식으로 네비게이션 만들어줘"
Claude: (사이드바 네비게이션 구현)
나: "커밋해줘"

-> 브라우저에서 확인, 스크린샷과 비교

[결정하기]
나: "사이드바가 더 낫네. master에 merge 해줘"
Claude: master로 이동 후 merge했습니다.

나: "실험/탭-네비 브랜치는 삭제해줘"
Claude: 삭제했습니다.

비유: 옷가게에서 두 벌을 번갈아 입어보고
     마음에 드는 쪽만 사고, 안 사는 옷은 다시 걸어놓는 것
     두 옷을 동시에 사지 않아도 됨!
```

### 케이스 3: 급한 수정과 진행 중 작업 분리

```
상황: "연락처 페이지" 브랜치에서 열심히 작업 중인데,
     친구가 "야 메인 페이지에 오타 있어!" 라고 알려줌

[진행 중 작업 임시 저장]
나: "지금 작업 중인 거 커밋해줘"
Claude: "연락처 페이지 작업 중 (미완성)" 커밋했습니다.

[오타 수정하러 이동]
나: "master로 돌아가줘"
Claude: master로 이동했습니다.

나: "메인 페이지에 '안녕하세여'를 '안녕하세요'로 수정해줘"
Claude: (오타 수정)
나: "커밋하고 푸시해줘"
Claude: 오타가 수정되어 사이트에 반영되었습니다.

[다시 원래 작업으로]
나: "다시 기능/연락처 브랜치로 돌아가줘"
Claude: '기능/연락처' 브랜치로 이동했습니다.

-> 아까 하던 연락처 페이지 작업 이어서 계속!

비유: 보고서 작성 중에 상사가 "지난 보고서 오타 수정해"
     지금 쓰던 보고서를 저장(커밋)해두고
     지난 보고서 수정 후 제출(푸시)
     다시 원래 보고서로 돌아와서 이어 쓰기
```

## Stash: 임시 보관함

커밋하기에는 애매하지만 작업 내용을 잃고 싶지 않을 때 씁니다.

### Stash란?

```
커밋 = "여기까지 완성! 저장"   (완성된 요리를 접시에 담기)
stash = "아직 미완성, 잠깐 치워두기"  (만들던 재료를 냉장고에 넣기)

언제 쓰나?
- 커밋하기엔 너무 미완성인데 다른 브랜치로 가야 할 때
- "잠깐만, 이거 나중에 쓸 건데 일단 치워두자"
- 브랜치 이동 시 "변경사항이 있어서 이동 못 합니다" 에러가 날 때

비유: 요리 중에 갑자기 택배가 옴
     불 끄고, 만들던 재료를 냉장고에 넣어두고 (stash)
     택배 수령 후
     냉장고에서 꺼내서 이어서 요리 (stash pop)
```

### 사용법 (대화 예시)

```
상황: 기능/갤러리 브랜치에서 페이지를 만들고 있는데,
     아직 반쯤밖에 안 했음.
     그런데 갑자기 master에서 급한 수정을 해야 함.
     커밋하기엔 반쪽짜리라 찝찝함.

[임시 보관하기]
나: "지금 변경사항 stash 해줘"
Claude: 변경사항을 stash에 보관했습니다.
        작업 디렉토리가 깨끗한 상태로 복원되었습니다.
-> 내가 수정하던 파일들이 "원래 상태"로 돌아감
   하지만 사라진 게 아니라 "서랍"에 들어간 것!

[급한 작업 처리]
나: "master로 이동해줘"
나: "푸터의 연락처 이메일을 contact@example.com으로 바꿔줘"
나: "커밋하고 푸시해줘"
-> 급한 수정 완료!

[원래 작업으로 복귀]
나: "다시 기능/갤러리 브랜치로 돌아가줘"
나: "아까 stash 한 거 다시 꺼내줘"
Claude: stash에서 변경사항을 복원했습니다.
-> 반쯤 만들던 갤러리 페이지가 다시 나타남! 이어서 작업 계속

비유: 시험 답안지 작성 중에 화장실 가야 할 때
     답안지를 서랍에 넣어두고 (stash)
     화장실 다녀오고
     서랍에서 꺼내서 이어서 풀기 (stash pop)
     중간까지 쓴 내용이 사라지지 않음!
```

### Stash 명령어 요약

| 하고 싶은 것 | Claude에게 말하기 | git 명령어 |
|-------------|------------------|-----------|
| 임시 보관 | "stash 해줘" | `git stash` |
| 꺼내기 | "stash 꺼내줘" | `git stash pop` |
| 목록 보기 | "stash 목록 보여줘" | `git stash list` |
| 보관 + 메시지 | "stash 해줘, 메시지는 '갤러리 작업 중'" | `git stash push -m "메시지"` |

```
stash vs 커밋, 뭐가 다르지?

커밋: "1장 완성!" 하고 저장. 이력에 남음. 다른 사람도 볼 수 있음.
stash: "1장 작성 중..." 하고 임시 보관. 이력에 안 남음. 나만 알고 있음.

언제 뭘 쓸까?
- 기능이 완성됨 -> 커밋
- 아직 미완성이지만 잠깐 치워야 함 -> stash
- 미완성이어도 상관없이 기록 남기고 싶음 -> 커밋 (메시지에 "작업 중" 표시)
```

## Git Diff: 뭐가 바뀌었는지 확인

커밋하기 전에 "내가 뭘 바꿨지?" 점검하는 기능입니다.

### 대화 예시

```
상황: Claude에게 여러 수정을 부탁한 뒤,
     커밋하기 전에 뭐가 바뀌었는지 확인하고 싶음

나: "지금까지 뭐가 변경됐는지 보여줘"
Claude:

  수정된 파일: 2개

  📄 index.html
  - <h1>나의 포트폴리오</h1>     (삭제된 줄)
  + <h1>김철수의 포트폴리오</h1>  (추가된 줄)

  📄 style.css
  - background-color: white;     (삭제된 줄)
  + background-color: #f0f0f0;   (추가된 줄)
  + font-family: 'Pretendard';   (새로 추가된 줄)

나: "좋아, 이대로 커밋해줘"

비유: 워드 문서의 "변경 내용 추적" 기능
     빨간 취소선 = 삭제된 것
     파란 밑줄 = 추가된 것
     한눈에 뭐가 바뀌었는지 보임
```

### 활용 예시들

```
1. 커밋 전 점검
   나: "변경사항 확인하고 문제없으면 커밋해줘"
   -> Claude가 변경 내용을 보여주고, 이상 없으면 커밋

2. 브랜치에서 뭘 했는지 전체 확인
   나: "이 브랜치에서 master 대비 뭐가 달라졌는지 보여줘"
   -> 브랜치에서 작업한 모든 변경사항을 한눈에 확인
   -> merge 하기 전에 최종 점검할 때 유용

3. 특정 파일만 확인
   나: "index.html만 뭐가 바뀌었는지 보여줘"
   -> 해당 파일의 변경 부분만 표시
```

## Git Log: 변경 이력 보기

"지금까지 뭘 했더라?" 과거 기록을 확인하는 기능입니다.

### 대화 예시

```
상황: 어제까지 뭘 작업했는지 기억이 안 남

나: "최근 커밋 이력 5개만 보여줘"
Claude:

  최근 커밋 이력:
  ┌────────────────────────────────────────────────┐
  │ 5a2e6e2  에이전트 팀 문서 업데이트      (3월 8일) │
  │ 993b9c0  statusline 추천 설정 3가지 추가 (3월 7일) │
  │ 6d98dae  Statusline 가이드 문서 추가    (3월 6일) │
  │ 1aedd35  작업 중 채팅 & 중단하기 문서    (3월 5일) │
  │ ecf46bb  Cyclist 마인드맵 뷰어 문서     (3월 4일) │
  └────────────────────────────────────────────────┘

### 활용 예시들

```

   나: "최근 커밋 이력 보여줘. 언제부터 이 버그가 생겼는지 찾아보자"
   -> 커밋을 하나씩 확인하며 문제 발생 시점 추적

## 기능 비교 요약

```
브랜치와 함께 알아두면 좋은 Git 안전 도구들:

┌─────────────┬──────────────────────────┬─────────────────────┐
│   기능       │  용도                    │  비유               │
├─────────────┼──────────────────────────┼─────────────────────┤
│ branch      │ 별도 공간에서 안전하게 실험│ 연습용 주방         │
│ stash       │ 미완성 작업 임시 보관     │ 냉장고에 넣어두기    │
│ diff        │ 뭐가 바뀌었는지 확인      │ 워드 변경 내용 추적  │
│ log         │ 과거 작업 이력 보기       │ 은행 거래 내역       │
│ checkpoint  │ Claude 작업 자동 저장     │ 게임 자동 저장       │
│ rewind      │ Claude 작업 되돌리기      │ Ctrl+Z 무한 버전     │
└─────────────┴──────────────────────────┴─────────────────────┘

어떤 상황에서 뭘 쓸까?

  "이 기능 넣어볼까 말까 고민"     -> branch 만들어서 시도
  "급한 일 생겼는데 지금 작업 중"  -> stash로 임시 보관
  "커밋 전에 한번 점검하고 싶어"   -> diff로 확인
  "어제 뭘 했더라?"              -> log로 이력 확인
  "Claude가 방금 한 거 마음에 안 들어" -> rewind로 되돌리기
```

## 브랜치 이름 짓기 팁

```
좋은 이름 예시:
  기능/다크모드          새 기능을 만들 때
  수정/로그인-버그       버그를 고칠 때
  실험/새레이아웃        시도해보는 것
  문서/가이드-추가       문서 작업할 때

영어로 지을 때:
  feature/dark-mode
  fix/login-bug
  experiment/new-layout
  docs/add-guide

팁: "카테고리/설명" 형식이 깔끔
    나중에 브랜치 목록을 볼 때 어떤 작업인지 바로 알 수 있음
```

## 흔한 질문

```
Q: 브랜치를 만들면 파일이 복사되나요? 용량이 두 배?
A: 아닙니다! Git은 "변경된 부분"만 따로 기록합니다.
   파일 100개짜리 프로젝트에서 브랜치를 만들어도
   용량은 거의 늘지 않습니다.
   비유: 책 전체를 복사하는 게 아니라
        "3페이지 5번째 줄을 이렇게 바꿨다"만 메모하는 것

Q: 브랜치를 여러 개 만들어도 되나요?
A: 네! 개수 제한 없습니다.
   다만 너무 많으면 관리가 복잡해지니
   완료되거나 불필요한 브랜치는 삭제하는 게 좋습니다.

Q: 지금 어떤 브랜치에 있는지 어떻게 알아요?
A: "지금 어떤 브랜치에 있어?" 라고 물어보면 됩니다.
   또는 "git 상태 확인해줘"라고 하면
   현재 브랜치와 변경사항을 함께 알려줍니다.

Q: 브랜치에서 작업한 걸 master에 합칠 때 충돌이 나면?
A: 같은 파일의 같은 부분을 master에서도 수정한 경우
   "충돌(conflict)"이 발생할 수 있습니다.
   이때 Claude에게 "충돌 해결해줘"라고 하면
   어떤 쪽을 유지할지 물어보고 처리해줍니다.

Q: stash에 넣어둔 걸 까먹으면?
A: "stash 목록 보여줘"라고 하면 보관 중인 목록이 나옵니다.
   메시지를 붙여서 보관하면 더 알아보기 쉽습니다.
   예: "stash 해줘, 메시지는 '갤러리 반쯤 완성'"
```
